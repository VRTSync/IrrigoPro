## Design Context (locked decisions)

These were agreed in conversation and are referenced by every slice in this
rebuild. Do not deviate from any of them.

**Scope.** Only billing sheets whose `billingNumber` starts with `BS-WC-` are
affected by the broader rebuild today. After Phase 2 completes, those rows
will live in `wet_check_billings` with `WC-YYYY-NNNN` numbering.

**Source of truth for the new layout.** `wetCheckFindings` joined to
`wetCheckZoneRecords`. Findings have `billingSheetId` (legacy) and
`wetCheckBillingId` (new, added in Slice 1). Both columns coexist during
migration; consumers must handle either source.

**Wet check billing status enum.** Wet check billings auto-submit on creation
(no `'draft'` state). Possible statuses: `'submitted'`,
`'pending_manager_review'`, `'approved_passed_to_billing'`, `'billed'`. This
is narrower than the billing_sheets enum which also has `'draft'` and
`'completed'`.

**Invoice items are already polymorphic.** `invoice_items.sourceType` accepts
`"work_order"`, `"billing_sheet"`, or `"wet_check_billing"` (the last added by
Slice 1). The `invoices.totalAmount` is the sum of all invoice items, so any
endpoint that queries invoices is automatically wet-check-aware.

**No PDF backward compatibility.** All wet check billings live in the current
billing cycle.

## Slice 3 — Dashboard + reporting consumers: dual-source aggregation

**Scope.** Update the three aggregate consumers (two route handlers + one
storage method) that compute monetary totals or pending-review counts across
billing sheets, so they also include `wet_check_billings`. Other read-side
endpoints stay billing-sheets-only — wet check billings will surface to users
via the dedicated `/wet-check-billings` page in Slice 4, not by being mixed
into existing billing-sheets lists. No user-visible change on day 1: the
`wet_check_billings` table is empty after Slice 1, so the new code paths are
dormant until Slice 5 (migration) populates the table.

  They must stay billing-sheets-only because the page they power will stop

  list; same reasoning.
- `/api/billing-workspace/queue` and `/api/billing-workspace/status-strip` —
  manager workflow queue stays billing-sheets-only.
- `storage.getDashboardStats.totalRevenue` — already sums `estimates`, not
