# Replit Task — Auth & Tenancy Hardening, Slice 4

This is a single self-contained task. The **Design Context** is in
`Auth-Tenancy - Slice 0 Hotfix.md` — re-read it before starting. The
actual work for this task is in the **Slice 4** section below. Do not
deviate from any file paths, function names, schema field names, or

> **Architectural note.** Slices 1–3 enforced tenancy at the route and
> storage layers without changing the schema. Slice 4 lands the
> structural change: `work_orders` and `billing_sheets` each get a
> `company_id` column. Existing rows are backfilled from the customer
> link inside a single migration. After backfill, both columns become
> `NOT NULL`. An index on `company_id` (and a composite on `company_id,
> status, scheduled_date` for work orders) supports the now-direct
> tenant filter. Storage methods from Slice 3 are switched from a
> customer-subquery to a direct column filter in the same slice. We
> also make `estimates.company_id` `NOT NULL` while we're in the
> migration, since it's the only existing tenant column that is
> currently nullable.

### 2. Migration SQL — `lib/db/migrations/0004_company_id_columns.sql`

Exact content (model on `0002_financial_pulse_foundation.sql` for
style; safe to re-run via `IF NOT EXISTS` guards on the column add and
the index create):

```sql
-- Slice 4 — Auth & Tenancy hardening.
--
-- Adds direct company_id columns to work_orders and billing_sheets,
-- backfills from the customer link, asserts no nulls remain, applies
-- NOT NULL, and indexes. Also tightens estimates.company_id to NOT
-- NULL. Idempotent guards make the column-add re-runnable; the
-- backfill is a single UPDATE.
--
-- Day-to-day this repo applies schema changes via
-- `pnpm --filter @workspace/db run push`. This file is the explicit,
-- hand-runnable equivalent for production rollout.

-- 1. Add columns nullable so existing rows continue to satisfy the
-- column list.
ALTER TABLE "work_orders"
  ADD COLUMN IF NOT EXISTS "company_id" integer REFERENCES "companies"("id");
ALTER TABLE "billing_sheets"
  ADD COLUMN IF NOT EXISTS "company_id" integer REFERENCES "companies"("id");

-- 2. Backfill from the customer link.
UPDATE "work_orders" wo
SET "company_id" = c."company_id"
FROM "customers" c
WHERE wo."customer_id" = c."id"
  AND wo."company_id" IS NULL;

UPDATE "billing_sheets" bs
SET "company_id" = c."company_id"
FROM "customers" c
WHERE bs."customer_id" = c."id"
  AND bs."company_id" IS NULL;

-- 3. Backfill estimates.company_id for any legacy null rows.
UPDATE "estimates" e
SET "company_id" = c."company_id"
FROM "customers" c
WHERE e."customer_id" = c."id"
  AND e."company_id" IS NULL;

-- 4. Assert no nulls remain. If this fails the migration aborts
-- before NOT NULL is applied; investigate orphaned rows before
-- retrying.
DO $$
BEGIN
  IF EXISTS (SELECT 1 FROM "work_orders" WHERE "company_id" IS NULL) THEN
    RAISE EXCEPTION 'Backfill incomplete: % work_orders rows have null company_id',
      (SELECT count(*) FROM "work_orders" WHERE "company_id" IS NULL);
  END IF;
  IF EXISTS (SELECT 1 FROM "billing_sheets" WHERE "company_id" IS NULL) THEN
    RAISE EXCEPTION 'Backfill incomplete: % billing_sheets rows have null company_id',
      (SELECT count(*) FROM "billing_sheets" WHERE "company_id" IS NULL);
  END IF;
  IF EXISTS (SELECT 1 FROM "estimates" WHERE "company_id" IS NULL) THEN
    RAISE EXCEPTION 'Backfill incomplete: % estimates rows have null company_id',
      (SELECT count(*) FROM "estimates" WHERE "company_id" IS NULL);
  END IF;
END $$;

-- 5. Apply NOT NULL.
ALTER TABLE "work_orders" ALTER COLUMN "company_id" SET NOT NULL;
ALTER TABLE "billing_sheets" ALTER COLUMN "company_id" SET NOT NULL;
ALTER TABLE "estimates" ALTER COLUMN "company_id" SET NOT NULL;

-- 6. Indexes.
CREATE INDEX IF NOT EXISTS "work_orders_company_idx"
  ON "work_orders" ("company_id");
CREATE INDEX IF NOT EXISTS "work_orders_company_status_scheduled_idx"
  ON "work_orders" ("company_id", "status", "scheduled_date");
CREATE INDEX IF NOT EXISTS "billing_sheets_company_idx"
  ON "billing_sheets" ("company_id");
```

### 3. Storage filter switch

Inside each Slice-3 method, replace the `customerId IN (...)` predicate
with `eq(workOrders.companyId, companyId)` (or the `billingSheets` /
`invoices` equivalent). Example — `getWorkOrders`:

```ts
async getWorkOrders(companyId: number | null, options?: { limit?: number; offset?: number }): Promise<WorkOrder[]> {
  try {
    const baseQuery = db.select().from(workOrders).orderBy(desc(workOrders.createdAt));
    const filteredQuery = companyId == null
      ? baseQuery
      : baseQuery.where(eq(workOrders.companyId, companyId));
    let query = filteredQuery;
    if (options?.limit) query = query.limit(options.limit);
    if (options?.offset) query = query.offset(options.offset);
    return await query;
  } catch (error) {
    console.error("Error fetching work orders:", error);
    return [];
  }
}
```

Single-ID reads switch from the subquery-AND-eq form to a single
`and(eq(workOrders.id, id), eq(workOrders.companyId, companyId))`.

The same pattern applies to invoices — by Slice 4 they need a
`company_id` column too. **Important:** check the invoices schema before
applying the migration. If `invoices` has no `company_id` column today,
add the column, backfill, and index in this same migration with the
same pattern. Update the migration SQL above with the third table if
so.
