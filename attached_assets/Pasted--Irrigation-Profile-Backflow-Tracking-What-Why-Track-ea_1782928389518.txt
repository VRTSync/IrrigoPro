# Irrigation Profile â€” Backflow Tracking

**What & Why**
Track each customer's **backflow preventers** â€” how many they have and each device's attributes â€” as a section in their Irrigation System profile, alongside Controllers. Because backflows require **annual certification testing** in most jurisdictions, include test/compliance tracking (last tested, next due, result) so due dates are visible and reportable.

**Confirmed decisions:** a **Backflows section inside the Irrigation System profile**; **attributes + test/compliance tracking**; capture **brand, model, size, type, serial #, location, install date, photo, notes**. Company-scoped (Rule 1). VRTSync-ready (external id) per the VRTSync design note. Single canonical store (no legacy duplication).

**Visual reference:** `backflow-tracking-mockup.html`.

## Slice 1 â€” Schema
### File: `lib/db/src/schema/schema.ts` + migration
New **`irrigation_backflows`** table:
- `id, companyId (FK, NOT NULL), customerId (FK), branchName (text NOT NULL default '')` â€” same tenancy/branch model as `irrigation_controllers`.
- Attributes: `name (label), brand, model, size, deviceType (enum: rpz | double_check | pvb | spill_resistant_pvb | other), serialNumber, location, installDate (date), photoUrl (nullable), notes`.
- Test/compliance: `lastTestedDate (date), nextTestDueDate (date), lastTestResult (enum: pass | fail | null), lastTestedBy (text)`.
- `isActive (default true), vrtSyncId (external id, nullable â€” for VRTSync match-and-update), lastUpdatedByUserId, lastUpdatedByName, lastUpdatedAt, createdAt, updatedAt`.
- Index by `(companyId, customerId, branchName)`; optional uniqueness on `(companyId, serialNumber)` where serial present.
- (Optional, if per-test history is wanted later) a `backflow_tests` child table â€” **out of scope for now**; the current model keeps the latest test on the device row plus a "Log test" action that updates it.

## Tests
- Add/edit a backflow with all attributes + test fields; it persists and renders with the correct status chip (overdue/due-soon/current) from the due date.
- "Log test" stamps the test and rolls the next-due date forward (+1 yr); status flips to Current.
- Company isolation: a user can't read/edit another company's backflows (404); customer must be in caller's company.
- Backflow photo displays (signed) and is authorized via `assertCanViewPhoto`; not viewable cross-company.
- Count badge/summary reflects the devices and their statuses.
