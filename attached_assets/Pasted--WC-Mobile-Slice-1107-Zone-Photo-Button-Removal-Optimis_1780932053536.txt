# WC Mobile — Slice 1107: Zone Photo Button Removal + Optimistic Finding Display

**Source:** Conversation 2026-06-08 — tech reported "Couldn't attach photo" 500 error and findings not appearing until manual refresh.  
**Severity:** P1 (broken photo flow causes loose-photo warnings; finding disappearance breaks tech confidence in data being saved).  
**Prerequisite slices:** None.  
**Single file touched:** `artifacts/irrigopro-mobile/app/wet-check/[id]/zone/[zoneRecordId].tsx`

---

## Why

### Bug 1 — Zone-level "Add Photo" button is inherently broken

`onAddPhoto` (line 538) calls `captureZonePhoto` with `findingId: null` (line 556). The server requires every wet check photo to be attached to a specific finding — a null `findingId` produces a 500 "Couldn't attach photo" error every time. There is no fix that makes zone-level photos work without a finding; the concept itself is invalid. The per-finding Photo button (rendered per finding row) already handles photo attachment correctly and is the only valid entry point.

Side effect: any upload attempt via this button adds a `PendingPhoto` to state, which renders the "1 loose photo — not attached to a work item" warning strip. Removing the button eliminates both the 500 and the spurious warning.

### Bug 2 — Newly added findings don't appear until manual refresh

`addFindingMutation.onSuccess` (line 413) only calls `queryClient.invalidateQueries`. This marks the cache stale and fires a background refetch — but the finding doesn't appear in the list until the server round-trip completes. The tech sees an empty list after tapping Save and has no confirmation the finding was recorded.

The `statusMutation` immediately above (lines 344–387) already implements the correct pattern: optimistic `setQueryData` in `onMutate`, rollback on `onError`, and `invalidateQueries` on `onSettled`. `addFindingMutation` needs the same treatment.

---

## Scope

1. Delete the zone-level Add Photo button and all code that exists solely to support it.
2. Add optimistic update to `addFindingMutation` so the new finding renders immediately on Save.

## Part 2 — Optimistic update for `addFindingMutation`

Mirror the `statusMutation` pattern exactly. The `WetCheckDetail` cache shape has a `zoneRecords` array; each zone record has a `findings` array. The optimistic update inserts a placeholder finding into the correct zone's `findings` list immediately on `onMutate`, then lets `onSettled` replace it with the real server row via `invalidateQueries`.

### Updated `addFindingMutation` (lines 390–427)

```ts
const addFindingMutation = useMutation({
  mutationKey:
    wetCheckId != null && zoneRecordId != null
      ? wetCheckMutationKey(wetCheckId, "finding-add", zoneRecordId)
      : ["wet-check", "mutation", -1, "finding-add"],
  mutationFn: async (input: {
    issueType: string;
    partId: number | null;
    partName: string | null;
    partPrice: string | null;
    quantity: number;
    laborHours: string;
    notes: string | null;
  }) => {
    if (zoneRecordId == null) throw new Error("Missing zone id");
    return wetCheckMutate<WetCheckFinding, typeof input>({
      path: `/api/wet-checks/zone-records/${zoneRecordId}/findings`,
      method: "POST",
      body: input,
      wetCheckId: wetCheckId ?? undefined,
      label: "Add finding",
    });
  },
  onMutate: async (input) => {
    if (wetCheckId == null || zoneRecordId == null) return { previous: undefined };
    await queryClient.cancelQueries({ queryKey: wetCheckDetailQueryKey(wetCheckId) });
    const previous = queryClient.getQueryData<WetCheckDetail>(
      wetCheckDetailQueryKey(wetCheckId),
    );
    if (previous) {
      // Build a placeholder finding to render immediately.
      // id: -1 is a sentinel — the real row (with its server id) replaces
      // it when onSettled fires invalidateQueries.
      const placeholder: WetCheckFinding = {
        id: -1,
        zoneRecordId: zoneRecordId,
        issueType: input.issueType,
        partId: input.partId,
        partName: input.partName,
        partPrice: input.partPrice,
        quantity: input.quantity,
        laborHours: input.laborHours,
        notes: input.notes,
        status: "needs_manager_review",
        photos: [],
        createdAt: new Date().toISOString(),
        updatedAt: new Date().toISOString(),
      };
      queryClient.setQueryData<WetCheckDetail>(
        wetCheckDetailQueryKey(wetCheckId),
        {
          ...previous,
          zoneRecords: previous.zoneRecords.map((z) =>
            z.id === zoneRecordId
              ? { ...z, findings: [...z.findings, placeholder] }
              : z,
          ),
        },
      );
    }
    return { previous };
  },
  onSuccess: () => {
    setFindingError(null);
    Haptics.notificationAsync(Haptics.NotificationFeedbackType.Success).catch(
      () => undefined,
    );
    setEditorOpen(false);
  },
  onError: (err, _input, ctx) => {
    // Roll back the optimistic insert.
    if (wetCheckId != null && ctx?.previous) {
      queryClient.setQueryData(wetCheckDetailQueryKey(wetCheckId), ctx.previous);
    }
    const m = errorMessage(err);
    if (m != null) setFindingError(m);
  },
  onSettled: () => {
    // Replace the placeholder with the real server row.
    if (wetCheckId != null) {
      queryClient.invalidateQueries({ queryKey: wetCheckDetailQueryKey(wetCheckId) });
    }
  },
});
```

### Notes on the placeholder shape

`WetCheckFinding` is the type returned by the server. Check the actual interface (search `WetCheckFinding` in the mobile codebase) and make sure the placeholder satisfies it — particularly any non-nullable fields. If the type has required fields not in `input` (e.g. `resolution`, `convertedAt`, `findingIndex`), set them to their zero/null/default values. The placeholder only needs to render without crashing — `onSettled` replaces it with the canonical server row as soon as the POST completes.

If the finding list renders a `key` based on `finding.id`, use `id: -1` as a stable key. If there's any chance of two simultaneous adds (unlikely on mobile), use a negative uuid instead.
