# Replit Task #1150 — Invoice Detail PDF Cleanup, Slice 1 (Reconciliation WC line, duplicate footer, VRT logo, per-zone labor)

This is a single self-contained task addressing four defects plus a small
set of layout-efficiency changes on the **monthly invoice detail PDF**
(`generateInvoiceDetailPDF`). All four defects were surfaced from a real
production export: *Prospect at Settlers Chase — Irrigation Billing
Detail (5/1/26–5/31/26), Invoice #94354*. Do not deviate from the file
paths, function names, or formula shapes specified. If you believe a
deviation is necessary, surface it and stop.

> **TL;DR.** Four bugs, one template family:
> 1. The reconciliation page never lists the Wet Check Billing rows, yet
>    `totals.grandTotal` already includes them — so the itemized list
>    doesn't reconcile to the grand total. (For #94354: Billing Sheets
>    subtotal $2,840.85 + WCB $3,473.91 = $6,314.76 grand total, but the
>    $3,473.91 line is invisible.)
> 2. A second, in-body footer (`pageFooter()`) renders on every page
>    showing "Invoice #94354 · Page 0" directly above the real puppeteer
>    footer. The CSS page counter can't resolve inside a `position:fixed`
>    body element, so it prints `0` on every page. It's pure duplication —
>    delete it.
> 3. The VRT-generated "Repairs Summary" section needs the VRT logo in
>    its header.
> 4. The per-zone wet-check breakdown shows only a **parts** subtotal per
>    zone. Add a per-zone **labor** line so labor is clearly broken out,
>    honoring the Slice 4c stale-labor guard.
>
> Plus layout-efficiency changes (Change 5) that remove ~1.5 pages of
> duplication and dead columns.

> **Existing infrastructure to NOT rebuild.**
> - The canonical page footer is the puppeteer `footerTemplate` in
>   `generateInvoiceDetailPDF` (`pdf-generator.ts:284–289`) — "Powered by
>   VRT · vrtsync.com | Page X of Y". It is correct and stays. Do NOT
>   add page numbering anywhere else.
> - `WetCheckBillingView` already exposes `zoneLaborSubtotal`,
>   `repairLaborHours`, and `zonesHaveStaleLaborData` per Slice 4c
>   (`wet-check-billing-view.ts:296,292,349`). Reuse them — do NOT
>   re-derive per-zone labor.
> - `vm.totals` already folds WCB labor/parts/total into the invoice
>   totals (`pdf-view-model.ts:376–389`). The reconciliation grand total
>   is already right; only the *itemized list* is missing the WCB rows.
>   Do NOT change `totals`.

## Design Context (locked decisions)

**The reconciliation list must enumerate every reference that feeds the
grand total.** Today `reconciliationPage` (`pdf-helpers.ts:722`)
destructures only `workOrders` and `billingSheets` from the view model
and renders sections for each, but `vm.wetCheckBillings` is never
iterated. Because `totals.grandTotal` *does* include the WCB rows, the
printed list silently fails to reconcile. The fix is presentational
only: add a "Wet Check Billings" section (rows + subtotal) to the recon
table, in both the branch-grouped and the non-branch code paths. No
totals math changes.

**One footer, owned by puppeteer.** The in-body `pageFooter()` call
(`pdf-generator.ts:327`) and its `.pdf-footer` CSS
(`pdf-helpers.ts:1607–1627`) are a vestige from the pre-`displayHeaderFooter`
era. `counter(page)` evaluates to `0` inside the fixed element, which is
where "Page 0" comes from. Delete the call, the `pageFooter` export, and
the now-dead CSS. The puppeteer footer remains the single source.

**The "Repairs Summary" block is the VRT-branded section.** That section
header (`partsBlockForWetCheckBS`, `pdf-helpers.ts:510`) is where the
VRT logo belongs — it labels the machine-generated wet-check repair
rollup. The logo is a fixed brand asset (not the customer's company
logo), so it is bundled with the PDF code, not passed through
`logoDataUri`.

**Per-zone labor is best-effort, exactly as Slice 4c established.** When
`view.zonesHaveStaleLaborData === true`, do NOT print a fabricated
per-zone labor number — show the zone parts subtotal, omit the per-zone
labor line, and rely on the WCB ticket's FINANCIAL BREAKDOWN labor total
(which is snapshot-backed). When labor data is fresh, show it per zone.

## Slice 1 — Five changes

### Change 1 — Add the Wet Check Billings section to the reconciliation page

In `reconciliationPage` (`pdf-helpers.ts:722`), `vm.wetCheckBillings` is
currently unused. Add a WCB section mirroring the existing Billing Sheets
section, in **both** branches of the function.

**Non-branch path** (`pdf-helpers.ts:799–864`). After the `bsSubtotal`
block, add:

```typescript
const wcbGroupTotal = wetCheckBillings.reduce(
  (s, r) => s + parseFloat(String(r.wetCheckBilling.totalAmount || '0')),
  0,
);

const wcbSectionHeader = wetCheckBillings.length > 0 ? `
  <tr class="recon-group-header recon-group-wcb">
    <td colspan="3">Wet Check Billings</td>
  </tr>` : '';

const wcbRows = wetCheckBillings.map(r => `
  <tr>
    <td class="recon-ref recon-ref-wcb">${r.wetCheckBilling.billingNumber}</td>
    <td class="recon-type recon-type-wcb">Wet Check</td>
    <td class="recon-total">${formatCurrency(parseFloat(String(r.wetCheckBilling.totalAmount || '0')))}</td>
  </tr>`).join('');

const wcbSubtotal = wetCheckBillings.length > 0 ? `
  <tr class="recon-subtotal">
    <td colspan="2" class="recon-subtotal-label">Wet Check Billings Subtotal</td>
    <td class="recon-total">${formatCurrency(wcbGroupTotal)}</td>
  </tr>` : '';
```

Then insert `${wcbSectionHeader}${wcbRows}${wcbSubtotal}` into the
`<tbody>` **between** `${bsSubtotal}` and `${warningRow}`
(`pdf-helpers.ts:857–858`). Add `wetCheckBillings` to the destructure on
line 723.

**Branch-grouped path** (`pdf-helpers.ts:733–797`). Resolved against
source: `PdfBranchSubtotal` (`pdf-view-model.ts:158–161`) carries only
`workOrders` and `billingSheets`, and the branch grouping
(`pdf-view-model.ts:405–432`) never folds WCBs into a branch. Therefore
render WCBs as a **single ungrouped "Wet Check Billings" section after
all branch blocks** and before the grand-total row (same three snippets
as above). Do NOT attempt to attach WCBs to individual branches — that
would require extending `PdfBranchSubtotal` and is out of scope for this
slice. `vm.wetCheckBillings` is the full list to render in both paths;
`invoice-pdf-service.ts` already populates it via `buildPdfViewModel`
(`invoice-pdf-service.ts:391`), so no service-layer change is needed.

Add CSS for the new classes next to the existing `recon-ref-bs` /
`recon-type-bs` rules so WCB rows are visually distinct (reuse the
existing color tokens; e.g. tint `recon-group-wcb` like
`recon-group-bs`).

**Acceptance:** For Invoice #94354 the recon table shows, in order:
Billing Sheets section (7 rows + $2,840.85 subtotal), Wet Check Billings
section (WC-2026-1008 → $3,473.91 + $3,473.91 subtotal), then GRAND
TOTAL $6,314.76. Billing-Sheets subtotal + WCB subtotal == grand total.

### Change 4 — Per-zone labor line in the wet-check breakdown

In `partsBlockForWetCheckBS` (`pdf-helpers.ts:527–584`), each zone block
currently ends with a single parts-only subtotal row
(`pdf-helpers.ts:541–545`). Add a per-zone labor line and make the zone
subtotal = parts + labor, gated on the Slice 4c stale-data flag.

`partsBlockForWetCheckBS` needs the labor rate. **Add a `laborRate:
number` parameter** and pass it from `ticketPageWCB` (it already computes
`laborRate` at `pdf-helpers.ts:359`). Update both call sites in
`ticketPageWCB` (`pdf-helpers.ts:390–392`).

For each zone, when `!view.zonesHaveStaleLaborData`:

```typescript
const zoneLaborHrs = parseFloat(zone.repairLaborHours) || 0;
const zoneLaborAmt = parseFloat(zone.zoneLaborSubtotal) || 0;
const zonePartsAmt = parseFloat(zone.zonePartsSubtotal) || 0;
const zoneTotal = zonePartsAmt + zoneLaborAmt;

const laborRow = zoneLaborHrs > 0 ? `
  <tr class="zone-labor-row">
    <td colspan="4">Zone ${zone.zoneLabel} Labor (${zoneLaborHrs} hrs × ${formatCurrency(laborRate)}/hr)</td>
    <td class="text-right">${formatCurrency(zoneLaborAmt)}</td>
  </tr>` : '';

const subtotalRow = `
  <tr class="zone-subtotal-row">
    <td colspan="4" style="font-weight:700; color:${black};">Zone ${zone.zoneLabel} Subtotal</td>
    <td class="text-right" style="font-weight:700; color:${brown};">${money(String(zoneTotal))}</td>
  </tr>`;
```

Render `${laborRow}` immediately before `${subtotalRow}` inside the zone
`<tbody>`.

When `view.zonesHaveStaleLaborData === true`: omit the labor row, keep
the subtotal as parts-only (current behavior), and render one small note
under the Repairs Summary header:

```html
<div class="zone-labor-note">Per-zone labor is summarized in the
Financial Breakdown above; zone subtotals below reflect parts only.</div>
```

Add CSS:

```css
.zone-labor-row td { color: #6b7280; font-size: 12px; }
.zone-labor-note { font-size: 11px; color: #6b7280; margin: 4px 0 8px; }
```

**Acceptance:**
- For a wet check with fresh labor data, each zone block shows a labor
  line ("Zone A-2 Labor (0.50 hrs × $75.00/hr) … $37.50") and a subtotal
  that equals parts + labor.
- The sum of all per-zone subtotals equals the WCB ticket's TOTAL
  (FINANCIAL BREAKDOWN) within $0.01.
- For a wet check flagged `zonesHaveStaleLaborData`, no fabricated labor
  appears; the note is shown and zone subtotals stay parts-only.

### Change 5 — Layout efficiency (recommended; confirm before shipping if unsure)

These remove duplicated content and dead columns. They are presentational
and independent of Changes 1–4.

**5a. Collapse the flat Repairs Summary table into an aggregated
rollup.** Today the Repairs Summary table
(`partsBlockForWetCheckBS`, `summaryRows`, `pdf-helpers.ts:493–522`)
lists **every individual line item**, and then the per-zone blocks below
list the **same** items again — ~1.5 redundant pages on #94354. Replace
the per-line summary with an aggregate grouped by `issueDisplayLabel`
(+ `partName`): one row per repair type with summed Qty and summed Parts
Total, plus a final "Repairs Total" row. The exhaustive per-zone
breakdown below remains the line-item record. Example shape:

| Repair Type | Part | Qty | Parts Total |
|---|---|---|---|
| Nozzle Replace | Set Nozzle 8/10/12/15 | 71 | $248.50 |
| Regular Head Replace | K-Rain 1804 Pop-Up | 38 | $902.50 |
| … | | | |

**5b. Drop the always-zero "Labor Hrs" column from per-sheet parts
tables.** In `partsTableFromBS` and `partsTableFromWO`
(`pdf-helpers.ts:589–653`), the "Labor Hrs" column is always `0` because
labor is billed as the separate "Irrigation Labor" line in FINANCIAL
BREAKDOWN. Remove the column (header + cell) to declutter; keep Part
Description / Qty / Unit Price / Total.

**5c. De-duplicate the part description on billing-sheet rows.** Several
rows print the part name both above and below the row (e.g. "Head –
K-Rain 1804…" twice; "1\" FEBCO Seal Kit" / "1\" FEBCO Repair Kit"). In
`partsTableFromBS` (`pdf-helpers.ts:626–636`) the `subLines` currently
appends `partDescription` even when it equals `partName`. Only render
`partDescription`/`notes` when they differ from `partName`.

**5d. (Optional) Cover page polish.** The cover wraps the company name
and billing period awkwardly and shows a bare street address ("14847
Madison St", no city/state/zip, and a raw phone "7208109673"). If the
company record has structured address fields, format them; otherwise
leave as-is. Flag this as a data-shape question rather than guessing.

**Acceptance for Change 5:** Page count for #94354 drops (the duplicated
flat repairs list is gone); no parts table shows a Labor Hrs column; no
row prints its part name twice. Per-zone line items are unchanged and
still complete.
