# Replit Task — Billing Approval Visibility, Slice 1 (Pending Approval List + Irrigation Manager access + drill-down from Command Center totals)

This is a single self-contained task. Do not deviate from the file paths,
endpoint paths, role lists, or status sets specified. If you believe a
deviation is necessary, surface it and stop.

> **TL;DR.** The Command Center page at `/billing/command-center`
> (`pages/customer-billing.tsx`) displays a header summary including
> `Unapproved: $17,233.49`, but there is no way to click into that figure
> to see *which* items make it up. The unified approval queue that
> already exists at `/billing-workspace` (Task #709,
> `pages/billing-workspace.tsx`) is the right list to show, but it is
> currently gated to `billing_manager` / `company_admin` / `super_admin`
> only — the irrigation manager who actually owns approvals for
> billing sheets, work orders, and wet-check billings cannot reach it.
> This slice (a) makes the Unapproved/Approved totals on the Command
> Center clickable drill-downs into a deep-linked pending-approval view,
> (b) grants `irrigation_manager` access to the same unified queue with
> a role-appropriate sidebar entry and badge, and (c) adds a new
> `awaitingApproval` sidebar badge that counts pending items for both
> roles so the irrigation manager gets a passive nudge without an
> additional email or notification system.

> **Existing infrastructure to NOT rebuild.**
>
> - `pages/billing-workspace.tsx` already implements: filter bar
>   (type/customer/tech/age/status), sort, sticky 50/page table,
>   right-hand detail drawer with **Approve**, **Save edits**,
>   **Kickback (reason)**, **Flag**, activity log, and J/K/A/B/F/Esc/?
>   keyboard shortcuts. Reuse it as the destination page — do not fork
>   it. Slice changes are role-gating, URL-state plumbing, badge
>   wiring, and a new bulk-approve action on top.
> - `GET /api/billing-workspace/queue` already returns the unified item
>   list with filters (`artifacts/api-server/src/routes/billing-workspace-routes.ts`
>   lines ~449–500). Reuse as-is for company_admin and billing_manager.
>   Add `irrigation_manager` to the role gate; do NOT widen the queue
>   contents.
> - `POST /api/billing-sheets/:id/approve`, `POST /api/work-orders/:id/approve`,
>   and `POST /api/wet-check-billings/:id/approve` already exist
>   (`artifacts/api-server/src/routes/approve-routes.ts:78, :170, :283`)
>   and already accept `irrigation_manager`. No new approve endpoint is
>   needed. The bulk endpoint introduced in this slice fans out to
>   these three.
> - The Command Center totals (`approvedTotal`, `unapprovedTotal`) are
>   computed in `GET /api/customers/billing-preview`
>   (`artifacts/api-server/src/routes/routes.ts:6019` ff.). Do NOT
>   reshape that endpoint. The drill-down uses the existing definitions
>   verbatim so the figure on the card and the row count on the list
>   page can never disagree.

## Design Context (locked decisions)

**One canonical "unapproved" definition, shared by the card and the list.**
The card-level `unapprovedTotal` is the sum of these three sets, all
filtered to rows where `invoiceId IS NULL`:

| Source | Status values counted as unapproved |
|--------|--------------------------------------|
| Billing Sheets | `pending_manager_review`, `completed`, `submitted` |
| Work Orders | `pending_manager_review`, `work_completed` |
| Wet Check Billings | `submitted`, `pending_manager_review` |

These are the exact sets used in `routes.ts` at the
`/api/customers/billing-preview` rollup (`unapprovedWorkOrders`,
`unapprovedBillingSheets`, `unapprovedWetCheckBillings`). They are
also identical to the `ACTIVE_BS` / `ACTIVE_WO` / `ACTIVE_WCB` sets in
`billing-workspace-routes.ts` (lines ~38–50). **Do not introduce a
fourth list.** If a definition needs to change later, it changes in
those two files at the same time, with a regression test that the card
total equals `SUM(items.total)` from the queue endpoint for the same
filter window.

**The destination is the existing Billing Workspace page, not a new page.**
The user-facing surface is `/billing-workspace` with a deep-linked URL
that pre-applies a "type=all, status=unapproved" filter and (when
drilled in from a customer card) a `customer=<id>` filter. A separate
`/pending-approval` page would either duplicate the table or diverge
from it; we have already paid that price once with the legacy
`/estimates/pending-approval` → `/estimates/command-center` migration
(`pages/estimate-command-center.tsx` header comment). Don't pay it
again.

**Irrigation manager gets the full queue, not a subset.** Approvals for
billing sheets and wet-check billings are explicitly an irrigation
manager responsibility in this codebase: `MANAGER_ROLES` in
`pages/customer-billing.tsx:122` and the role gate on
`POST /api/wet-check-billings/:id/approve` both include
`irrigation_manager`. The existing `manager-workspace-routes.ts` queue
covers wet checks and work-order routing but **excludes billing sheets
and wet-check billings** by design (see `MW_ROLES` and the item-type
switch around lines ~287–360). The right fix is to grant irrigation
manager access to `billing-workspace`, not to widen
`manager-workspace`.

**No email/digest in this slice.** The user chose "sidebar badge count"
only. Do not add a notification endpoint, cron, or email template.

**Bulk and per-customer approve are new; per-row Approve and edit-then-
approve already work.** The existing Billing Workspace drawer already
does per-row Approve and Save-then-Approve. The two additions in this
slice are:

1. **Bulk approve selected rows** — a checkbox column, an "Approve N
   selected" button, and a server-side bulk endpoint that fans out to
   the existing per-item approve handlers (so audit, status guards,
   and lifecycle hooks are unchanged).
2. **Approve all for a customer** — a one-click action available when
   the queue is filtered to a single `customer=<id>`. Same fan-out.

### 2. Bulk approve endpoint (`billing-workspace-bulk-approve.ts`)

```typescript
// POST /api/billing-workspace/bulk-approve
//
// Body: { items: Array<{ type: 'billing_sheet' | 'work_order' | 'wet_check_billing'; id: number }> }
//
// Behaviour: for each item, invoke the existing per-item approve
// handler **as if a separate request had been made**. We do this by
// calling the underlying storage transitions directly (the same
// functions `approve-routes.ts` calls), NOT by re-issuing HTTP
// requests. Audit + lifecycle hooks fire once per item.
//
// Response: { approved: number; skipped: Array<{ id: number; type: string; reason: string }> }
//
// Skip reasons:
//   - "status_not_active"  — item's current status isn't in the active set
//   - "not_found"          — item doesn't exist or isn't in caller's tenant
//   - "already_invoiced"   — invoiceId is set
//
// Failures during a single item's transition are recorded as a skip
// with reason: "transition_failed: <message>" — the whole call still
// returns 200 so the UI can show "Approved 7 of 10, 3 skipped".
```

The handler MUST reuse the same status-set constants
(`ACTIVE_BS` / `ACTIVE_WO` / `ACTIVE_WCB`) imported from
`billing-workspace-routes.ts`, so the "what counts as approvable"
contract stays single-sourced.

### 4. Badge query (`desktop-shell.tsx:107` ff.)

Add a fourth `useQuery` inside `useNavBadges`:

```typescript
const { data: queueProbeRaw } = useQuery<{ total?: number } | null>({
  queryKey: ["/api/billing-workspace/queue", { pageSize: 1, badgeProbe: true }],
  enabled,
  refetchInterval: 60000,
});
const awaitingApprovalCount =
  queueProbeRaw && typeof queueProbeRaw.total === "number" ? queueProbeRaw.total : 0;
```

Extend the return object:

```typescript
return {
  partsPendingApproval: pendingParts.length + pendingReviews.length,
  wetCheckReviews: wetCheckPending.length,
  estimatesPendingApproval: pendingEstimates.length,
  awaitingApproval: awaitingApprovalCount,
};
```

**Why a separate probe with `pageSize: 1`:** the queue list pages are
50 rows; we only need `total`. Keeping the badge query out of the
table cache means navigating to `/billing-workspace` doesn't have to
wait on a no-filter probe for SSR-style staleness. The query key
includes `{ badgeProbe: true }` solely so the cache key never collides
with the table's own list query.

Add a one-time read of `URLSearchParams` at the top of the component
(mirror the pattern in `pages/estimate-command-center.tsx` lines
~50–63):

```typescript
function readSearchOnce(): URLSearchParams {
  if (typeof window === "undefined") return new URLSearchParams();
  return new URLSearchParams(window.location.search);
}

// inside the component, alongside the existing filter state setup:
```

Then, in the existing `useState` initializer for the customer filter,

string `"approved_passed_to_billing"` (this filter slot already exists
in the queue endpoint via `req.query.status`).

### 6. Bulk approve UI wiring (`billing-workspace.tsx`)

Add a `selected: Set<string>` state alongside the existing
`activeId` / `kickbackReason` / `editPanelOpen` state. The keys are
the item `id` values returned by the queue endpoint (the
`type-refId` composite). The table head gets a checkbox that
selects/deselects all currently-rendered rows; each row gets a
checkbox. Selection survives page changes (do NOT clear on next-page).

The mutation:

```typescript
const bulkApproveMutation = useMutation({
  mutationFn: async (ids: string[]) => {
    const items = ids
      .map((id) => itemsById.get(id))
      .filter((i): i is QueueItem => Boolean(i))
      .map((i) => ({ type: i.type, id: i.refId }));
    return apiRequest("POST", "/api/billing-workspace/bulk-approve", { items });
  },
  onSuccess: (res) => {
    toast({
      title: `Approved ${res.approved}`,
      description: res.skipped.length
        ? `${res.skipped.length} skipped — see details`
        : undefined,
    });
    setSelected(new Set());
    qc.invalidateQueries({ queryKey: ["/api/billing-workspace/queue"] });
    qc.invalidateQueries({ queryKey: ["/api/billing-workspace/status-strip"] });
    qc.invalidateQueries({ queryKey: ["/api/customers/billing-preview"] });
  },
});
```

The `BulkApproveBar` component receives `count={selected.size}`,
`onApprove={() => bulkApproveMutation.mutate([...selected])}`,
`onClear={() => setSelected(new Set())}`, and a `pending` flag.

### 7. "Approve all for this customer" affordance

In the filter bar (`billing-workspace.tsx`), when `customerFilter` is

next to the filter chip:

```tsx
<Button
  variant="outline"
  size="sm"
```

button is hidden and replaced with a one-line note: "Showing first 50
— refine filter to bulk-approve."

## Acceptance criteria

- Logging in as `irrigation_manager`:
  - The sidebar shows a **Billing** group with a **Billing Workspace** leaf below Operations.
  - The leaf shows a numeric badge equal to the count of unapproved BS + WO + WCB across the user's tenant.
  - Clicking it opens `/billing-workspace` and the queue renders without a 403.
  - Per-row Approve, edit-then-Approve, kickback, and the new bulk-approve and per-customer-approve actions all work.

- Logging in as `company_admin` at `/billing/command-center`:
  - The header "Unapproved $X" amount is now an underline-on-hover link that navigates to `/billing-workspace?status=unapproved`.
  - The per-customer-card "Unapproved" amount navigates to `/billing-workspace?status=unapproved&customer=<id>`.
  - The same is true for "Approved" amounts, routing to `?status=approved`.
  - The destination page applies the customer filter on first paint (no flash of unfiltered queue).
  - The sidebar Billing Workspace leaf carries the same `awaitingApproval` badge as the irrigation manager sees.

- Bulk approve:
  - Selecting N rows and clicking "Approve N selected" closes the toast as `Approved K — M skipped` where `K + M === N`.
  - Underlying status transitions match what a per-row Approve would have done (verified by snapshot assertions on each row's audit log).

- Card / list parity:
  - For a freshly-seeded tenant fixture, the dollar amount in the Command Center "Unapproved" summary equals `SUM(items.total)` from `GET /api/billing-workspace/queue?type=all` for the same tenant, to within $0.01.

- No regression:
  - `field_tech` still 403s on `/api/billing-workspace/*`.
  - `billing_manager` and `company_admin` see the page unchanged except for the new checkbox column, bulk bar, and per-customer-approve button.
  - The existing keyboard shortcuts (J/K/A/B/F/Esc/?) still work; selection state does not consume any of them. (Recommended: bind `Shift+A` to bulk-approve only when `selected.size > 0`, documented in the `?` cheat sheet.)

## Out of scope

- Reshaping the `/api/customers/billing-preview` rollup or the customer-billing.tsx layout beyond wrapping the currency spans.
- Adding email or daily-digest notifications.
- Adding billing-sheet or wet-check-billing entries to `manager-workspace-routes.ts`. The irrigation manager gets the unified queue via `billing-workspace`, not via a second source in `manager-workspace`.
- Estimates. They have their own approval surface at `/estimates/command-center`; not part of this slice's scope per the user's item-type selection.
- Changing the `approved` vs `approved_passed_to_billing` vocabulary anywhere. The drill-down filter uses the existing literal `"approved_passed_to_billing"` because that is the status the queue endpoint accepts.
