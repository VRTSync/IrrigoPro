Most of the heavy stuff is now handled. The smoke-test checklist, env var inventory, DB safety, OpenAPI explicit-skip, PWA/SW callouts, rollback tag, and the TS-error policy all directly address the worries I had. That's a real improvement — this version is shippable in a way the original wasn't.
But the rewrite introduced a few internal contradictions you'll want to clean up before this runs, because an agent following both instructions literally will either freeze or pick wrong:
1. Step 2 contradicts pre-flight.
Step 2 still says: "Read ONLY these from .migration-backup/… Do NOT read anything else. The copy scripts handle the rest." But pre-flight step 1 requires confirming a directory's worth of files exist, and pre-flight step 2 requires grepping process.env. across .migration-backup/server/ and .migration-backup/shared/. Either remove the "do NOT read anything else" prohibition or scope it: "Beyond pre-flight reads, do not read additional files from .migration-backup/ until you hit an actual port issue requiring it." The blanket prohibition is the wrong default for an app with this much non-route logic.
2. Step 2's justification is now stale.
Step 2 says "you need their content for the OpenAPI spec" — but the new "OpenAPI decision" section explicitly says skip the spec. So why is the agent reading routes + schema? It's still useful (sanity-check that the route file moved, sanity-check schema for the DB push), but the stated reason no longer applies and the agent may get confused. Update the justification or drop step 2 entirely and let the copy scripts handle it.
3. Out of scope still says "use // @ts-ignore to keep moving."
This directly contradicts the new "TypeScript errors" section which says do NOT blanket-suppress. The Out of scope bullet should be rewritten to match: "Strict typecheck compliance for pre-existing errors only — confirmed against .migration-backup/ before suppressing. New errors caused by the move must be investigated."
4. The reviewer note still says "Do NOT reject for typecheck errors — those are explicitly permitted."
Same contradiction. If new TS errors are now disallowed (correctly, in my view), the reviewer needs to know that and reject for them. Reviewer note needs to align: "Reject for new TS errors that don't exist in .migration-backup/. Pre-existing TS errors are not regressions."
5. Step 3 says "Run codegen + DB push (parallel)" — pre-flight step 3 says don't push to prod first.
These reconcile only if DATABASE_URL was already pointed at a branch DB by the time step 3 runs. Worth being explicit at step 3: "By this point DATABASE_URL must point at the branch DB from pre-flight 3. If you instead chose the drizzle-kit generate review path, do not run db push here — surface the SQL diff to the user and wait for approval." Otherwise an agent that skipped or misread pre-flight will happily push against prod.
A couple of smaller things worth tightening:
6. QB OAuth smoke test can false-fail. "Confirm the existing QB token still refreshes" — if the QB refresh token has aged out for unrelated reasons (sandbox tokens are short-lived; production refresh tokens have their own renewal window), the test fails for reasons that have nothing to do with the port. Phrase it as: "If the QB token is currently valid, confirm a refresh-via-API still succeeds. If the token is already expired, confirm the OAuth re-connect flow loads correctly — both outcomes count as PASS."
7. Pricing visibility test should specify JSON inspection. "Confirm money fields are stripped" via UI is fragile because UI hides prices for other reasons too. Make it explicit: "Inspect the raw JSON response from /api/work-orders (or similar) as a field_tech user and confirm unitPrice, totalAmount, cost, markupAmount, etc. are absent — not just hidden in the rendered UI."
8. Push notification test needs HTTPS context. Service worker registration and push subscription require a secure context. If the smoke test is run in a non-HTTPS dev preview, registration may behave differently than in prod. Note: "Run the SW + push registration check against the deployed preview URL, not localhost, since push requires HTTPS."
9. lib/db forcing. Still slightly worried that IrrigoPro's DB code (with connect-pg-simple session table, billing_number_counters, the pricing-visibility logic that imports schema types) will fight a generic lib/db shape. The pre-flight DB-safety step protects the data, but doesn't address the structural fit. Worth one explicit line: "If lib/db's scaffold doesn't accommodate IrrigoPro's session table, billing counters, or other custom tables, surface the conflict before forcing a fit. Schema parity matters more than scaffold conformance."
10. Communication frequency. "Short, non-technical updates at each phase" is fine for routine work, but for this port, when the agent makes a non-trivial decision (which route signature to keep, which DB pattern to preserve, whether to adjust the pricing middleware to fit the new layout), it should pause and surface the decision rather than just narrate forward motion. Add: "On any non-trivial decision affecting auth, pricing visibility, the session table, or the QB/Postmark/Twilio integrations, pause and ask the user before committing."
If you patch those ten and run the pre-flight gates honestly, this is a reasonable plan for a hard task. The only one I'd consider blocking is #1 — the "do NOT read anything else" prohibition combined with the new pre-flight reading requirements is the kind of contradictory instruction that produces an agent which either does nothing or invents its own resolution. Worth fixing before kicking it off.
