## Design Context (locked decisions)

These were agreed in the audit
(`WC Billing Integration - Slice 0 End-of-Month Audit.md`) and apply to
every slice in this family.

**Primary mechanism.** The main "mark WCBs as billed" loop runs at
`routes.ts:7133+`. Every WCB whose ID was passed in
`selectedWcbs` is stamped `invoiceId / billedAt / status='billed'` in
that loop. Stragglers are theoretically impossible in the current
code path.

**Safety-net purpose.** The reconciliation block at `routes.ts:7141+`
exists for the case where the main loop and the
`invoice_items.sourceId` set diverge — either because a future edit
to the main loop introduces a skip, or because a manual repair script
inserts items but forgets to stamp the source row. The BS branch
inside the block has caught a real issue once historically; the WCB
absence is the kind of asymmetry that bites later.

**Idempotency.** The reconciliation only updates rows where the
`invoiceId` is still null. A WCB that was already stamped by the main
loop above is left untouched.

### Tests

#### New: `invoice-monthly-wcb-straggler.test.ts`

```typescript
describe("invoice creation reconciliation — WCB straggler safety net (Slice 3)", () => {
  it("stamps a WCB as billed when its invoice item exists but the main loop skipped it", async () => {
    // Setup: an invoice that has a WCB invoice item, but the WCB itself
    // is in 'approved_passed_to_billing' status with no invoiceId set —
    // simulating a future-broken main loop or a manual data-repair
    // scenario the safety net is meant to catch.
    const { customer, wcb } = await setupCustomerAndOrphanWcb({
      wcbStatus: "approved_passed_to_billing",
      wcbInvoiceId: null,
    });

    // Manufacture the orphan: an invoice with an invoice_item that
    // references the WCB but where the WCB row was never stamped.
    const [invoice] = await db.insert(invoices).values({
      customerId: customer.id,
      invoiceNumber: "INV-TEST-001",
      invoiceMonth: 5,
      invoiceYear: 2026,
      periodStart: new Date("2026-05-01"),
      periodEnd: new Date("2026-05-31"),
      partsSubtotal: "0",
      laborSubtotal: "200",
      totalAmount: "200",
    }).returning();

    await db.insert(invoiceItems).values({
      invoiceId: invoice.id,
      sourceType: "wet_check_billing",
      sourceId: wcb.id,
      wetCheckBillingId: wcb.id,
      description: "WC Billing WC-2026-1042",
      workDate: wcb.workDate,
      quantity: "1",
      unitPrice: "200",
      totalPrice: "200",
      laborTotal: "200",
    });

    // Invoke the route — the body intentionally targets a DIFFERENT
    // customer/period so the main WCB loop does not pick up the orphan,
    // forcing the safety net to be the only mechanism that could fix it.
    // (Test setup may vary — adapt to the existing harness in
    // invoice-monthly-wcb.test.ts.)
    await runReconciliationForInvoice(invoice.id);

    const after = await storage.getWetCheckBillingById(wcb.id);
    expect(after.invoiceId).toBe(invoice.id);
    expect(after.status).toBe("billed");
    expect(after.billedAt).toBeInstanceOf(Date);
  });

  it("does not touch a WCB that is already billed (idempotent)", async () => {
    const billedAt = new Date("2026-05-01T00:00:00Z");
    const { customer, wcb, invoice } = await setupAlreadyBilledWcb({
      billedAt,
    });

    await runReconciliationForInvoice(invoice.id);

    const after = await storage.getWetCheckBillingById(wcb.id);
    expect(after.billedAt?.getTime()).toBe(billedAt.getTime());
  });
});
```

### Acceptance criteria

- The reconciliation block at `routes.ts:7141+` includes an `else if
  (item.sourceType === 'wet_check_billing' && item.sourceId)` branch.
- The comment on the block mentions both line types.
- An orphan WCB invoice item is correctly stamped by the safety net
  during invoice creation (test passes).
- An already-billed WCB is NOT re-stamped by the safety net
  (idempotency test passes).
- Existing BS straggler behavior is unchanged (run the existing
  `billing-workspace-routes.test.ts` or any other BS-reconciliation
  test if present and confirm green).
- The reconciliation remains fire-and-forget — a storage error in
  the new branch is caught by the existing outer `catch
  (reconcileError)` at `routes.ts:7158+`.
- `pnpm typecheck` and `pnpm test` pass.

### Out of scope

- Refactoring or renaming the main WCB-billed-stamping loop at
  `routes.ts:7133+`. The main loop is the correctness mechanism; the
  safety net is purely defensive.
- Adding WO straggler reconciliation — WO is stamped by a different
  upstream mechanism and is not on this slice's scope.
- Changing `storage.updateWetCheckBilling` or
  `storage.getWetCheckBillingById`.
- Surfacing reconciliation hits in any UI or admin tool.
