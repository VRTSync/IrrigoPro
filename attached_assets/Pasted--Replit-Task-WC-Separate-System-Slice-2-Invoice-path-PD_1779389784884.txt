This is a single self-contained task. The **Design Context** below is locked
decisions from the broader rebuild — it is reference material so you understand
the intent. The actual work for this task is in the **Slice 2** section at the

> **Architectural note for this slice.** Wet check billings are being moved out
> of the shared `billing_sheets` table into their own parallel
> `wet_check_billings` table (Slice 1 added the table; nothing reads from it
> yet). This slice teaches the invoice path — preview, monthly generation, and
> PDF rendering — to recognize wet check billings as a polymorphic source
> alongside work orders and billing sheets.

> **Critical safety property:** This slice adds new code paths but produces

> table is empty until Slice 5 (migration) populates it. New invoice generation
> and PDF rendering will look exactly the same to users — the new code is
> waiting for data. If you find yourself making changes that affect what users
> see today (with the table still empty), you've drifted from the slice scope.

## Design Context (locked decisions)

These were agreed in conversation and are referenced by every slice in this
rebuild. Do not deviate from any of them.

**Scope.** Only billing sheets whose `billingNumber` starts with `BS-WC-` are
affected by the broader rebuild today. After Phase 2 completes, those rows
will live in `wet_check_billings` with `WC-YYYY-NNNN` numbering.

**Source of truth for the new layout.** `wetCheckFindings` joined to
`wetCheckZoneRecords`. Each finding has `billingSheetId` (legacy) and
`wetCheckBillingId` (new, added in Slice 1). Both columns coexist during
migration; consumers must handle either source.

**Layout.** Zone-grouped. Each zone displays as: zone header
(`<controllerLetter> <zoneNumber>` + issue display label), part lines, per-zone
repair labor row showing `hours × rate = total`, per-zone subtotal. A single
wet-check-level "Inspection Labor" line appears once per sheet.

**Labor rate.** Always the per-customer global rate, snapshotted onto the
billing record at creation (`appliedLaborRate`). Never reprice based on later
customer-rate changes.

**Issue type display.** Always use `issueTypeConfigs.displayLabel` for the
customer's company. Never display the raw `issueType` field. If no config row
exists, fall back to title-cased `issueType.replace(/_/g, " ")`.

**$0.00 line item suppression on PDF.** Suppress when `lineTotal === 0 AND
laborHours === 0 AND NOT isLaborOnly`. Labor-only findings (qty=0, partPrice=0,
noPartNeeded=true) are kept because they represent real work performed.

**PDF treatment.** Repairs Summary table at the top, each zone wrapped in a
div with `page-break-inside: avoid`, labor math shown on PDF
(`hours × rate = total`), labor subtotal collapsed into a single "Irrigation
Labor" line (per Task #766), issue type display labels everywhere.

**Invoice items are already polymorphic.** `invoice_items.sourceType` accepts
`"work_order"` or `"billing_sheet"` today. Slice 1 added `"wet_check_billing"`
as a third value and added a nullable `wetCheckBillingId` FK column on
`invoice_items`. This slice extends consumers to use that pattern.

**No PDF backward compatibility.** All wet check billings live in the current
billing cycle. There is no "old vs new" PDF reconciliation to handle.

**(4) Extend `invoice-pdf-service.ts`** (~lines 268–297):

After the existing for-loop that builds the `workOrders` and `billingSheets`
arrays, add a parallel block:

```typescript
const wetCheckBillings: PdfWetCheckBillingRow[] = [];

for (const item of invoice.items) {
  if (item.sourceType === 'wet_check_billing' && item.wetCheckBillingId) {
    const wcb = await this.storage.getWetCheckBillingById(item.wetCheckBillingId);
    if (!wcb) continue;
    const existing = wetCheckBillings.find(
      (w) => w.wetCheckBilling.id === wcb.id,
    );
    if (existing) continue;
    const wetCheckView = await this.storage.getWetCheckBillingViewById(
      wcb.id,
      customer.companyId,
    );
    if (!wetCheckView) continue;
    wetCheckBillings.push({
      wetCheckBillingId: wcb.id,
      wetCheckBilling: wcb,
      wetCheckView,
    });
  }
}
```

Pass `wetCheckBillings` into the `buildPdfViewModel` call (~line 322).
