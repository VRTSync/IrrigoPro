## Design Context (locked decisions)

These were agreed in conversation and are referenced by every slice in the
broader rebuild.

**Scope.** Wet check billings are now a first-class entity in their own
`wet_check_billings` table (`WC-YYYY-NNNN` numbering), separate from
`billing_sheets`. The backend endpoints have been wet-check-aware since
Phase 2 Slices 2–3 deployed. This slice is purely frontend wiring of the
Billing Command Center page to consume that already-existing capability.

**Polymorphic invoice items.** `invoice_items.sourceType` accepts
`'work_order'`, `'billing_sheet'`, or `'wet_check_billing'`. The
`/api/invoices/monthly` endpoint creates these polymorphic items based on
the selection IDs passed in.

**Backend parameter naming convention** (verified from Phase 2 Slice 2):
- `/api/invoices/preview` accepts `wetCheckBillingIds: number[]`
- `/api/invoices/monthly` accepts `selectedWetCheckBillingIds: number[]`
  (note the `selected` prefix on the create endpoint — match the existing
  `selectedBillingSheetIds` naming convention used on that endpoint)

**Response shape from `/api/customers/:id/billing`** (verified from Phase 2
Slice 3) — already includes:
- `wetCheckBillings: WetCheckBilling[]`
- `unbilledWetCheckBillings: WetCheckBilling[]`
- `totalUnbilledAmount` (already includes WCB amounts in the sum)

**(4) Add toggle handler** (~line 241, paralleling

```typescript
const toggleWetCheckBillingSelection = (wetCheckBillingId: number) => {
  setSelectedWetCheckBillingIds(prev => {
    const newSet = new Set(prev);
    if (newSet.has(wetCheckBillingId)) {
      newSet.delete(wetCheckBillingId);
    } else {
      newSet.add(wetCheckBillingId);
    }
    return newSet;
  });
};
```
