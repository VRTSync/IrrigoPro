# Findings-Lane Cleanup — Remove the Orphaned `findings_to_route` Server Code + Stale Copy + Orphan Diagnostic

**What & Why**
Findings must never be standalone routable items in Manager Workspace — they're routed only via the wet-check review/triage engine. The Manager Workspace Simplification (#1258) removed the findings *UI* and **already tombstoned `POST /api/manager-workspace/findings/bulk-route` to a 404** (guarded by `manager-workspace-slice1.test.ts`). But the server-side `findings_to_route` **stage assembly and counts are still live and orphaned**, a dead type field remains in the page, and a wet-check dialog still tells users to use a "Manager Workspace findings queue" that no longer exists. This ticket deletes the orphaned code, fixes the stale copy, and adds a read-only diagnostic to classify the existing orphan-finding backlog.

This is **two slices** (1 = removal/cleanup, 2 = read-only diagnostic). No schema changes. Keep the existing `bulk-route` 404 tombstone and its test as-is.

## Slice 2 — Read-only orphan-findings diagnostic

### Done looks like
- A read-only script classifies every finding with all four routing FKs null and `resolution != 'documented_only'` by its parent `wet_checks.status`, so we can see which orphans are now invisible after the lane removal.
- Mutates nothing. Prints class counts + writes a CSV.

### Why it matters now
With the lane gone, orphan findings on **active** (`submitted`/`pending_manager_review`) and **`partially_converted`-with-unrouted** wet checks still surface — via the wet-check Needs-Review queue. But orphans on **`approved`/`converted`** wet checks surface nowhere (Needs-Review excludes those by design). This diagnostic finds that hidden Class C so we can decide remediation.

### Steps
1. Add `artifacts/api-server/scripts/diagnose-orphaned-findings.ts` (read-only).
2. Classify by parent wet-check status: **A active** (`submitted`/`pending_manager_review`), **B `partially_converted`**, **C `approved`/`converted`** (the hidden true orphans). Also bucket by age, by `resolution`, by `companyId`, and flag any with non-null `partPrice`/`quantity` (would have carried a dollar amount).
3. Print a summary table; write `orphaned-findings-report.csv` (findingId, wetCheckId, wcStatus, resolution, techDisposition, companyId, customerName, partPrice, quantity, ageDays).

### Remediation (do NOT implement — for Randy after he reviews counts)
For **Class C** only: (a) sweep to `documented_only`, (b) reopen parent wet check to `partially_converted` so triage can route them, or (c) leave as DB-only. One-paragraph follow-up slice once counts are known.

### Relevant files
- `artifacts/api-server/scripts/diagnose-orphaned-findings.ts` (new)
- `lib/db/src/schema/schema.ts` (read-only reference)

### Acceptance (whole ticket)

- Stale wizard copy corrected.
- Diagnostic runs read-only, prints A/B/C counts + dollar/age breakdowns, writes the CSV; no rows mutated.
