# Slice 6 ‚Äî Desktop Shell + Left-Rail Navigation

**Date:** 2026-05-05
**Status:** Draft for Replit implementation
**Decisions locked:**
- The mobile-first navigation (top header + fixed bottom 5-slot bar) **stays exactly as-is for field techs and irrigation managers**. This slice only changes the desktop chrome for `company_admin` and `billing_manager`.
- At `‚â• lg` (1024px), those two roles get a **collapsible left-rail sidebar** plus a **TopStrip** (sidebar trigger + breadcrumb + page-actions slot + notifications).
- At `< lg`, those two roles still see the **existing mobile chrome** (`<Navigation />`) ‚Äî unchanged. The split is purely chrome; the route tree is shared.
- Sidebar is collapsible to an icon-only rail. **Default open at first visit.** Collapsed state persists to `safeStorage` under key `irrigopro_desktop_sidebar_open`.
- Top strip has breadcrumb + page title + a `<PageActions>` portal slot. **No Cmd+K command palette in v1** ‚Äî fast-follow.
- Company Admin gets sidebar links to **everything** they currently have routed (this includes Parts Pending Approval, which today is routed but hidden from the nav ‚Äî close that gap).
- Billing Manager gets a new sidebar link to **Wet Check Pending Review** (today routed but hidden from nav). Billing Manager **also gets Wet Check Issue Types** as a Settings entry ‚Äî add the route to `App.tsx`.
- The Reports group (audit + missing-photo pages) is **a nested collapsible sub-section under Billing** for both roles, not a top-level group.
- Active-state highlighting uses **longest-prefix wins** ‚Äî when on `/work-orders/missing-photos`, only the deepest matching leaf highlights, not both `Work Orders` and `Missing Photos ‚Äì Work Orders`.
- The shadcn `Sidebar` primitive at `client/src/components/ui/sidebar.tsx` is the component foundation. **Do not rebuild it**; reuse its `Sidebar / SidebarProvider / SidebarMenu*` family.

**Depends on:** nothing ‚Äî this is a pure UI/chrome change. No schema, no API, no data work.

**Roadmap note:** Cmd+K command palette is the obvious fast-follow once this lands; both roles touch enough screens that typing-to-jump is high-leverage. Out of scope for this slice.

---

## Executive Summary

Today the Company Admin and Billing Manager roles use the same mobile-first navigation as field techs: a top bar at desktop breakpoints with five dropdowns and a fixed bottom 5-slot bar at mobile. That works for techs, who have five screens. It strains for these two roles ‚Äî the company admin's `Admin` dropdown alone holds seven items, the `Operations` dropdown is two-deep, and several routed pages (Parts Pending Approval, Wet Check Pending Review, audit reports) never made it into the nav at all. The result is a hover-and-hunt experience that's actively making the app feel less professional than it is.

This slice introduces a **second navigation chrome** ‚Äî a collapsible left-rail sidebar plus a top strip ‚Äî that activates only at `‚â• lg` and only for these two roles. Everyone else is untouched. The mobile layout for the two affected roles also stays as-is; a billing manager on their phone in the field still gets the bottom nav. The result is a single app that feels native on the device the user happens to be on: phone-style chrome on a phone, desktop-style chrome on a laptop.

**Slice 6 ships when:** a Company Admin or Billing Manager opening IrrigoPro in a desktop browser sees a collapsible left rail with grouped navigation matching the spec below; the top strip shows a breadcrumb of the current page; collapsing the rail to icons persists across reloads; every routed page in the role is reachable via the rail (closing two pre-existing gaps); on the same browser at narrower widths the existing mobile chrome appears unchanged; the field tech / irrigation manager / super admin experience is byte-identical to today.

## Why Now

Three reasons.

**The two roles outgrew the chrome.** Company Admin's `Admin` dropdown holds seven items today; `Operations` is a dropdown of dropdowns; the `Billing` dropdown only exposes two of the four billing surfaces. Billing Manager has six top-level dropdowns and only one of them (`Billing`) is multi-level. Both flow charts have outgrown the horizontal real estate, and dropdowns at this density are slow even for users who know the app.

**Routed pages are unreachable.** Parts Pending Approval is routed for Company Admin and the nav even fetches the badge count for it, but the link itself isn't in any dropdown. Billing Manager has Wet Check Pending Review routed but hidden. The audit pages (zero-price, labor-rate, missing-photos) are routed for both roles but never linked. Closing these gaps is overdue.

**The desktop format is a force multiplier for everything else.** Once a left rail and a top strip exist, every page can pin a primary action ("New Work Order") to the strip. Every page can drop a breadcrumb without re-implementing it. Adding a Cmd+K palette later becomes trivial because the route metadata already exists. The chrome change pays itself back across the rest of the roadmap.

## Design Principles

The same principles that govern the rest of the app ‚Äî minimum chrome, sentence case, generous whitespace ‚Äî apply here. A few specific to this slice:

**One nav surface per breakpoint, not two.** At any given viewport width, the user sees either the desktop chrome or the mobile chrome ‚Äî never both, never overlapping, never two competing places to look. The CSS does the switching; React renders one tree.

**Field tech is the canary.** If a field tech's experience changes by even one pixel after this slice, something is wrong. The split is keyed on `user.role`, and the mobile chrome under `<DesktopShell>` is rendered by inserting the existing `<Navigation />` inside an `lg:hidden` wrapper ‚Äî exactly the same component that ships today. We aren't recreating mobile; we're just adding a desktop sibling.

**The rail is grouped, not flat.** The two affected roles each have 15+ destinations. A flat list of 15+ links is unreadable at a glance. Group labels are uppercase eyebrow text (`SidebarGroupLabel`'s default style), each group is a contiguous block, and the visual hierarchy is: Dashboard ‚Üí daily work ‚Üí record-keeping ‚Üí setup. Order matters: Operations comes before Billing for the admin (do the work, then get paid); Billing comes before Operations for the billing manager (their day starts with money).

**Active state is exactly one item.** Longest-prefix wins. When on `/work-orders/missing-photos`, the only highlighted item is `Missing Photos ‚Äì Work Orders`. The `Operations` group label still appears as the breadcrumb root in the top strip (so the user sees their location), but the `Work Orders` leaf in Operations is **not** also highlighted.

**Collapsed sidebar shows tooltips.** When the rail is collapsed to icons, hovering an icon shows the label as a tooltip on the right side. shadcn's `SidebarMenuButton` accepts `tooltip={leaf.label}` for this; pass it on every leaf.

**Breadcrumb is a path, not a back-button.** Each segment except the last is a link to the parent path (group label is non-linkable; the leaf-level segment is the current page and is unstyled-as-link). On `/billing-sheets/zero-price-audit` the breadcrumb reads `Billing ‚Ä∫ Reports ‚Ä∫ Zero Price Audit`.

**Color and density match the existing app.** The shadcn `Sidebar` ships with a `bg-sidebar` token that maps to a near-white in this app. Use it. Don't introduce a new accent. Badges on Parts Pending Approval and Wet Check Reviews use the existing `destructive` variant so they read the same as everywhere else.

## Active State, Breadcrumb, and PageActions

**Active state.** Implement two helpers in `desktop-shell.tsx`:

```
function pathMatches(itemPath, location) {
  if (itemPath === "/") return location === "/";
  if (location === itemPath) return true;
  return location.startsWith(itemPath + "/");
}

function pickActiveLeafPath(config, location) {
  const matches = walk all leaves and collect those where pathMatches(leaf.path, location);
  if (matches.length === 0) return null;
  return matches.reduce((a, b) => a.path.length >= b.path.length ? a : b).path;
}
```

Pass `activePath` (computed once via `useMemo` from `useLocation()` and the config) down into every leaf. A leaf is active iff `activePath === leaf.path`. A group is "active" iff any descendant leaf has `activePath === leaf.path` ‚Äî used to drive auto-expand on the Reports nested group.

**Breadcrumb.** `resolveRouteMeta(location, config)` returns `{ title, breadcrumb: [{label, path?}, ...] }`. Walk the config to find a leaf whose path matches `location` exactly (after trimming a trailing slash); return its parent group chain (group labels, no `path`) followed by the leaf itself (with `path`). For routes not in the config, match against a small ordered list of regex patterns:

- `/^\/customers\/[^/]+\/profile$/` ‚Üí `[Customers, "Customers" ‚Üí /customers, "Profile"]` title `Customer Profile`
- `/^\/customers\/[^/]+\/site-maps$/` ‚Üí `[Customers, "Customers" ‚Üí /customers, "Site Maps"]` title `Customer Site Maps`
- `/^\/wet-checks\/[^/]+\/review$/` ‚Üí `[Operations, "Wet Check Reviews" ‚Üí /wet-checks/pending-review, "Review"]` title `Wet Check Review`
- `/^\/wet-checks\/[^/]+$/` ‚Üí `[Operations, "Wet Checks"]` title `Wet Check`
- `/^\/wet-checks$/` ‚Üí `[Operations, "Wet Checks"]` title `Wet Checks`
- `/^\/user-profile$/` ‚Üí `[My Account]` title `My Account`
- `/^\/switch-user$/` ‚Üí `[Switch User]` title `Switch User`
- `/^\/license-agreement$/` ‚Üí `[License Agreement]` title `License Agreement`
- `/^\/privacy-policy$/` ‚Üí `[Privacy Policy]` title `Privacy Policy`
- `/^\/operations$/` ‚Üí `[Operations]` title `Operations`
- `/^\/admin$/` ‚Üí `[Dashboard]` title `Dashboard`

Final fallback: humanize the last path segment (`split("/")` filter empty, take last, replace dashes with spaces, title-case).

**PageActions.** Provide via context the current `HTMLDivElement` ref of the actions-slot div in TopStrip. The `PageActions` component reads context, returns `null` if context is null (mobile, or before TopStrip mounts), otherwise `createPortal(children, target)`. The actions div sits in TopStrip's `ml-auto` flex group, before `<NotificationSystem />`. Pages adopt it later, page by page ‚Äî out of scope for this slice apart from the slot existing.

## Persistence

Sidebar collapsed state goes to `safeStorage` (the existing wrapper around `localStorage` that falls back to `sessionStorage` for Safari Private Browsing). Key: `irrigopro_desktop_sidebar_open`. Initial state defaults to `true` on first visit. Read once in a `useState` initializer; write on every change via `onOpenChange`. Use shadcn's `SidebarProvider` in **controlled mode** (pass both `open` and `onOpenChange`); its built-in cookie persistence is write-only on the client and won't restore on reload.

The one-time onboarding hint key (Sub-slice 6C) is `safeStorage` key `irrigopro_desktop_shell_seen` storing `"true"` once dismissed. If absent on first sidebar open, render the popover; on dismiss set the key.

**No new endpoints.** The badge counts come from existing queries:

If `user.id` isn't a number (e.g., `safeGet("user")` returned an empty object), don't render it ‚Äî the existing top bar rendering doesn't check, but the strict type and the user-coming-from-localStorage shape make a guard worth adding.

## Responsive Behavior

**`< lg` (under 1024 px):** the existing `<Navigation />` renders inside an `lg:hidden` wrapper inside `<SidebarInset>`. Since the sidebar wrapper is `hidden lg:contents`, it has `display: none` here ‚Äî both the inline desktop sidebar branch and the mobile-Sheet branch of the shadcn primitive render zero pixels. Bottom 5-slot bar and mobile top header behave exactly as today.

**`lg` and up (1024 px and wider):** the wrapper switches to `display: contents`, allowing `<Sidebar>` to slot into the `SidebarProvider` flex layout as a peer of `<SidebarInset>`. The `<div className="lg:hidden">` around `<Navigation />` hides it. The `<TopStrip>` (which is `hidden lg:flex` on its root `<header>`) becomes visible. Content fills the remaining space.

The `lg:contents` Tailwind utility is included in v3.4 (the project uses 3.4.17). No config changes needed.

**No tablet breakpoint.** Between 768 and 1023 px (md but not lg), the user sees the mobile chrome. This is deliberate ‚Äî the rail at narrower widths is cramped, and a manager opening the app on an iPad in landscape (1024+) gets the desktop chrome anyway.

## Accessibility

- `SidebarTrigger` already has `<span className="sr-only">Toggle Sidebar</span>` ‚Äî keep it.
- The breadcrumb `<nav>` has `aria-label="Breadcrumb"`.
- Every nav link is a real anchor (via `Link` + `asChild`). Keyboard tab order flows: SidebarTrigger ‚Üí breadcrumb links ‚Üí page content ‚Üí sidebar links (the rail comes last in DOM order, which is fine for sighted users; screen readers can use the landmarks: `<aside>` for the rail, `<main>` for content, `<header>` for the strip).
- `SidebarMenuButton` exposes `tooltip` for collapsed-rail labels ‚Äî pass `tooltip={leaf.label}` on every leaf so screen readers and hover users both get text.
- Color is never the only signal: badges have a number; active state has a background fill and an `data-active="true"` attribute alongside its color.
- 4.5:1 contrast on all text ‚Äî the shadcn `bg-sidebar` / `text-sidebar-foreground` pair clears this; verify after first render and adjust if any custom color slips below.
- Cmd+B / Ctrl+B keyboard shortcut is built into `SidebarProvider`; verify it still fires when the provider is in controlled mode (it should ‚Äî shadcn calls `setOpen` which calls our `setOpenProp`).

## Risks and Rollback

**Risk: shadcn Sidebar's internal `useIsMobile` (768 px breakpoint) renders a Sheet drawer between 0‚Äì767 px.** Our wrapper is `hidden lg:contents`, so the Sheet renders into a `display: none` parent. At < 768 the Sheet would render via portal if `openMobile` were ever set true ‚Äî but `openMobile` only flips when the SidebarTrigger is clicked, and the SidebarTrigger only renders in TopStrip which is `hidden lg:flex`. Net: no leaked Sheet. Verify in 6C.

**Risk: the `peer` selector relationship that shadcn relies on might break with our wrapper.** Looking at the primitive, `peer` is used only for `SidebarRail` styling (which lives inside Sidebar's outer div) and for SidebarMenuBadge sizing (scoped within MenuItem). Neither requires Sidebar to be a top-level sibling of SidebarInset. Variant `inset` would have used peer-data on SidebarInset; we use variant `sidebar` (default), which doesn't.

**Risk: a sub-slice ships and breaks the build because TypeScript can't see a path alias.** The project uses `@/` (mapped in `tsconfig.json` and `vite.config.ts`) and `@assets/` (mapped in `vite.config.ts` only). The existing `Navigation` imports the logo via `@assets/...`; do the same. No new path aliases.

**Risk: a routed page that's only reachable via the sidebar is broken in production.** Each new sidebar entry should be visited at least once in the test pass for each role. Specifically: Parts Pending Approval (Company Admin), Wet Check Reviews (both), Wet Check Issue Types (Billing Manager ‚Äî new route), all four Reports leaves (both), Invoices (Company Admin ‚Äî new route).

**Rollback path:** revert the three new files plus the four lines of route additions. The app returns to its current navigation. No data side effects.
