  (`artifacts/irrigopro/vite.config.ts:98`).

The PDF is generated server-side under `artifacts/api-server` via
puppeteer, where the Vite `@assets` alias is not available — so the asset
must be embedded as a base64 data-URI (consistent with how photos and the
company logo are pre-embedded in the PDF).

## The change

Replace the placeholder body of
`artifacts/api-server/src/assets/vrt-logo.ts` so `VRT_LOGO_DATA_URI`
reads the real PNG at module load and base64-encodes it:

```typescript
import { readFileSync } from 'fs';
import { join } from 'path';

// Same asset the web app's powered-by footer uses
// (@assets/FINAL-02_1754876518960.png → repo-root attached_assets/).
// Resolve relative to the compiled api-server output; adjust the hop
// count if the build layout differs. Surface which path you used.
const VRT_LOGO_PATH = join(
  import.meta.dirname, '..', '..', '..', '..',
  'attached_assets', 'FINAL-02_1754876518960.png',
);

let dataUri = '';
try {
  const b64 = readFileSync(VRT_LOGO_PATH).toString('base64');
  dataUri = `data:image/png;base64,${b64}`;
} catch (err) {
  console.warn('[PDF] VRT logo asset not found; header falls back to text-only label', err);
  dataUri = '';
}

export const VRT_LOGO_DATA_URI = dataUri;
```

The `partsBlockForWetCheckBS` header from #1150 already renders the logo
only when the data-URI is non-empty (it should — confirm the `<img>` is
guarded by `VRT_LOGO_DATA_URI ?` and falls back to the text-only label
when empty). If #1150 did not guard it, add the guard now so a missing
asset degrades to the plain "Repairs Summary — …" label instead of a
broken image.

If resolving the repo-root path from the build output proves fragile,
copy the PNG into `artifacts/api-server/src/assets/` as a build step and
read it from there. Reuse the real file either way — do not regenerate
or recreate the logo.
