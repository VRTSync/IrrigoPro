# Public Approval Page ‚Äî Show the Full Estimate Detail

**What & Why**
The customer-facing approval page (the token link) shows a **stripped-down** estimate: a parts-only line-item table and a single lump **Total** that doesn't reconcile (e.g. ~$85 of parts under a **$1,434.75** total ‚Äî the labor is hidden). No scope of work, no labor breakdown, no location/map, and inspection estimates aren't zone-grouped. A customer can't tell what they're approving or why the total is what it is.

Bring the approval page to **full parity with the estimate PDF** ‚Äî everything from the estimate, not just the parts list.

**Root cause:** `pages/estimate-approval.tsx` renders only Part/Qty/Unit/Total columns and `Total {totalAmount}`, omitting labor and subtotals, even though the data (per-item `laborHours`, `totalLaborHours`, `laborRate`, `workDescription`, location, photos) is largely available. The `view-by-token` payload signs photos and returns items but should be confirmed to include everything below.

Frontend-heavy; small backend payload completeness check. Public token page ‚Äî already token-scoped (no auth), returns only that estimate.

## Done looks like
The approval page shows, like the PDF:
- **Scope of Work / work description** (full text).
- **Line items with labor** ‚Äî each line shows part, qty, unit, **labor (hours + $)**, line total.
- **Totals that reconcile** ‚Äî **Parts subtotal**, **Labor subtotal** (hours √ó rate), **Total** ‚Äî so the big number is explained.
- **Location** ‚Äî project address **and** the pinned spot with a "View on map" link (lat/lng), as on the email.
- **Photos** ‚Äî site photos and any inspection/finding photos, signed for the unauthenticated viewer.
- **Attachments** ‚Äî listed/downloadable.
- **Inspection estimates render zone-grouped** (Repairs-Summary-by-zone ‚Üí per Controller¬∑Zone), matching the estimate detail/PDF; standard estimates use the improved flat table with labor.
- Existing branded header/footer, secure-link note, Approve (‚Üí signature)/Decline unchanged.

## Steps
1. **Line items + totals** (`estimate-approval.tsx` ~720‚Äì755): add a **Labor** column (hours + $) to the line-item table, and replace the single Total with a **Parts subtotal / Labor subtotal (h √ó rate) / Total** block ‚Äî reuse the same computation as the estimate email redesign so the math matches the PDF. The total must equal parts + labor.
2. **Scope of Work**: render `workDescription` (and any scope/notes) as its own section above line items.
3. **Location**: render `projectAddress` **and** the pinned spot ‚Äî `workLocationAddress` + a Google Maps "View on map" link from `workLocationLat/Lng` (mirror the email's location block).
4. **Photos**: ensure all relevant photos (site + inspection/finding photos tied to the estimate) are included and pre-signed by `view-by-token`; render the gallery (already present) ‚Äî just make sure the data flows for inspection-origin estimates too, not only manually attached site photos.
5. **Zone grouping for inspection estimates**: when the estimate is inspection-origin (items carry `controllerLetter`/`zoneNumber`/`issueType`), render the **zone-grouped** layout (reuse `groupEstimateItemsByZone` / the shared `EstimateZoneGroupedView`) instead of the flat table ‚Äî same as the in-app estimate detail and PDF.
6. **Backend payload check** (`view-by-token` in `estimate-routes.ts`): confirm the response includes everything the page now needs ‚Äî per-item `laborHours`, `totalLaborHours`, `laborRate`, `workDescription`, `projectAddress`, `workLocationLat/Lng/Address`, `controllerLetter/zoneNumber/issueType`, all signed photo URLs, attachments, and the totals (or enough to compute them). Add any missing field. Keep it token-scoped (no extra data leakage beyond this estimate).
