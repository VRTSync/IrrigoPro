# Command Center Inline Edit — Slice 1: Inline Edit, Emergency Rate Toggle, Wet Checks Tab

**Source:** product feedback (Randy, 2026-06-03) — "edit flow for billing sheets and work orders needs to be much simpler than going through the wizard again. Optimizing for speed and accuracy in editing."  
**Severity:** P1 (workflow blocker — managers currently re-enter the wizard for every correction)  
**Prerequisite slices:** none (independent of any in-flight WC Manager Experience slice)

This is a single self-contained slice. Do not deviate from the scope below. If a question arises mid-implementation that is not answered here, stop and flag it rather than guessing.

---

## Design Context (locked decisions)

1. **"Command Center" = `/billing/command-center` = `billing-workspace.tsx`.** The page name in code (`BillingWorkspacePage`) is legacy; the UI label and the user's mental model are "Command Center." Do not rename the route or the file.
2. **Inline edit replaces the wizard hop for the 95% case.** Today the detail drawer (`DetailPane` inside `billing-workspace.tsx`) only exposes a "note" textarea and an "Open" link that boots the full wizard. The wizard is heavyweight and contains controls (technician, work date, customer pin, AI description, photo capture) that managers do not need when correcting an already-submitted ticket. The wizard remains available via "Open" — it is not removed — but inline edit becomes the default path.
3. **Editable in place (this slice):**
   - Line items: labor hours per line (when `laborMode === 'per_part'`), parts qty, parts unit price.
   - Add/remove line item rows.
   - Per-ticket labor rate **mode** (normal ↔ emergency), which swaps the rate value used for the whole ticket.
   - `totalHours` (the flat-mode authoritative hours field) when `laborMode === 'flat'`.
4. **Out of scope (this slice):**
   - Header field edits (work date, technician, customer, work description). Those still require the wizard. Document this in the UI with a "Edit header (wizard)" link for now.
   - Photos/attachments add/remove.
   - Multi-ticket bulk edit.
   - Per-line emergency-rate overrides. The toggle is per-ticket only; do not introduce a per-line override.
5. **Emergency rate source is the customer record.** `customers.emergencyLaborRate` already exists (default `125.00`, see `lib/db/src/schema/schema.ts:85`). `customers.laborRate` (line 84, default `45.00`) is the normal rate. The toggle does not let the user type a new emergency value at the ticket — if a customer's emergency rate is wrong, fix it in the customer record (existing `/api/customers/:id/labor-rates` endpoint).
6. **Rate mode persistence.** A new column, `rateMode text not null default 'normal'`, is added to `billing_sheets`, `work_orders`, and `wet_check_billings`. Values: `'normal' | 'emergency'`. The toggle flips this column and triggers a server-side total recompute that pulls the corresponding rate from the customer record. We do NOT add a separate `emergency_labor_rate` snapshot — the rate snapshot already lives in `laborRate` / `appliedLaborRate`. After a toggle the snapshot is overwritten with the customer's current emergency (or normal) value.
7. **Lock semantics are unchanged.** Inline edit, line-item edit, and the rate-mode toggle are all blocked when `status === 'billed'` or `invoiceId != null`. After `approved_passed_to_billing`, only `billing_manager`, `company_admin`, and `super_admin` can edit (mirrors the existing PATCH guard at `routes.ts:10778` for billing sheets and `routes.ts:11966` for work orders). Field techs never see inline edit affordances; they continue to use their normal completion flow.
8. **Wet Checks tab in Command Center** is the same list/detail pattern as the existing `WetCheckBillings` page (`artifacts/irrigopro/src/pages/wet-check-billings.tsx`) but lives as an additional tab inside the Command Center. The existing standalone `/wet-check-billings` route stays, unchanged. The new tab simply mounts a slim variant of the same list + the modal in place.
9. **Audit trail.** Every inline-edit save and every rate-mode flip writes an entry to the existing billing-sheet / work-order activity log (`/api/billing-sheets/:id/activity`, `/api/work-orders/:id/activity`). Use the same `recordActivity` helper that the PATCH endpoints already use. For wet-check billings, no activity endpoint exists today — record nothing for now (note this as a follow-up).

## Slice 1.1 — Schema: `rate_mode` column

### Scope
Add `rate_mode` to three tables. Backfill all existing rows to `'normal'`.

### Files added
- `lib/db/migrations/0006_add_rate_mode.sql` (latest existing is `0005_wcb_approval_snapshots.sql`).

### Files modified
- `lib/db/src/schema/schema.ts` — add `rateMode` column to `billingSheets`, `workOrders`, `wetCheckBillings`.

### Concrete changes

In `schema.ts`, inside `billingSheets` (around line 260, next to `appliedLaborRate`):

```ts
// 'normal' | 'emergency' — which customer-level rate this ticket bills at.
// Toggled from Command Center. Locked once the ticket is billed/invoiced.
rateMode: text("rate_mode").notNull().default("normal"),
```

Same column on `workOrders` (around line 694) and `wetCheckBillings` (around line 1410).

Migration SQL:

```sql
ALTER TABLE billing_sheets
  ADD COLUMN IF NOT EXISTS rate_mode text NOT NULL DEFAULT 'normal';
ALTER TABLE billing_sheets
  ADD CONSTRAINT billing_sheets_rate_mode_chk CHECK (rate_mode IN ('normal','emergency'));

ALTER TABLE work_orders
  ADD COLUMN IF NOT EXISTS rate_mode text NOT NULL DEFAULT 'normal';
ALTER TABLE work_orders
  ADD CONSTRAINT work_orders_rate_mode_chk CHECK (rate_mode IN ('normal','emergency'));

ALTER TABLE wet_check_billings
  ADD COLUMN IF NOT EXISTS rate_mode text NOT NULL DEFAULT 'normal';
ALTER TABLE wet_check_billings
  ADD CONSTRAINT wet_check_billings_rate_mode_chk CHECK (rate_mode IN ('normal','emergency'));
```

### Acceptance
- All three tables have a `rate_mode` column, `NOT NULL`, default `'normal'`, with a CHECK constraint limiting values.
- Existing rows are `'normal'` after backfill.
- Drizzle types include `rateMode: 'normal' | 'emergency'` (or `string` if Drizzle doesn't narrow it — the API layer will narrow).

## Slice 1.2 — Server: rate-mode toggle + line-item PATCH endpoints

### Scope
Three new endpoints, each mirroring the slim, totals-recomputing pattern of `PATCH /api/wet-check-billings/:id/labor-rate` (`routes.ts:9773` → `storage.recomputeWcbTotalsForLaborRate` at `storage.ts:4044`).

### Files modified
- `artifacts/api-server/src/routes/routes.ts` — three new routes.
- `artifacts/api-server/src/storage.ts` — three new storage methods.

### New endpoints

```
PATCH /api/billing-sheets/:id/rate-mode      { mode: 'normal' | 'emergency' }
PATCH /api/work-orders/:id/rate-mode         { mode: 'normal' | 'emergency' }
PATCH /api/wet-check-billings/:id/rate-mode  { mode: 'normal' | 'emergency' }
```

Each:
1. Auth: role must be `billing_manager`, `company_admin`, or `super_admin`. Otherwise 403.
2. Tenant guard: same pattern as the existing labor-rate route (super_admin bypass, others scoped to `authenticatedUserCompanyId`).
3. Lock guard: 409 if `status === 'billed'` or `invoiceId != null`.
4. Lookup the customer for this ticket; read `customers.laborRate` and `customers.emergencyLaborRate`. If the customer record is missing (legacy data with `customerId = null` on `wet_check_billings`), 422 with `"Customer record required to switch rate mode"`.
5. Choose the new rate: `mode === 'emergency' ? emergencyLaborRate : laborRate`.
6. In one transaction: update the ticket's `rateMode`, `laborRate`, and `appliedLaborRate` to the new rate; recompute `laborSubtotal` and `totalAmount` from the existing `totalHours` (or, for per-part mode, `Σ items.laborHours`); write activity log for billing-sheet / work-order paths.
7. Return the updated record.

### New endpoints — line-item edit (drawer-driven)

These are NOT separate granular routes. The existing `PATCH /api/billing-sheets/:id` and `PATCH /api/work-orders/:id` both accept `items` in the body and replace-all on update (see `routes.ts:10784` and `routes.ts:11972`). Inline edit reuses these by sending only the touched fields plus the full items array; no new endpoint is needed.

**However,** for the drawer UX to feel inline (no full-record validation failures when the user changes one cell), add one helper:

```
PATCH /api/billing-sheets/:id/items   { items: [{ id?, partId?, partName, quantity, unitPrice, laborHours, notes }] }
PATCH /api/work-orders/:id/items      { items: [{ id?, partId?, partName, partPrice, quantity, laborHours, notes }] }
```

Each:
1. Same auth + lock guards as PATCH `/api/billing-sheets/:id`.
2. Replace-all the items rows in a transaction (same as the existing line-item replace logic at `storage.ts:3818`).
3. Recompute `partsSubtotal`, `laborSubtotal`, `totalAmount` from the new items + the ticket's current `laborRate` and `laborMode`.
4. Return the updated ticket-with-items shape.

### Storage methods to add (signatures)

```ts
recomputeBillingSheetTotalsForRateMode(
  id: number,
  mode: 'normal' | 'emergency',
  companyId: number | null,
): Promise<BillingSheetWithItems>;

recomputeWorkOrderTotalsForRateMode(
  id: number,
  mode: 'normal' | 'emergency',
  companyId: number | null,
): Promise<WorkOrder>;

recomputeWcbTotalsForRateMode(
  id: number,
  mode: 'normal' | 'emergency',
  companyId: number | null,
): Promise<WetCheckBilling>;

replaceBillingSheetItems(
  id: number,
  items: Array<{ id?: number; partId?: number | null; partName: string; quantity: number; unitPrice: number; laborHours: number; notes?: string | null }>,
  companyId: number | null,
): Promise<BillingSheetWithItems>;

replaceWorkOrderItems(
  id: number,
  items: Array<{ id?: number; partId?: number | null; partName: string; partPrice: number; quantity: number; laborHours: number; notes?: string | null }>,
  companyId: number | null,
): Promise<WorkOrderWithItems>;
```

Throw the same coded errors used elsewhere: `WCB_LOCKED`, `WCB_NOT_FOUND`, `WCB_CROSS_COMPANY` for WCB; analogous `BS_LOCKED` / `WO_LOCKED` / etc. for billing sheets / work orders.

### Tests

- `artifacts/api-server/src/routes/rate-mode-routes.test.ts` (new):
  - Role gate: `field_tech` PATCH → 403; `billing_manager` → 200.
  - Lock gate: ticket with `status='billed'` → 409.
  - Tenant gate: cross-company ticket → 403.
  - Toggle to `emergency` flips `rateMode`, `laborRate`, `appliedLaborRate` to `customer.emergencyLaborRate`, and `totalAmount` recomputes.
  - Toggle back to `normal` restores `customer.laborRate`.
  - Missing customer → 422.
- `artifacts/api-server/src/routes/line-items-patch.test.ts` (new):
  - Replacing items recomputes `partsSubtotal` and `totalAmount`.
  - Items request on billed ticket → 409.

### Acceptance
- All six endpoints exist, are guarded, and behave as above.
- Existing `PATCH /api/billing-sheets/:id` and `PATCH /api/work-orders/:id` are unchanged — inline edit code never sends a full-record PATCH; it only uses the new slim endpoints.
