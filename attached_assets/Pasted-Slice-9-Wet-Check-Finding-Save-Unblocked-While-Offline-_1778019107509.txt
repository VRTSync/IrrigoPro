Slice 9 — Wet Check Finding Save Unblocked While Offline Queue Drains
Date: 2026-05-05
Status: Ready for Replit implementation
Author: Field-tech bug triage
Type: Bug fix (UI gate only — no schema, API, or storage changes)
Estimated effort: 15–30 minutes
Related: Builds on Slice 4 (Offline Support for Field Wet Checks) and Slice 2 (Wet Check System). Does not change any of the offline engine, mutation queue, mirror schema, or server endpoints. Pure UI fix in wet-checks.tsx.

Executive Summary
Field techs adding work needed to consecutive zones see intermittent failures: the Save Finding button works on one zone, is greyed out on the next, then works again if they wait a few seconds. The cause is a UI gate in FindingSheet (inside client/src/pages/wet-checks.tsx) that disables Save until the server zoneRecordId arrives. With the offline queue enabled (the default since Slice 4), a freshly tapped "Needs Work" zone exists in the IndexedDB mirror with a clientId only — the server id only appears once the queued zone_record.upsert mutation drains over the network. While the queue is still draining, the button is disabled even though the save mutation itself can already complete the operation by routing through offlineCreateFinding with the {{zr}} placeholder.
Slice 9 ships when: the Save Finding button is enabled the moment a zone has been tapped "Needs Work" (online or offline), regardless of whether the server id has propagated back yet, and the related photo controls inside the FindingSheet stop disappearing during the same window.

Why Now
Multiple field techs have hit the issue. The symptom — "works on this zone, doesn't work on the next, works again if I wait" — is exactly the shape of a queue-drain race. It's a one-line gate change with no behavioral risk to online-only callers (offline queue disabled → behavior unchanged). The save logic itself already handles the clientId-only case correctly; only the disabled gate is wrong.

Root Cause
In client/src/pages/wet-checks.tsx, the FindingSheet component:

Save Finding button (around line 1692) is disabled when zoneRecordId (server id) is null in create mode.
Pre-save photo capture button (around line 1624) is hidden when zoneRecordId is null.
Empty-state message (around line 1637) reads "Pick a zone before adding photos" when zoneRecordId is null.

When the offline queue is enabled, setStatus.mutate("checked_with_issues") calls offlineUpsertZoneRecord, which writes a row to wetCheckZoneRecords in IndexedDB with clientId only and enqueues a zone_record.upsert mutation. The wet-check query invalidates and refetches immediately, the preset buttons render, but the zone record's id is still undefined until the queue drains. Tapping a preset opens FindingSheet → Save is greyed out.
Inside saveMut.mutationFn for mode === "create", the offline branch is gated correctly:
tsif (isOfflineQueueEnabled() && zoneRecordClientId) {
  await offlineCreateFinding({
    zoneRecordClientId,
    zoneRecordId: zoneRecordId ?? undefined,
    wetCheckId,
    payload: { ...payload, issueType },
    clientId: findingClientId,
  });
}
offlineCreateFinding enqueues a finding.create mutation with {{zr}} → zoneRecordClientId placeholder, which the engine resolves to the server id when the parent zone_record.upsert drains. The save flow is correct end-to-end. Only the button disabled gate is wrong.

Scope of Slice 9
In Scope

Update the Save Finding button's disabled prop in FindingSheet to accept zoneRecordClientId as sufficient when isOfflineQueueEnabled() is true.
Update the pre-save photo capture button's visibility gate in FindingSheet (the one inside the !editing branch of the photo block) with the same condition.
Update the empty-state message ("Pick a zone before adding photos.") with the same condition so it reads correctly when a zone has been picked but only has a clientId.
No changes to saveMut.mutationFn, offlineCreateFinding, the offline engine, or any server-side code.

Out of Scope

Auto-invalidating React Query when the offline engine successfully drains a zone_record.upsert / finding.create / wet_check.create (separate future improvement — would close a class of "stale id" UI bugs but is not required for this fix).
React key collisions on zoneRecord.findings.map(f => <div key={f.id} … />) when multiple findings exist with id === undefined (separate cleanup; switch to f.clientId ?? f.id).
Anything related to wet-check submit, auto-bill, manager review, or photo upload internals.

Acceptance Criteria

With the offline queue enabled (default), a tech can tap "Needs Work" on a zone, immediately tap a preset, fill out the FindingSheet, and tap Save Finding with no waiting required — the button is enabled the moment the zone-record exists in the IDB mirror.
The pre-save photo capture button is visible the moment the zone is picked, regardless of whether the server zoneRecordId has propagated back.
The "Pick a zone before adding photos" empty-state message no longer appears when a zone is selected but only has a clientId.
Online-only mode (offline queue flag disabled) is unchanged — Save still requires zoneRecordId.
Edit mode is unchanged — Save was never gated in edit mode.
Repeated rapid use across multiple zones in a row produces no greyed-out Save button on any zone.

Risk Assessment
Low. Three local conditional changes inside one component. The save mutation logic, offline engine, mirror schema, and server endpoints are untouched. Online-only mode is unchanged. The offline path was already handling clientId-only zone records correctly in saveMut.mutationFn, offlineCreateFinding, and the engine's placeholder resolution — only the UI gates were lagging.

Out-of-Scope Followup (Recommend a Separate Slice)
The offline engine's applyServerIdToMirror writes the server-assigned id back to the IDB mirror but does not invalidate the React Query cache. Consider adding an engine.on("state", …) listener (e.g., in conflict-toast-bridge.tsx, which already mounts globally) that calls queryClient.invalidateQueries(["/api/wet-checks"]) when the pending/syncing count drops. This would make any future "stale id" UI bugs self-heal without per-component gates.
