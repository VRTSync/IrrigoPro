Master scope paragraph for Replit

We need to harden Irrigo’s QuickBooks Online connection so it stays healthy and does not appear to disconnect prematurely. The goal is not to fight Intuit’s token lifetimes, but to build a reliable token-management layer that correctly handles 1-hour access tokens, rolling 100-day refresh tokens, refresh-token rotation, and failure scenarios like invalid_grant, stale token use, duplicate refresh attempts, and revoked connections. The system must always store and use the latest token pair returned by Intuit, refresh in a controlled and serialized way per QuickBooks company (realmId), and surface a clean reconnect-required state only when reauthorization is truly necessary.

#QB1 - Audit the current QuickBooks token lifecycle
What & Why

Before changing behavior, identify exactly why the connection feels like it disconnects fast. The most common causes are stale refresh-token storage, concurrent refresh attempts, old SDK client state, or weak error handling around token refresh. Intuit explicitly warns that refresh-token values can rotate and that duplicate exchanges/refreshes can cause invalid_grant.

Done looks like
Replit maps the full current token flow
every storage location for access and refresh tokens is identified
all refresh triggers are listed
any concurrent refresh paths are identified
any stale-token or partial-save risks are identified
any SDK/client instances using cached token objects are identified
Tasks
Trace the full OAuth lifecycle from initial connect to refresh to disconnect handling
Identify where realmId, access token, and refresh token are stored
Identify whether the newest refresh token is always persisted after refresh
Identify whether more than one worker/request can refresh the same realm at once
Identify whether the app waits for 401s only or refreshes proactively
Identify how revoked or expired connections are detected and surfaced
Return confirmed findings before implementation
#QB2 - Canonical token storage and metadata model
What & Why

The app needs one authoritative QuickBooks connection record per connected company. Intuit’s guidance is to always store and use the latest access and refresh tokens from the most recent response.

Done looks like
one canonical token record exists per realmId
token metadata is complete enough to debug failures
token updates can happen atomically
token/company mixups are prevented
Tasks

Create or normalize a QuickBooks connection model that stores at minimum:

realmId
access token
refresh token
access token expiry
last refresh attempt
last refresh success
last refresh failure
connection status
reconnect-required reason
token source environment (sandbox vs production if relevant)

Also ensure:

records are uniquely scoped by realmId
token writes can occur atomically
token reads for API calls always resolve through this canonical source
#QB3 - Safe refresh pipeline with per-realm locking
What & Why

Intuit warns that multiple refresh attempts using the same code/token can cause invalid_grant, and stale or reused refresh tokens are a common failure source. Refresh must happen one at a time per company.

Done looks like
only one refresh can run at a time per realmId
concurrent API requests do not stampede refresh
stale token use is prevented
refresh failures are deterministic and diagnosable
Tasks
Implement a per-realmId refresh mutex/lock
Ensure only one refresh request can run at a time for a company
Make waiting requests reuse the newly refreshed token instead of starting their own refresh
Add timeout/failure protection so locks cannot hang indefinitely
Ensure stale refresh-token reuse cannot happen after a successful refresh rotates the token
#QB4 - Atomic token rotation handling
What & Why

Intuit says refresh-token values can change and recommends always storing the latest token pair from the most recent server response. Partial writes are dangerous.

Done looks like
every successful refresh stores both the new access token and new refresh token together
no old refresh token remains active in app state after a successful save
partial update failure cannot leave the system inconsistent
Tasks
On refresh success, atomically persist:
new access token
new refresh token
new access expiry
refresh success timestamp
Ensure in-memory client state is also updated immediately after persistence
Fail the refresh transaction if both token values cannot be safely saved together
Remove any logic that preserves the old refresh token after refresh success
#QB5 - Proactive refresh strategy
What & Why

Access tokens last 1 hour, while refresh tokens roll forward up to 100 days and extend when used. Refreshing proactively is safer than waiting for failures everywhere.

Done looks like
the system refreshes before access-token expiry
ordinary API calls rarely hit expired-token failures
refresh tokens remain in active use and do not silently age out
Tasks
Add a pre-expiry refresh threshold
If access token is near expiry, refresh before making the QBO API request
Preserve fallback handling for unexpected 401s
Avoid refreshing on every request
Ensure refresh frequency is sensible and not wasteful
#QB6 - Reconnect-required state and failure classification
What & Why

Not every token failure means the same thing. Expired access token, stale refresh token, revoked authorization, wrong environment keys, and invalid_grant need different handling. Intuit documents that revoked access and expired/unused refresh tokens require reauthorization.

Done looks like
the app distinguishes recoverable refresh failures from true reconnect-required failures
revoked/expired connections do not loop endlessly
users see a clean reconnect state only when necessary
Tasks
Classify refresh failures into:
retryable transient failure
stale token / invalid_grant
revoked connection
reconnect required
Mark the connection status accordingly
Stop repeated failing refresh loops when reconnect is required
Provide a clean UI/API signal that the QuickBooks connection must be reauthorized
#QB7 - Background health check / keep-alive job
What & Why

If a refresh token is not used for 100 days, it expires and the user must reauthorize. A health job reduces silent expiration risk for low-activity accounts.

Done looks like
inactive but valid connections are periodically checked
low-activity companies do not silently fall off due to non-use
unhealthy connections are surfaced before billing runs fail
Tasks
Add a scheduled health-check job for active QuickBooks connections
Refresh or validate connections that are approaching inactivity risk
Flag unhealthy connections early
Avoid aggressive refresh spam; keep it controlled and realm-specific
#QB8 - Observability and diagnostics
What & Why

Without logs and metrics, QuickBooks disconnects look random. This slice makes failures explainable.

Done looks like
every refresh attempt is traceable
token failures can be diagnosed by realm and failure type
support/debugging becomes fast instead of guesswork
Tasks

Log and track:

refresh attempt start/end
realmId
success/failure
failure code/message
token age
whether token rotated
whether request waited on another refresh lock
whether reconnect-required was set

Also add:

a compact admin/debug view of QuickBooks connection health
last successful refresh timestamp
last failure reason
#QB9 - End-to-end regression test plan for token resilience
What & Why

This cannot ship on faith. It needs scenario-based verification.

Done looks like
core refresh scenarios are covered
duplicate refresh behavior is proven safe
rotated refresh-token handling is proven safe
reconnect-required behavior is proven correct
Tasks

Test these cases:

initial authorization stores token set correctly
normal proactive refresh
access token expired, refresh succeeds
refresh token rotated, newest token persists
two simultaneous requests for same realm trigger only one refresh
stale old refresh token is not reused
revoked connection moves to reconnect-required
sandbox/prod credential mismatch surfaces clearly
inactive connection nearing expiry is handled safely
API client uses latest in-memory token after refresh
Recommended build order

Do them in this order:

#QB1 audit current lifecycle
#QB2 canonical token storage
#QB3 per-realm refresh lock
#QB4 atomic rotation handling
#QB5 proactive refresh
#QB6 reconnect-required classification
#QB8 observability
#QB7 health check job
#QB9 regression testing

That order fixes correctness first, then durability, then visibility.
