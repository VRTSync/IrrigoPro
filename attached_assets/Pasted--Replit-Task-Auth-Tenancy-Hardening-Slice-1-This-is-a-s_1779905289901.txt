# Replit Task — Auth & Tenancy Hardening, Slice 1

This is a single self-contained task. The **Design Context** is in
`Auth-Tenancy - Slice 0 Hotfix.md` — re-read it before starting. The
actual work for this task is in the **Slice 1** section below. Do not
deviate from any file paths, function names, schema field names, or
payload shapes specified. If you believe a deviation is necessary,
surface it and stop rather than proceeding.

> **Architectural note.** Slice 1 closes the largest cross-tenant
> exposure on the work-order surface without touching the schema. We
> introduce one new middleware — `requireSameCompanyAsWorkOrder` —
> that loads the target row, derives its companyId via the customer
> join, and rejects when it doesn't match `req.authenticatedUserCompanyId`.
> We also wrap every bare-auth work-order route with
> `requireAuthentication` so unauthenticated callers stop getting
> through. Storage is unchanged in this slice (Slice 3 refactors
> storage signatures). The `companyId` column on `work_orders` is
> added in Slice 4 — until then the guard derives tenancy via the
> existing `customerId → customers.companyId` link.

**Concrete changes.**

### 1. Add `requireSameCompanyAsWorkOrder` middleware

Place adjacent to `requireWorkOrderUpdateAccess`. Exact shape — do not
deviate:

```ts
// Tenant guard for work-order routes. Loads the work order by
// req.params.id, derives its companyId via the customer link, and
// rejects with 403 when it doesn't match req.authenticatedUserCompanyId.
// Super admins (companyId === null) bypass the check.
//
// Until Slice 4 lands the work_orders.companyId column, the row's
// tenant is read through the existing customerId → customers.companyId
// path. Slice 4 swaps that one fetch for a direct column read.
const requireSameCompanyAsWorkOrder = async (req: any, res: any, next: any) => {
  const userRole = req.authenticatedUserRole;
  if (userRole === 'super_admin') {
    return next();
  }
  const userCompanyId = req.authenticatedUserCompanyId;
  if (userCompanyId == null) {
    res.status(403).json({ message: "Company scope required for this action." });
    return;
  }
  const workOrderId = parseInt(req.params.id);
  if (!Number.isFinite(workOrderId) || workOrderId <= 0) {
    res.status(400).json({ message: "Invalid work order ID" });
    return;
  }
  try {
    const workOrder = await storage.getWorkOrder(workOrderId);
    if (!workOrder) {
      res.status(404).json({ message: "Work order not found" });
      return;
    }
    // Derive tenant via customer link. Slice 4 will replace this with
    // a direct workOrder.companyId read once the column is backfilled.
    const customer = workOrder.customerId
      ? await storage.getCustomer(workOrder.customerId)
      : null;
    const workOrderCompanyId = customer?.companyId ?? null;
    if (workOrderCompanyId == null) {
      // A work order with no resolvable tenant cannot be served
      // safely to a non-super-admin. Treat as 404 — never reveal
      // existence of a row outside the caller's tenant.
      res.status(404).json({ message: "Work order not found" });
      return;
    }
    if (workOrderCompanyId !== userCompanyId) {
      // Never reveal that the row exists in another tenant. Always
      // respond with 404 so cross-tenant probing returns the same
      // shape as a genuinely missing ID.
      res.status(404).json({ message: "Work order not found" });
      // Audit the attempt — wired up properly in Slice 7. For now,
      // a single structured console line is enough.
      console.warn(
        `[AUDIT] cross_tenant_work_order_access ` +
        `actor=${req.authenticatedUserId} actorCompany=${userCompanyId} ` +
        `targetWorkOrder=${workOrderId} targetCompany=${workOrderCompanyId} ` +
        `route=${req.method} ${req.path}`
      );
      return;
    }
    // Cache the loaded row so the handler can reuse it without a
    // second storage round trip. Handlers are not required to read
    // this — they may still call storage.getWorkOrder(id) for clarity
    // — but the field is available.
    req.tenantScopedWorkOrder = workOrder;
    next();
  } catch (error) {
    console.error('Error checking work-order tenant scope:', error);
    res.status(500).json({ message: "Couldn't verify tenant scope — please retry." });
    return;
  }
};
```

Note the **404-not-403 on cross-tenant.** This is intentional. A 403
leaks existence ("the row exists, you just can't see it"). A 404 makes
cross-tenant probing indistinguishable from random ID guessing. The
single audit line still gives us server-side visibility.

### 2. Wrap every route in the table above

For routes that already have `requireAuthentication`, slot
`requireSameCompanyAsWorkOrder` *immediately after* it and *before* the
existing role guard:

```ts
// before
app.patch("/api/work-orders/:id", requireAuthentication, requireWorkOrderUpdateAccess, async (req, res) => { ... });

// after
app.patch("/api/work-orders/:id", requireAuthentication, requireSameCompanyAsWorkOrder, requireWorkOrderUpdateAccess, async (req, res) => { ... });
```

For routes with no current auth at all, the full chain is:

```ts
// before
app.get("/api/work-orders/:id", async (req, res) => { ... });

// after
app.get("/api/work-orders/:id", requireAuthentication, requireSameCompanyAsWorkOrder, async (req, res) => { ... });
```

For `POST /api/work-orders` (create — no `:id`), add only
`requireAuthentication` (not the new guard) and *inside* the handler
add the line:

```ts
// Force-tenant the new row to the caller's company. A super_admin
// creating a work order must pass an explicit companyId in the body
// — handled by the policy engine in Slice 5.
if (req.authenticatedUserRole !== 'super_admin') {
  workOrderBody.companyId = req.authenticatedUserCompanyId;
}
```

This is harmless until Slice 4 lands the column (the field is silently
ignored by the insert), but the line is in place so Slice 4 only has
to add the schema column and Drizzle picks it up.

### 3. Update the existing handlers that ignore tenant on read

Two handlers currently re-fetch the work order and serve it without a
tenant check. Both fall under the wrap list above, but verify after
mounting the middleware that the handler bodies are still correct:

- `GET /api/work-orders/:id` (routes.ts:11543) — handler is fine; the
  guard runs before and returns 404 on mismatch.
- `PATCH /api/work-orders/:id` (routes.ts:11680) — handler is fine.

Do NOT remove the existing `storage.getWorkOrder(id)` calls inside
handlers — they remain the authoritative read and may be used to
catch races (the guard's snapshot is from before the handler ran).
`req.tenantScopedWorkOrder` is a cache, not a replacement.
