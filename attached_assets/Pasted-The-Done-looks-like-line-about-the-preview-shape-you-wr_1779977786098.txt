The "Done looks like" line about the preview shape — you wrote "ADD COLUMN company_id (nullable) on three tables + DROP NOT NULL on estimates.company_id. No truncate cascade."
The DROP NOT NULL on estimates.company_id line won't actually appear. Reason: production's estimates.company_id is already nullable in the database (per the diagnostic query you ran — only estimates came back, with is_nullable = YES). The current schema's .notNull() is what was making drizzle-kit push propose ALTER TABLE estimates ... SET NOT NULL. When the slice relaxes the schema to nullable, schema and DB then agree on estimates, so Drizzle Kit shows no diff for that table at all.
The actual expected preview is:

ADD COLUMN company_id (nullable, with FK to companies) on work_orders, billing_sheets, invoices.
No changes to estimates.
No DROP TABLE web_sessions (the new schema declaration handles it).
No truncate cascade anywhere.

I'd reword that line in the ticket to:

The Replit deploy preview for this slice shows only safe operations: ADD COLUMN company_id (nullable, FK to companies) on work_orders, billing_sheets, and invoices. No changes to estimates. No DROP TABLE web_sessions. No truncate cascade anywhere.

Won't cause data loss. The truncate line is the only one that's catastrophic and worth gating the deploy on.

One thing worth noting that's not in the ticket but is worth knowing as you watch the build: the applyCompanyIdColumns() startup migration in storage.ts from commit 00ce687 stays in place. The slice deliberately doesn't remove it. Both code paths (startup + admin page) share the app_settings marker ('company-id-columns-v1'), so whichever runs first wins and the other becomes a no-op.
