No schema changes — work_orders already has workLocationLat, workLocationLng, workLocationAddress, controllerLetter, zoneNumber (all nullable).Executive SummaryThe current work order create flow (client/src/components/work-orders/work-order-form.tsx) is structured as six labeled "Step 1–6" cards inside a single scrolling 95vh dialog — already a wizard mentally, just not gated. Location capture is optional, the map pin is buried inside a toggle that defaults to closed, no controller/zone selectors exist anywhere, and once a work order is created the pin is invisible and uneditable in edit-work-order-modal.tsx. The dispatch list (pages/work-orders.tsx line 1057) ignores the lat/lng entirely and routes the field tech to the customer's billing address text via Google/Apple Maps. Field techs have no way to refine the pin once they're physically at the work area.Slice 10 replaces WorkOrderForm with a five-step WorkOrderWizard that mirrors the EstimateWizard structure. Step 2 is Work Location & Site and Continue is gated until a map pin exists. The dispatch list switches to lat/lng-first navigation. The completion screen gets an "I'm here" button that captures the tech's current geolocation as the refined work pin. The edit modal exposes the pin so it can be corrected post-creation.Slice 10 ships when: every existing entry point opens the new wizard, Step 2 cannot be skipped without a pin, the map pin is visible and editable in edit-work-order-modal.tsx, the dispatch directions button uses the pin when present, the "I'm here" affordance writes to workLocationLat/Lng/Address, and the submit payload remains identical to what the server's existing POST /api/work-orders and PUT /api/work-orders/:id endpoints expect.Why NowThree concrete failures are driving this:First, techs are routed to the wrong place. The customer's billing address is often a corporate office, an HOA management address, or the front gate of a commercial property. The actual irrigation work is a half-mile of pipe away. Managers correctly pin the precise spot during creation, but pages/work-orders.tsx line 1064 throws that data on the floor and sends the tech to projectAddress instead. Every miss costs 15–30 minutes of windshield time.Second, billing accuracy depends on knowing where work happened. Auto-billed wet-check findings, manager review queues, and missing-photos reports all want a single source of truth for "where was this work performed." Right now the lat/lng sits in the database but nothing reads it.Third, the manager's mental model already matches a wizard. The form is six labeled "Step N" cards. Wizardizing it costs the same screen real estate, makes Step 2 truly required, and lets us delete the long single-form scroll.Slice 8 made this slice unblockable: the wizard pattern, container chrome, sticky header/footer, dirty-tracking + discard-confirm, and Esc handling are all proven and reusable. Slice 10 borrows the structure, swaps the steps to match work-order content, and adds the "I'm here" affordance.What a Work Order Creation Flow BecomesFive steps inside the same wizard container the estimate wizard uses (full-screen on mobile, centered dialog ≥768px, sticky header with progress bar, sticky mobile footer):Step 1 — Customer & Branch. Customer picker (existing CustomerSelector) at the top. Once a customer is locked, an inline display card shows name + address with a "Change customer" link. Branch selector renders only if customer.branches?.length > 0, required when shown. Customer email and phone are auto-populated from the customer record but editable inline (matches the Slice 9 estimate-wizard pattern with the helper text "Used for this work order only — won't update the customer record"). CTA: "Continue to Location →".Step 2 — Work Location & Site. The required step. Three discrete sub-cards inside the step body:
Address & Notes — LocationFields for project address, location notes, access instructions. "Use customer address" / "Use a different address" toggle works the same as in the estimate wizard.
Pin on Map (required) — LocationPicker, opened by default when no pin exists, collapsed when one does. Pinned-location confirmation uses the blue accent (matching Slice 9's color rationalization, not green). A required-asterisk on the section header and inline helper text below the picker: "Drop a pin so the field tech can navigate straight to the work area. Required to continue."
Controller & Zone — two selects driven by /api/properties/:customerId/controllers. Optional. Loading state distinguishes "loading…" from "no controllers on file." Auto-clear when the customer changes.
CTA strip: "← Back" and "Continue to Description →". The Continue button is disabled until workLocation !== null. Inline helper text below explains why when disabled (no tooltip, per Slice 9).Step 3 — Description & Priority. Sticky strip at top: customer/project/branch summary + a "Pinned at: {address}" chip with a small "Edit pin" link that jumps back to Step 2. Below that: description (required), special instructions, internal notes, priority select. CTA: "← Back" / "Continue to Schedule →".Step 4 — Schedule & Assign. Sticky strip same as Step 3. Body: scheduled-date picker, assignee select (managers and field techs grouped), tags (existing tags text array if it's used; otherwise omit). CTA: "← Back" / "Continue to Review →".Step 5 — Review & Submit. Read-only summary of everything: customer/branch block, location block (showing the pinned address + a small map thumbnail rendered from the existing map component if cheap, otherwise just the address + lat/lng), controller/zone if set, work description, scheduling, assignee. Optional collapsible "Site Photos" section using the existing FileUpload. Two CTAs: secondary "← Back to Schedule" and primary "Create Work Order" (or "Save Changes" in edit mode). On submit: POST /api/work-orders (or PUT /api/work-orders/:id). On success: toast "Work order created" (or "Work order updated"), wizard closes, the calling page's React Query cache invalidates ["/api/work-orders"].Edit mode opens on Step 2 (the most-likely-edit step is location, given the existing data quality issue).Scope of Slice 10In Scope

  const isIos = /iPhone|iPad|iPod|Android/i.test(navigator.userAgent);
  if (wo.workLocationLat != null && wo.workLocationLng != null) {
    const lat = wo.workLocationLat;
    const lng = wo.workLocationLng;
    const label = encodeURIComponent(wo.workLocationAddress ?? wo.projectAddress ?? wo.customerName ?? "Work site");
    return isIos
      ? `https://maps.apple.com/?ll=${lat},${lng}&q=${label}`
      : `https://www.google.com/maps/search/?api=1&query=${lat},${lng}`;
  }
  const encoded = encodeURIComponent(wo.projectAddress ?? "");
  return isIos
    ? `https://maps.apple.com/?q=${encoded}`
    : `https://www.google.com/maps/search/?api=1&query=${encoded}`;

Acceptance Criteria
Every existing "New Work Order" entry point opens <WorkOrderWizard>. Step 1 lands on Customer & Branch.
The Continue button on Step 2 is disabled while workLocation === null. The disabled state shows inline helper text, not a hover tooltip.
On Step 2, the map pin can be dropped, cleared, and re-dropped, and the address auto-populates if the existing LocationPicker returns one.
Controller & Zone selects on Step 2 are populated from /api/properties/:customerId/controllers. Selecting a controller enables the zone select with the correct number of options.
Edit mode (<WorkOrderWizard workOrderId={n}>) opens directly on Step 2 with the existing pin pre-loaded and editable.
Submit produces the exact same payload shape the current WorkOrderForm produces, with workLocationLat, workLocationLng, workLocationAddress always present, and controllerLetter/zoneNumber populated when set.
The directions button on pages/work-orders.tsx (and every other place that opens a maps URL for a work order) navigates to the lat/lng pin when present, falling back to projectAddress otherwise.
The "I'm here" button on work-order-completion.tsx captures the tech's geolocation, persists it, and surfaces an undo toast. The customer-facing projectAddress is unchanged.
Pressing Enter inside any text or number input does NOT advance the wizard (Slice-9 fix applied).
Discarding via Esc or click-outside while dirty prompts the discard-changes dialog with the same wording as the estimate wizard's, branched for create vs edit.
Online and offline behavior of the existing flows is unchanged. No server-side changes required beyond what the existing endpoints already accept.
