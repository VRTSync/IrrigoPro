# Replit Task — WC Billing Integration, Slice 4 (PDF format parity snapshot)

This is a single self-contained task. The **Design Context** below is locked
decisions from the broader integration audit — it is reference material so
you understand the intent. The actual work for this task is in the **Slice 4**
section at the bottom. Do not deviate from any file paths, function names,
schema field names, or payload shapes specified. If you believe a deviation
is necessary, surface it and stop rather than proceeding.

> **No production code changes. Test-only slice.** This is insurance. The
> `ticketPageWCB` (`pdf-helpers.ts:347`) and `ticketPageBS`
> (`pdf-helpers.ts:258`) helpers are intentionally siblings and share most
> of their structure, but no test pins that contract today. A future edit
> to either helper that breaks the parity would only be caught by eye on
> a real customer invoice — which is the wrong place to catch it.

> **Pattern: assert the *intentional* parity, not blanket equality.** The
> two templates have three deliberate differences:
>   1. The header title (`Billing Sheet #…` vs `WC Billing #…`)
>   2. BS has an optional WORK PERFORMED section between the header and
>      the financial breakdown; WCB does not.
>   3. The parts block is always WC-style on WCB; on BS it's WC-style
>      only when `bs.wetCheckView` is present.
> Everything else — the outer chrome, header structure, FINANCIAL
> BREAKDOWN block, totals row, approval block, photo-fail warning, photo
> gallery — should match. The new test asserts the match-set while
> tolerating the difference-set.

## Design Context (locked decisions)

These were agreed in the audit
(`WC Billing Integration - Slice 0 End-of-Month Audit.md`) and apply to
every slice in this family.

**The two helpers will continue to evolve in parallel.** When the post-Task
#766 financial breakdown layout shipped on BS, it was also applied to WCB
(see the comment at `pdf-helpers.ts:342`). Future format work on either
should generally land on both. The parity test makes that drift visible
in CI.

**The shared section labels are the contract surface.** The CSS class
names (`ticket-page`, `ticket-header`, `ticket-section`,
`ticket-financial`, `ticket-fin-row`, `ticket-fin-total`,
`ticket-approval`) and the literal section labels (`FINANCIAL
BREAKDOWN`, `TOTAL`) are what customers visually parse. The test
asserts on those, not on prose copy that may legitimately drift.

**Snapshots are stored in-repo and reviewed in PR.** No external
snapshot service. The snapshot file lives next to the test file. PR
reviewers see snapshot diffs in `git diff` and accept or reject the
change deliberately.

## Slice 4 — Add a PDF ticket-page parity assertion

### Scope

A single new test file that:
1. Builds a fixture `PdfBillingSheetRow` and `PdfWetCheckBillingRow`
   with overlapping data (same labor, same parts, same hours, same
   approval state, same photos).
2. Renders each via `ticketPageBS` and `ticketPageWCB`.
3. Asserts the shared structural-element set is present in both.
4. Snapshots each rendered HTML to a `.snap` file for visual diff
   review.
5. Asserts the three intentional differences exist (so accidentally
   *removing* a difference is also caught).

No production helper is modified. No new helper is added.

### Files modified

- New: `artifacts/api-server/src/pdf-ticket-parity.test.ts` — the test
  file
- New: `artifacts/api-server/src/__snapshots__/pdf-ticket-parity.test.ts.snap`
  — committed snapshot file (generated by first test run)

#### New test file

```typescript
// artifacts/api-server/src/pdf-ticket-parity.test.ts
//
// Slice 4 — Pins the parity contract between ticketPageWCB
// (pdf-helpers.ts:347) and ticketPageBS (pdf-helpers.ts:258). The two
// templates are intentional siblings; this test catches accidental
// drift between them in CI rather than on a real customer invoice.

import { describe, it, expect } from "vitest";
import { ticketPageBS, ticketPageWCB } from "./pdf-helpers";
import type {
  PdfBillingSheetRow,
  PdfWetCheckBillingRow,
} from "./pdf-view-model";

// ─── Fixtures ────────────────────────────────────────────────────────

const APPROVED_AT = new Date("2026-05-15T14:30:00Z");
const WORK_DATE = new Date("2026-05-10T09:00:00Z");
const INVOICE_NUMBER = "INV-2026-0042";
const LOGO = null;
const COMPANY = "Test Irrigation Co.";

function bsFixture(overrides: Partial<PdfBillingSheetRow> = {}): PdfBillingSheetRow {
  return {
    billingSheetId: 101,
    billingSheet: {} as any, // unused by ticketPageBS at the field level
    billingNumber: "BS-2026-0020",
    customerName: "Acme Farms",
    propertyAddress: "123 Oak Ridge Rd",
    workDate: WORK_DATE,
    technicianName: "J. Smith",
    technicianId: 7,
    totalHours: "3",
    laborRate: "85",
    laborSubtotal: "255.00",
    partsSubtotal: "100.00",
    rowTotal: "355.00",
    workDescription: "Replaced two spray heads on zone 4.",
    aiDetailedDescription: null,
    approvedBy: "Manager Mary",
    approvedAt: APPROVED_AT,
    items: [
      // Minimal items list — exact shape depends on PdfBillingSheetRow.items
      // contract. Use one part item.
      { partName: "1804-PRS40", description: "Spray head", quantity: "2", unitPrice: "50.00", totalPrice: "100.00", laborHours: "0", laborRate: "85", laborTotal: "0" } as any,
    ],
    wetCheckView: undefined, // ensures BS uses partsTableFromBS, not WC-style
    branchName: undefined,
    ...overrides,
  };
}

function wcbFixture(overrides: Partial<PdfWetCheckBillingRow> = {}): PdfWetCheckBillingRow {
  return {
    wetCheckBillingId: 202,
    wetCheckBilling: {
      billingNumber: "WC-2026-1042",
      customerName: "Acme Farms",
      propertyAddress: "123 Oak Ridge Rd",
      workDate: WORK_DATE,
      technicianName: "J. Smith",
      technicianId: 7,
      totalHours: "3",
      laborRate: "85",
      appliedLaborRate: "85",
      laborSubtotal: "255.00",
      partsSubtotal: "100.00",
      totalAmount: "355.00",
      approvedBy: "Manager Mary",
      approvedAt: APPROVED_AT,
      branchName: null,
    } as any,
    wetCheckView: {
      // Minimal WetCheckBillingView; one zone, one finding.
      zones: [
        {
          zoneId: 1,
          zoneLabel: "Zone 4",
          repairLaborHours: 3,
          findings: [
            { findingId: 11, description: "Spray head replacement", partName: "1804-PRS40", quantity: 2, unitPrice: "50.00", totalPrice: "100.00" },
          ],
        },
      ],
    } as any,
    zonePhotoGroups: undefined,
    mergedPhotoUrls: undefined,
    ...overrides,
  };
}

// ─── Tests ──────────────────────────────────────────────────────────

describe("PDF ticket-page parity — BS and WCB (Slice 4)", () => {
  const bsHtml = ticketPageBS(bsFixture(), INVOICE_NUMBER, [], LOGO, COMPANY);
  const wcbHtml = ticketPageWCB(wcbFixture(), INVOICE_NUMBER, [], LOGO, COMPANY);

  // ─── Shared structural elements ─────────────────────────────────

  const SHARED_CLASSES = [
    "ticket-page",
    "ticket-header",
    "ticket-header-bs",      // WCB reuses the BS header class intentionally
    "ticket-header-line1",
    "ticket-header-line2",
    "ticket-header-line3",
    "ticket-approval",
    "ticket-approval-by",
    "ticket-approval-at",
    "ticket-section",
    "ticket-financial",
    "ticket-fin-rows",
    "ticket-fin-row",
    "ticket-fin-total",
    "ticket-fin-label",
    "ticket-fin-value",
  ];

  const SHARED_LABELS = [
    "FINANCIAL BREAKDOWN",
    "Irrigation Labor",
    "Parts Subtotal",
    "TOTAL",
    "Approved By:",
    "Approved At:",
  ];

  it.each(SHARED_CLASSES)("both templates include CSS class %s", (cls) => {
    expect(bsHtml).toContain(`class="${cls}`); // matches both `"x"` and `"x y"` after the quote
    // Simpler: just match presence
    expect(bsHtml).toMatch(new RegExp(`class="[^"]*\\b${cls}\\b`));
    expect(wcbHtml).toMatch(new RegExp(`class="[^"]*\\b${cls}\\b`));
  });

  it.each(SHARED_LABELS)("both templates include literal label %s", (label) => {
    expect(bsHtml).toContain(label);
    expect(wcbHtml).toContain(label);
  });

  // ─── Intentional differences ────────────────────────────────────

  it("BS header title says 'Billing Sheet #…'; WCB header says 'WC Billing #…'", () => {
    expect(bsHtml).toContain("Billing Sheet #BS-2026-0020");
    expect(wcbHtml).toContain("WC Billing #WC-2026-1042");
    expect(bsHtml).not.toContain("WC Billing #");
    expect(wcbHtml).not.toContain("Billing Sheet #");
  });

  it("BS includes WORK PERFORMED section; WCB does not", () => {
    expect(bsHtml).toContain("WORK PERFORMED");
    expect(wcbHtml).not.toContain("WORK PERFORMED");
  });

  it("WCB parts block is zone-grouped; BS parts block is item-table when wetCheckView is absent", () => {
    // The zone-grouped block has a recognizable wrapper class from
    // partsBlockForWetCheckBS. The flat parts table has a different
    // wrapper. Assert by the class names emitted by each helper.
    expect(wcbHtml).toMatch(/class="[^"]*\b(wc-zone-block|wcb-zone-block|zone-block)\b/);
    expect(bsHtml).toMatch(/<table[^>]*>/); // BS without wetCheckView uses a flat parts table
  });

  // ─── Snapshots ──────────────────────────────────────────────────

  it("matches BS ticket page snapshot", () => {
    expect(bsHtml).toMatchSnapshot();
  });

  it("matches WCB ticket page snapshot", () => {
    expect(wcbHtml).toMatchSnapshot();
  });

  // ─── Cross-template totals row sanity ───────────────────────────

  it("both templates render the same TOTAL value for identical financial inputs", () => {
    // Strip HTML and look for $355.00 in the totals row neighborhood
    expect(bsHtml).toMatch(/TOTAL.*\$355\.00/s);
    expect(wcbHtml).toMatch(/TOTAL.*\$355\.00/s);
  });
});
```

**Notes for the implementer.**

- The exact field shape of `PdfBillingSheetRow` and
  `PdfWetCheckBillingRow` is defined at `pdf-view-model.ts:111+` and
  the related interfaces above. **Before writing the fixtures, read
  those types** and adjust the fixture object literals to satisfy
  TypeScript strictly. Do NOT use `as any` to satisfy the compiler
  except on the deeply-nested
  `billingSheet`/`wetCheckBilling`/`wetCheckView` payloads where the
  ticket helpers only read a few specific keys.
- The `SHARED_CLASSES` and `SHARED_LABELS` lists are the **contract
  surface**. If a future code change intentionally renames a class or
  label on one template, the failing test forces a deliberate update
  on both templates plus this list — that's the desired friction.
- The fourth difference test (zone block vs flat table) uses a
  regex against the class wrapper emitted by
  `partsBlockForWetCheckBS` (defined nearby in `pdf-helpers.ts`). If
  the actual class name doesn't match the regex's options, look at
  `partsBlockForWetCheckBS` and update the regex to the actual
  emitted class name. Do not invent a new class.
- Use `toMatchSnapshot()` (built-in to vitest with no config). The
  resulting `.snap` file goes in
  `artifacts/api-server/src/__snapshots__/pdf-ticket-parity.test.ts.snap`
  and is committed. PR review of the snapshot diff is the human
  guardrail.
