## Slice 1 ‚Äî Stop new duplication: row-lock the wet-check conversion (the root cause)
**File:** `artifacts/api-server/src/storage.ts` ‚Äî `convertWetCheckToWetCheckBilling` (fn ~9780; `db.transaction` ~9791; `allFindings` loaded ~9802; deferred branch ~9817‚Äì10130).

**Problem.** This transaction reads `allFindings`, filters `eligible` (excludes `convertedAt != null` / `workOrderId != null`), then inserts a `work_order_items` batch ‚Äî with **no row lock** on the wet check. Two concurrent calls for the same wet check (double-click "Submit Review," slow request + client retry, two tabs) both pass the eligibility check before either commits and both insert a full batch ‚Üí duplicate items. Contrast `approveEstimateAndCreateWorkOrder` (~3845), which takes `SELECT ‚Ä¶ FOR UPDATE` on the estimate row precisely to serialize this.

**Fix.** The function **already** loads the wet-check row as the first statement in the transaction (`storage.ts:9792`), company-scoped, before `allFindings`. Do **not** add a second select ‚Äî just append `.for("update")` to the existing one. (Note: the param is `id`, not `wetCheckId`.)
```ts
return await db.transaction(async (tx) => {
  // Serialize concurrent conversions of THIS wet check (mirrors
  // approveEstimateAndCreateWorkOrder's estimate row lock ~3845). The second
  // waiter blocks here, re-reads the row AFTER the first commits ‚Äî findings are
  // stamped (workOrderId/convertedAt) and excluded by `eligible`, so it inserts
  // nothing. Closes the race without touching the incremental-append design.
  const [wc] = await tx.select().from(wetChecks)
    .where(and(eq(wetChecks.id, id), eq(wetChecks.companyId, companyId)))
    .for("update");                                 // <-- only change: add .for("update")
  if (!wc) throw new Error(`Wet check ${id} not found for company ${companyId}`);
  // ...existing status check, cust load, allFindings, etc. unchanged
```
- `and`/`eq` are already imported and used here; the existing where-clause keeps the `companyId` tenancy scope. The second caller re-reads fresh status under the lock, so the existing status guard rejects it if the first conversion flipped the wet check out of `submitted`/`partially_converted`; otherwise `eligible` finds nothing.
- Apply the identical lock to any sibling convert transaction that appends children off the same wet check (the sent-to-estimate / repaired-in-field branches, if they run in their own tx) so the whole conversion surface is serialized.
- **Do NOT** add a zone-scoped delete-before-insert ‚Äî it would drop legitimately incremental (multi-session) items and silently under-bill. The lock is the correct and sufficient fix.

**Done:** two concurrent conversions of one wet check produce exactly one item batch; incremental multi-session triage still appends correctly.

## Slice 1B ‚Äî Stop the SECOND duplication vector: completion must REPLACE items, not append
**File:** `artifacts/api-server/src/routes/routes.ts` ‚Äî `/api/work-orders/complete` (~9482‚Äì9497), and `work-order-completion.tsx` (prefill ~301).

**Problem (deterministic doubling ‚Äî separate from Slice 1's race).** The completion screen prefills `usedParts` **from the WO's existing `work_order_items`** (`work-order-completion.tsx:301`, `source: 'estimate'`). On submit, the handler loops `usedParts` and calls `storage.addWorkOrderItem(...)` for each (`routes.ts:9483`) **without deleting the pre-existing items** ‚Äî so it writes a *second copy* of every part. Every completed estimate-/finding-origin WO doubles its parts at completion, independent of the Slice-1 concurrency race. (This is why WO #26 is ~3.46√ó, not a clean 2√ó/3√ó ‚Äî completion-copy **plus** race-copies.)

**Fix.** Completion should treat the final `usedParts` set as **the** authoritative WO item set and REPLACE, not append:
- Build the final item list from `usedParts` (both `source: 'estimate'` and `field_added`), each with `totalPrice = money(quantity) √ó money(partPrice)` and `findingId` where known (the prefilled ones carry it once Slice 3 lands; `field_added` = null).
- Write them via `replaceWorkOrderItemsWithResync(workOrderId, finalItems, companyId)` (~storage.ts:4905) ‚Äî delete existing + insert + recompute in one tx ‚Äî instead of the `for (part of usedParts) addWorkOrderItem(...)` loop.
- Net: the WO's stored items = exactly what the tech confirmed in the field (the actuals), once. No pre-existing copy survives, no doubling. `field_added` parts are preserved because they're part of `usedParts`.
- **Guard:** if `usedParts` is somehow empty on a completion that had prefilled items, do **not** wipe the WO to zero ‚Äî treat empty-with-prior-items as "no change" and log, rather than deleting real parts.

**Done:** completing a work order leaves exactly one row per confirmed part; re-opening/re-submitting completion does not add copies.

## Slice 2 ‚Äî Close the completion NaN vector
**File:** `artifacts/api-server/src/routes/routes.ts` ‚Äî `/api/work-orders/complete` (~9429‚Äì9435).

**Problem.** Still unguarded:
```ts
laborHours   = parseFloat(totalHours || '0');       // "NaN" is truthy ‚Üí NaN
const partsCost   = parseFloat(totalPartsCost || '0');
const laborSubtotal = laborHours * appliedLaborRate; // NaN rate ‚Üí NaN
const totalAmount   = laborSubtotal + partsCost;
```
`parseFloat("NaN" || '0')` is `NaN` (the `|| '0'` only catches null/empty), and a NaN `appliedLaborRate` propagates ‚Äî writing `NaN` totals on completion.

**Fix.** Route every operand through the existing `money()` helper (`lib/money.ts`) and store `.toFixed(2)`:
```ts
laborHours = money(totalHours);
const partsCost    = money(totalPartsCost);
const rate         = money(appliedLaborRate);
const laborSubtotal = laborHours * rate;
const partsSubtotal = partsCost;
const totalAmount   = laborSubtotal + partsSubtotal;   // finite, = parts + labor
```
Also apply `money()` to the per-part `laborHours √ó quantity` sums just above (~9414‚Äì9423) for consistency. **Done:** no completion submission can persist a `NaN` total.

## Slice 3 ‚Äî Add lineage columns to `work_order_items` (detect + reverse this class forever)
**Files:** `lib/db/src/schema/schema.ts` (`workOrderItems`), + Drizzle migration.

**Add** two nullable columns:
- `findingId: integer("finding_id").references(() => wetCheckFindings.id)` ‚Äî the finding that produced this row (null for non-inspection items).
- `createdAt: timestamp("created_at").defaultNow()` ‚Äî insert time.

**Why.** Today there's no way to key an item row back to its finding or its insert batch, which is exactly why duplication needed a forensic rebuild. With `findingId`, a duplicate is trivially detectable (`COUNT(*) > 1 per (workOrderId, findingId)`) and future repairs are a one-liner. **Populate `findingId` going forward** in the Slice-1 conversion insert (the deferred loop already iterates findings ‚Äî stamp each item with its `f.id`). Backfill is not required for the repair (Slice 4 rebuilds from findings directly), but stamping new inserts closes the gap. Consider the same two columns on `estimate_items` / `billing_sheet_items` for parity (optional, same rationale).

## Slice 4 ‚Äî Repair the corrupted production work orders (rebuild from findings) ‚Äî **runs only after Slice 1 is live**
**File:** new `artifacts/api-server/src/lib/migrations/repair-duplicated-work-order-items.ts`, registered in `registry.ts`; follow the `MigrationDefinition` contract (`check`/`preview`/`run` + `appSettingsKey`), mirroring `repair-nan-totals.ts` (per-row audit log, per-step try/catch, idempotent done-marker, acknowledgement gate since this changes money).

**De-duplicate ‚Äî do NOT rebuild from findings.** Because completion (Slice 1B) appends the tech's **actual** `usedParts` ‚Äî which can include `field_added` parts that map to *no finding* ‚Äî a rebuild purely from `deferred_to_work_order` findings would erase legitimate field-added work and revert tech edits ‚Üí under-bill. Historical rows also have no `findingId` yet (Slice 3 only stamps going forward), so you can't separate finding-derived from field-added after the fact. Therefore **collapse exact-duplicate rows** rather than regenerate; this removes the doubled copies while preserving every distinct part the tech actually confirmed.

**Detection (`check`/`preview`).** For each WO, group its `work_order_items` by identity ‚Äî `(partId, partName, partPrice, quantity, laborHours, controllerLetter, zoneNumber)`. A group with count > 1 is duplicated; the correct set keeps **one** row per group. Flag any WO with at least one such group. Report per-WO: current parts/labor/total ‚Üí de-duplicated parts/labor/total, and the delta (**negative ‚Äî a reduction**; call this out in the acknowledgement gate, opposite of the add-parts NaN/reconcile repairs).

**Repair (`run`, per affected WO, transactional + wet-check row lock):**
1. Snapshot current `work_order_items` + WO header totals to a scratch table (or emit to the audit log) ‚Äî reversible.
2. Build the de-duplicated set: one row per identity group above (keep the lowest `id` of each group; drop the rest). This preserves genuinely distinct rows ‚Äî including `field_added` parts ‚Äî and removes only exact copies.
3. Write it via `replaceWorkOrderItemsWithResync(woId, dedupedItems, companyId)` (~storage.ts:4905) ‚Äî delete + insert + recompute in one tx.
4. Recompute and write header `partsSubtotal / laborSubtotal / totalAmount / totalItems` from the de-duplicated set (`total = parts + labor`, all `money()`).
5. **Reconciliation guard:** cross-check each WO's de-duplicated parts against its finding-derived total. If they don't reconcile (e.g., non-integer-multiple inflation, or field-added parts present), **do not auto-finalize ‚Äî flag the WO for manual review** and leave it out of the auto-billed set. Never silently overwrite a WO the algorithm can't fully explain.
6. Log before‚Üíafter per WO.

**Safety.**
- Company-scoped throughout (Rule 1).
- **Idempotent:** a clean re-run finds 0 duplicated WOs.
- **Run order in production:** (a) confirm Slice 1 is deployed; (b) run `reconcile-billing-sheet-invoice-totals` (BS-2026-0235); (c) confirm `repair-nan-totals` has run (NaN cleared); (d) run **this** repair last ‚Äî it supersedes the NaN migration's item-summed totals for the duplicated WOs and produces the truly-correct figures.
- **Verify WO-1781807328597-643 ("WO #26" Tanglewood)** lands at ~$2,125.20 parts + $1,520 labor. If it doesn't match the independently-derived figure, stop and reconcile before releasing billing.
- Hold billing on all flagged WOs until each rebuilt total is verified.

## Slice 5 ‚Äî Guardrail tests (prove it, keep it)
- **Concurrency regression (the key one):** fire two concurrent `convertWetCheckToWetCheckBilling` calls for the same wet check ‚Üí assert exactly one `work_order_items` batch is written and totals equal a single rebuild-from-findings. (Proves Slice 1.)
- **Incremental append still works:** defer finding A (convert), then defer findings B+C (convert) ‚Üí WO has A+B+C exactly once (no drop, no dup).
- **Completion replace-not-append (proves Slice 1B):** create a WO with items, open completion (usedParts prefilled), submit ‚Üí WO has exactly the confirmed set once (not doubled); re-submit completion ‚Üí still no copies; a `field_added` part is present exactly once.
- **Completion NaN:** POST `/complete` with `totalPartsCost = "NaN"` (or a NaN rate) ‚Üí stored total is finite and `= parts + labor`. (Proves Slice 2.)
- **Repair de-dup + preservation (proves Slice 4):** seed a WO with exact-duplicate rows **plus** one distinct `field_added` part ‚Üí run repair ‚Üí exact duplicates collapse to one each, the field-added part survives, `total = parts + labor`; re-run ‚Üí 0 changes. A WO that can't reconcile is flagged, not overwritten.
- **Property assertion** after every WO/estimate/BS money mutation: `Number.isFinite(totalAmount) && totalAmount === partsSubtotal + laborSubtotal`.
- **End-to-end:** one wet check with mixed findings ‚Üí estimate + deferred WO + repaired-in-field ‚Üí approve ‚Üí complete ‚Üí bill ‚Üí invoice ‚Üí PDF; assert finite & `total = parts + labor` at every stage, and a pre-invoice billing edit carries through unchanged to the PDF.
