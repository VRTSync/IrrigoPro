#88 - Fix Core Billing Calculation Discrepancy
What & Why

Work orders are completed in the field with a calculated total, but downstream systems discard that stored financial result and recompute using a stripped formula. This breaks financial continuity between field completion, billing manager review, invoice generation, and QuickBooks. The fix must establish a true source of truth for work order financials so that once a financial record is created in the field, it remains intact as it passes up the user chain.

Done looks like
The billing manager profile view shows the same total the work order carried after field completion or approved pre-billing edits
Invoice preview and invoice generation use the stored work-order financial snapshot, not a stripped recomputation
QuickBooks invoices reflect the stored work-order total
Work orders show a full breakdown: labor subtotal, parts subtotal, markup, tax, total
Hardcoded labor / markup / tax values are removed from work-order financial calculation paths
Work orders store the applied labor, markup, and tax rates used for that specific ticket at the time financials are calculated
Historical completed but unbilled work orders use stored totalAmount as the authoritative total unless an exact historical breakdown can be proven
Necessary schema additions

In addition to markupAmount and taxAmount, add:

appliedLaborRate
appliedMarkupRate
appliedTaxRate

Retain and populate:

laborSubtotal
partsSubtotal
totalAmount
Necessary calculation rule

When a work order financial total is first calculated, snapshot the effective customer contract values onto the work order. After that, downstream systems must use the work order’s stored financial snapshot, not the live customer record.

Necessary write-path rule

Any endpoint that can change work-order financial inputs before billing must recompute and persist the full financial breakdown using the work order’s stored applied rates. This includes, but is not limited to:

completion
manager edits to hours
manager edits to parts cost / parts rows
any billing-prep ticket edit flow that alters financial inputs
Historical backfill guardrail

For existing completed but unbilled work orders:

use stored totalAmount as authoritative
backfill detailed subtotals/rates only when exact historical applied rates are recoverable
do not derive historical markup/tax detail from current customer settings unless verified
Bottom line

I would keep #89 and #90 as-is.

I would approve #88 only after adding:

rate snapshot fields on the work order,
recompute on every financial write path,
a no-fabrication rule for historical backfill.

Those are not stylistic changes. They are the minimum needed to make the fix actually trustworthy.
