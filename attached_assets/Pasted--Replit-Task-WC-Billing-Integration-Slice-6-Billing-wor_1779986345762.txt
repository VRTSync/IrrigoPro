# Replit Task — WC Billing Integration, Slice 6 (Billing-workspace WCB queue support)

This is a single self-contained task. The **Design Context** below is locked
decisions from the broader integration audit — it is reference material so
you understand the intent. The actual work for this task is in the **Slice 6**
section at the bottom. Do not deviate from any file paths, function names,
schema field names, or payload shapes specified. If you believe a deviation
is necessary, surface it and stop rather than proceeding.

> **This slice expands the billing manager's daily dashboard surface.**
> Today, `/billing-workspace` is invisible to the wet check pipeline —
> every WCB awaiting approval lives on `/wet-checks/pending-review`
> (a manager-level surface). After this slice, the billing manager sees
> WCBs at the same lifecycle stages they see billing sheets: pending
> approval, approved this week, and unbilled WCBs queued for invoice.

> **Pattern: mirror the billing-sheet flow in this file exactly.** The
> backend route file (`billing-workspace-routes.ts`) and frontend page
> (`billing-workspace.tsx`) already model `billing_sheet` and
> `work_order` as parallel branches in every helper, queue collector,
> status-strip aggregator, and dispatch handler. The WCB branch slots
> in next to them with the same shape.

> **Read-first, write-second.** This slice surfaces WCBs in the queue
> (read) and dispatches approve/kickback through the existing
> wet-check-level endpoints (`POST /api/wet-checks/:id/approve` and the
> finding-route flow). It does NOT add parallel WCB-level approve/
> kickback endpoints — the approval flow stays single-sourced through
> the wet check itself. This matches the WC Separate System design
> context and keeps the audit trail in one place.

## Design Context (locked decisions)

These were agreed in the audit
(`WC Billing Integration - Slice 0 End-of-Month Audit.md`) and the WC
Separate System slices, and apply to every slice in this family.

**WCB approval flows through the parent wet check.** A WCB is created
when a wet check is auto-billed or routed-to-billing by a manager. Its
status transitions (`submitted` → `pending_manager_review` →
`approved_passed_to_billing` → `billed`) are driven by the wet check's
own approval path at `POST /api/wet-checks/:id/approve`
(`routes.ts:16015`). No parallel `/wet-check-billings/:id/approve` or
`/kickback` endpoint exists, and this slice does not introduce one.

**Status mapping.** For the queue, WCBs in
`{'submitted', 'pending_manager_review'}` are "active" (awaiting
approval), parallel to BS `ACTIVE_BS` and WO `ACTIVE_WO`. WCBs in
`{'approved_passed_to_billing', 'billed'}` are "approved" parallel to
BS `APPROVED_BS`. WCBs have no concept of "draft" — they auto-submit
on creation per WC Separate System Slice 6's design context —
so the "Drafts 24h" tile does not include WCBs.

**Billing-manager approval scope.** Per the May 2026 audit, the billing
manager is read-only on wet checks. The queue surfaces WCBs and lets
the billing manager click through to the wet check itself, where the
manager-role users do the approval. Slice 5 (parallel) decides
whether billing managers gain narrow edit rights (labor rate +
per-zone labor). The approve/kickback buttons in the queue stay
disabled for billing managers on WCB rows.

**Approve / kickback dispatch.** When a manager-role user (not a
billing manager) clicks Approve on a WCB queue row, the frontend
fires `POST /api/wet-checks/{wcb.wetCheckId}/approve` — the parent
wet check's endpoint. Kickback fires the existing finding-route
endpoint pattern. No new endpoints.

**Tenant-scoping note (important):** `wet_check_billings` has no
`companyId` column — verified at `schema.ts:1391+`. Scoping must go
through the customer join. The current `GET /api/wet-check-billings`

```typescript
async function scopedWetCheckBillings(req: any): Promise<any[]> {
  const role = req.authenticatedUserRole;
  const cid0 = req.authenticatedUserCompanyId ?? null;
  if (role === "super_admin") {
    return (await storage.getAllWetCheckBillingsWithCounts()) as any[];
  }
  if (cid0 == null) return [];
  // Join wet_check_billings -> customers and filter by companyId.
  // wet_check_billings.customerId is nullable (schema.ts:1394); any
  // row with a null customerId is dropped (it has no tenant anchor).
  const rows = await db
    .select({ wcb: wetCheckBillings })
    .from(wetCheckBillings)
    .innerJoin(customers, eq(wetCheckBillings.customerId, customers.id))
    .where(eq(customers.companyId, cid0))
    .orderBy(desc(wetCheckBillings.createdAt));
  return rows.map(r => r.wcb) as any[];
}
```

### Acceptance criteria

- `GET /api/billing-workspace/queue?type=wcb` returns only WCB rows
  in `ACTIVE_WCB` statuses; `type=all` returns the union.
- `GET /api/billing-workspace/status-strip` includes WCBs in
  `awaitingApproval` and `approvedThisWeek`, but NOT in
  `draftsLast24h`.
- Every WCB queue row carries a `wetCheckId` field for frontend
  dispatch.
- The page renders a `Wet Check Billings` chip in the type filter
  with a `Droplets` icon.
- A WCB row in the queue is clickable and renders the same row
  shape (number, customer, tech, total, status, age) as BS/WO rows.
- For a `billing_manager` role, the Approve and Kickback buttons are
  hidden on WCB rows.
- For `manager` / `company_admin` / `super_admin`, clicking Approve
  on a WCB row fires
  `POST /api/wet-checks/{wcb.wetCheckId}/approve` — the parent wet
  check's endpoint, not a new WCB endpoint.
- Tenant scoping is preserved: non-super-admin sees only their
  company's WCBs.
- All new and updated tests pass.
- `pnpm typecheck` and `pnpm test` pass.

### Out of scope

- Adding `/api/wet-check-billings/:id/approve` or
  `/api/wet-check-billings/:id/kickback` endpoints. The parent wet
  check is the source of truth.
- Adding WCBs to "Drafts last 24h" — no draft state exists.
- Adding a separate "WCB pending review" status tile parallel to the
  four existing tiles. The four-tile contract is intentional and
  shipped as Task #709. Adding a fifth tile is its own product
  decision.
- The QuickBooks tile's overdue logic — WCB invoicing flows through
  the same `/api/invoices/monthly` path as BS, so once invoiced the
  QB tile already reflects WCB-contributed totals.
- Changing the `/wet-checks/pending-review` page (the manager-role
  queue) — that surface stays as the WC-level review.
- Slice 5's edit affordances. The queue links out to
  `/wet-check-billings?id=…` (the list page), which itself opens the
  view modal where Slice 5's edit pencil lives. The two slices are
  independent surfaces.
