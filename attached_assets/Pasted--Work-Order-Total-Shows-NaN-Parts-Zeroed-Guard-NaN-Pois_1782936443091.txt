# Work Order Total Shows "$NaN" + Parts Zeroed â€” Guard NaN Poisoning Across Estimate â†’ Work Order â†’ Billing

**What & Why**
A work order that came through the pipeline **wet-check inspection â†’ estimate â†’ field completion** shows a headline total of **"$NaN"** with **Parts: $0.00** in the Command Center unbilled list (e.g. WO #26, Wellman-area, labor $1,520.00), even though its description lists dozens of parts. "$NaN" is JavaScript's "Not a Number" â€” a numeric field got corrupted into a non-number and every downstream total inherited it. This blocks the work order from being billed correctly (its total is unusable) and hides the real parts revenue.

## Root cause (confirmed in code)
One bad line item poisons the whole total, because number parsing is **unguarded** in the estimate total recompute â€” and the resulting `NaN` gets **persisted** and then **snapshotted** onto the work order.

1. **Origin â€” a line item with no/blank price becomes `NaN`.** The estimate carries adjustment / labor-only findings (the description is full of them: "Adjust MP Nozzle", "Bad Solenoid Fitting", "Pinch Clamp", "Decoder", etc.) that have no material price. When such an item's `totalPrice` is computed as `quantity Ă— partPrice` with a null/blank `partPrice`, the result is `NaN`. Postgres `decimal`/`numeric` columns **accept the special value `NaN`**, so `estimate_items.totalPrice` (a `NOT NULL` decimal) can and does store the literal string `"NaN"`.

2. **Estimate total recompute doesn't guard it.** `getEstimate` / `getEstimates` (`storage.ts:2448` and the singular variant) sum parts with:
   ```js
   const itemTotal = parseFloat(String(item.totalPrice)); // NO "|| 0"
   partsSubtotal += itemTotal;                             // += NaN  â†’ partsSubtotal = NaN
   ...
   const totalAmount = partsSubtotal + laborSubtotal;      // NaN + 1520 = NaN
   ```
   `parseFloat(String("NaN"))` is `NaN`, so `partsSubtotal` and `totalAmount` both become `NaN`. **Note the inconsistency:** other recompute paths in the same file already guard this â€” `storage.ts:2566` uses `parseFloat(String(item.totalPrice)) || 0`, `:4731` and `:4784` use `parseFloat(String(row.totalPrice || 0))`, and the field-completion path (`routes.ts:9414`) uses `(parseFloat(String(it.laborHours ?? '0')) || 0)`. The estimate read path is the unguarded outlier.

3. **The NaN is snapshotted onto the work order.** `createWorkOrderFromEstimate` (`storage.ts:3699â€“3704`) copies the estimate's totals verbatim: `partsSubtotal: estimate.partsSubtotal`, `totalAmount: estimate.totalAmount`. So the work order inherits `totalAmount = "NaN"` (and a broken parts figure). Labor is fine because it's computed from `hours Ă— rate`, not from the poisoned item sum â€” which is exactly why the card shows **clean labor $1,520.00 but total $NaN**.

4. **The card renders it faithfully.** Command Center builds the row with `parseFloat(wo.totalAmount || '0')` (`routes.ts:6323/6331`). The `|| '0'` guard only catches null/empty â€” it does **not** catch the string `"NaN"` (which is truthy), so `parseFloat("NaN") = NaN` prints as **"$NaN"** via `Intl.NumberFormat` (`customer-billing.tsx:805`). The huge parts text is the estimate's `workDescription` carried into the WO `description` (`storage.ts:3686`), unrelated to the numeric bug.

**Net:** a single unpriced line item â†’ `NaN` persisted in a decimal column â†’ unguarded estimate recompute â†’ snapshotted onto the WO â†’ "$NaN" total and zeroed parts. Same *class* as the billing-total drift bug (unguarded numeric handling), different spot.

## Slice 1 â€” Stop NaN at the source (never persist a non-number)
Any code that writes a money/number column must coerce to a finite number first. Add a tiny shared helper and use it everywhere totals/prices are written:
```js
const money = (v) => { const n = parseFloat(String(v)); return Number.isFinite(n) ? n : 0; };
```
- At estimate/work-order/billing **item creation** (where `totalPrice = quantity Ă— partPrice` is computed), coerce `quantity`, `partPrice`, and the product with `money()` â€” an unpriced adjustment line stores `0.00`, never `NaN`.
- Guard against writing `NaN`/`Infinity` into any `decimal` column app-wide (a write-time assertion or the `money()` wrapper on every `.toFixed(2)` total).

## Slice 2 â€” Guard every total recompute (make the reads NaN-proof)
- In `getEstimate` and `getEstimates` (`storage.ts:2448` + singular), change `parseFloat(String(item.totalPrice))` â†’ `money(item.totalPrice)` and `parseFloat(String(item.laborHours))` â†’ `money(item.laborHours)`. Same for the labor rate / hours reads so a bad rate can't yield `NaN` either.
- Sweep for the unguarded pattern and standardize on `money()`: `grep` for `parseFloat(String(` **without** an adjacent `|| 0` / `?? '0'`. Bring the outliers in line with the already-guarded paths (`:2566`, `:4731`, `:4784`, `routes.ts:9414`).
- **Guarantee:** no single line item can ever turn a subtotal or total into `NaN`.

## Slice 3 â€” Harden the snapshot + the display guard (defense in depth)
- `createWorkOrderFromEstimate` (`storage.ts:3699â€“3704`): run the snapshotted `partsSubtotal` / `laborSubtotal` / `totalAmount` through `money()` before insert, and recompute `total = money(parts) + money(labor)` rather than trusting the estimate's stored `totalAmount`. A poisoned estimate can't propagate into a work order.
- Command Center shaping (`routes.ts:6323`, `:6342â€“6344`, `:6371`) and `customer-billing.tsx:805` `formatCurrency`: treat non-finite as `0` (`Number.isFinite(n) ? n : 0`) so a legacy bad row degrades to `$0.00` (visibly wrong, fixable) instead of the alarming `$NaN`. Do **not** modify `customer-billing.tsx` beyond this display guard if it can be avoided â€” prefer fixing the data server-side; flag if the guard must live in the page.

## Slice 4 â€” Repair the corrupted data (sweep, not one-off)
Other estimates / work orders that went through the same path are silently corrupted too (you just haven't opened them). Add an admin migration (Super Admin â†’ Migrations, re-runnable/idempotent, preview beforeâ†’after):
- Scan `estimate_items`, `work_order_items`, `billing_sheet_items` for `totalPrice IS NULL OR totalPrice = 'NaN'` (Postgres: `totalPrice = 'NaN'::numeric` or `isnan(totalPrice)`); recompute each from `quantity Ă— partPrice` with `money()`, writing `0.00` where price is genuinely absent.
- Recompute `partsSubtotal` / `laborSubtotal` / `totalAmount` on the parent `estimates` and `work_orders` from the repaired items (parts from item sum, labor from hours Ă— rate), using **add-parts semantics** (bill the correct amount; never silently lower a real total) per the billing-total memory rule.
- Confirm **WO #26** â†’ total becomes `parts + labor` (real number), parts reflects the actually-priced valves/nozzles, and it becomes billable. Log before/after; report any rows that had `NaN`.

## Slice 5 â€” Guard test (lock the class)
- Behavioral test: create an estimate with one normally-priced part **and** one adjustment line with `partPrice = null` â†’ assert `partsSubtotal`, `laborSubtotal`, `totalAmount` are all finite numbers (the priced part's value, not `NaN`).
- Convert that estimate to a work order â†’ assert the WO's snapshotted totals are finite and equal `parts + labor`.
- Property assertion: after any estimate/WO/billing mutation, `Number.isFinite(totalAmount) === true` and `total === parts + labor`.
