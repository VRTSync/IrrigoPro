### Concrete changes

#### Step A — Reproduce on `WC-2026-1027`

On a dev environment or a production database snapshot:

1. Locate the wet check. `SELECT * FROM wet_checks WHERE id = (SELECT
   wet_check_id FROM wet_check_billings WHERE billing_number =
   'WC-2026-1027');`. Record the `customer_id` and (via
   `customers.company_id`) the company id.
2. Inspect catalog rows for that company:
   ```sql
   SELECT * FROM issue_type_configs WHERE company_id = :cid;
   ```
   If zero rows → Hypothesis 1 confirmed. If rows exist but their
   `issue_type` values look unusual (capitalized, spaced, etc.) →
   continue to step 3.
3. Inspect findings for that wet check:
   ```sql
   SELECT DISTINCT f.issue_type, COUNT(*) AS n
   FROM wet_check_findings f
   JOIN wet_check_zone_records zr ON zr.id = f.zone_record_id
   WHERE zr.wet_check_id = :wcId
   GROUP BY f.issue_type;
   ```
   Compare each value to the catalog keys from step 2. If they
   differ — Hypothesis 2 confirmed.
4. Manually walk the auto-compute logic at
   `_recomputeZoneRepairLaborIfAuto` (storage.ts:7673) against one zone
   from the wet check, by hand:
   - Pull `repairLaborManuallySet` for the zone.
   - Build the configMap from step 2's rows.
   - For each finding, would `configMap.get(f.issueType)` return
     anything? If no — that finding contributes 0. If yes — sum it.
5. **Note the missing quantity multiplication.** Line 7700 in the auto-
   compute reads `if (raw) totalHours += parseFloat(String(raw)) || 0;`
   — no multiplication by `f.quantity`. Record this in the audit as a
   secondary bug. It is real but smaller than the missing-catalog
   case: a 2× quantity nozzle replacement contributes 0.25 (the
   catalog default) instead of 0.50 (correct).

#### Step C — Scope the blast radius (run against production or recent snapshot)

Write each query as a file under
`docs/wc-labor-auto-compute-queries/` and run it. Capture results in
the audit doc.

**Q1 — Unbilled WCBs with zero labor (the safely-fixable bucket):**

```sql
SELECT
  COUNT(*) AS wcb_count,
  COALESCE(SUM(CAST(parts_subtotal AS NUMERIC)), 0) AS parts_total,
  COALESCE(SUM(CAST(total_amount AS NUMERIC)), 0) AS amount_total
FROM wet_check_billings
WHERE CAST(labor_subtotal AS NUMERIC) = 0
  AND invoice_id IS NULL;
```

**Q2 — Already-invoiced WCBs with zero labor (the accounting-impact
bucket):**

```sql
SELECT
  wcb.id,
  wcb.billing_number,
  wcb.customer_name,
  wcb.invoice_id,
  CAST(wcb.parts_subtotal AS NUMERIC) AS parts_subtotal,
  CAST(wcb.total_amount  AS NUMERIC) AS total_amount,
  i.invoice_number,
  CAST(i.total_amount AS NUMERIC) AS invoice_total,
  i.created_at AS invoice_created
FROM wet_check_billings wcb
JOIN invoices i ON i.id = wcb.invoice_id
WHERE CAST(wcb.labor_subtotal AS NUMERIC) = 0
ORDER BY i.created_at DESC;
```

**Q3 — Zones whose finding `issueType` strings don't match the catalog
for the parent company (Hypothesis 2 evidence):**

```sql
SELECT
  c.id AS company_id,
  c.name AS company_name,
  f.issue_type,
  COUNT(*) AS finding_count
FROM wet_check_findings f
JOIN wet_checks wc ON wc.id = f.wet_check_id
JOIN customers cu ON cu.id = wc.customer_id
JOIN companies c ON c.id = cu.company_id
LEFT JOIN issue_type_configs itc
  ON itc.company_id = c.id
 AND itc.issue_type = f.issue_type
WHERE itc.id IS NULL
GROUP BY c.id, c.name, f.issue_type
ORDER BY finding_count DESC;
```

**Q4 — Companies missing the catalog entirely (Hypothesis 1 evidence):**

```sql
SELECT
  c.id, c.name, COUNT(itc.id) AS catalog_rows
FROM companies c
LEFT JOIN issue_type_configs itc ON itc.company_id = c.id
GROUP BY c.id, c.name
HAVING COUNT(itc.id) = 0;
```

**Q5 — Per-company WCB labor health (sanity):**

```sql
SELECT
  c.name,
  COUNT(*) FILTER (WHERE CAST(wcb.labor_subtotal AS NUMERIC) = 0) AS zero_labor_count,
  COUNT(*) AS total_wcb_count,
  ROUND(100.0 * COUNT(*) FILTER (WHERE CAST(wcb.labor_subtotal AS NUMERIC) = 0)
        / NULLIF(COUNT(*), 0), 1) AS pct_zero_labor
FROM wet_check_billings wcb
JOIN customers cu ON cu.id = wcb.customer_id
JOIN companies c ON c.id = cu.company_id
GROUP BY c.name
ORDER BY zero_labor_count DESC;
```

The `pct_zero_labor` column is the most telling number — if it's near
100% for a given company, the catalog is missing or mismatched
company-wide. If it's spread across companies at moderate rates, the
problem is more nuanced.

#### Step D — Write the audit

Produce `docs/wc-labor-auto-compute-audit.md` with these sections (in
order):

1. **Reproducer summary** — `WC-2026-1027`, what it shows, the

   - "Seed `issue_type_configs` for all companies missing it." (if
     Q4 returns rows)
   - "Normalize `issueType` strings in the auto-compute." (if Q3
     returns rows)
   - "Fix the auto-compute regression: add quantity multiplication
     and any other arithmetic bug found in Step A." (if Steps A/C
     reveal arithmetic issues)
   - Often more than one.
7. **Recommendation for Slice 3 scope** — typically "recompute Q1
   WCBs, surface Q2 WCBs for accounting review," with the WCB
   counts.
