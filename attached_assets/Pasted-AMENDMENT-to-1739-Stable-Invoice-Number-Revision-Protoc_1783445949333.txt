AMENDMENT to #1739 — Stable Invoice Number + Revision Protocol (drop the -R suffix)

Fold this into #1739 before finalizing the version-chain slice (Step 7). #1739 keeps the number clean in the list (nesting under supersededByInvoiceId) but still renames the invoice to -R1 via deriveRevisionNumber. That's incompatible with the in-place QuickBooks update (which keeps the original DocNumber) and reads to the customer like a second invoice. Change the protocol so the invoice number is stable and immutable; a correction bumps an internal revision, not the number.

Why (governs the change)


The QB in-place update (#1711) keeps one QB invoice at its original DocNumber (#04723). If our number becomes #04723-R1, the system and QuickBooks permanently disagree and the customer (holding #04723) is confused. The customer/QB-facing number must not change on correction.
-R1 → -R2 → … gets ugly and reads as separate documents. A revision counter is cleaner and is the natural partner to the existing supersededByInvoiceId chain.
Root blocker today: invoices.invoiceNumber is .notNull().unique() (single-column) — which is the only reason a suffix was needed to avoid a collision.


Changes

Schema


Add revision: integer("revision").notNull().default(1) to invoices.
Drop the single-column unique on invoice_number; add composite unique (company_id, invoice_number, revision). (Migration: add column default 1, backfill existing rows to revision = 1, swap the constraint.)
Keep supersededByInvoiceId as-is.


Reissue logic (invoice-correction-routes.ts)


Do not call deriveRevisionNumber. The reissued invoice keeps the same invoiceNumber as the original and sets revision = original.revision + 1. Remove/retire deriveRevisionNumber.
Still: original → superseded + supersededByInvoiceId = new; new invoice status: "generated"; carry quickbooksInvoiceId + quickbooksSyncToken forward (from #1739).


QuickBooks


The QB DocNumber stays the base number (#04723) on the in-place update — do not send -R1. (updateQbInvoiceInPlace already targets the same QB invoice; just pass the base invoiceNumber.)


Display / PDF (update #1739 Step 7 wording)


The version-chain row shows the base number with the revision: "#04723 · Rev 2 (corrected)" — not #04723-R1. The superseded predecessor nests beneath as "Rev 1 — superseded [date]".
The reissued customer PDF carries a header marker: "Corrected invoice — supersedes original dated {originalDate}", base number #04723 unchanged.


Accounting gate (reserve the fork, don't build the alt path now)


Correct-in-place at the same number applies to unpaid/unapproved invoices (Huntington is unapproved). If an invoice is already paid, block in-place correction and surface "issue a credit note" instead — stub/flag it for a future credit-note flow. For now: only allow correction when status ∈ {generated, sent, overdue} and not paid.
