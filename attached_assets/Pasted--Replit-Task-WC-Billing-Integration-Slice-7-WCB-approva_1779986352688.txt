## Design Context (locked decisions)

These were agreed in the audit
(`WC Billing Integration - Slice 0 End-of-Month Audit.md`) and apply to
every slice in this family.

**What already exists on `wet_check_billings`:** `approvedAt`,
`approvedBy`, `approvedByUserId`, `approvedTotal` — verified at
`schema.ts:1417–1420`. The approval path at
`POST /api/wet-checks/:id/approve` (`routes.ts:16015`) already writes
these four fields when it transitions a WCB to
`approved_passed_to_billing`.

**What this slice adds:** two more nullable text columns
(`approvedLaborSnapshot`, `approvedPartsSnapshot`) and the
approval-path writes that populate them. The audit endpoint's parsing
logic (Slice 2) reads them once they exist.

**Backfill policy.** WCBs already approved before this slice ships do
NOT get backfilled snapshots — there is no source of truth for their
"at approval time" labor and parts breakdown. They stay `null`, and
the audit endpoint already handles null (the parse blocks at
`routes.ts:11167–11180` short-circuit on `if (bs.approvedLaborSnapshot) { … }`).
The WCB branch in Slice 2 uses the same null-tolerant pattern.

**JSON shape.** The snapshot is `JSON.stringify({ laborSubtotal,
partsSubtotal, totalAmount, totalHours, appliedLaborRate })` — a
single object that captures the financial state of the row at
approval time. The audit endpoint parses the relevant key per column.
This is the same shape `billing_sheets` uses (verified at the BS
approval path in `routes.ts:~10530` and the parse at `:11167`).

### Concrete changes

#### Schema

**(S1) Extend the `wetCheckBillings` table definition** at
`schema.ts:1417+`. Add immediately after `approvedTotal`:

```typescript
// Manager approval stamp fields.
approvedBy: text("approved_by"),
approvedByUserId: integer("approved_by_user_id").references(() => users.id),
approvedAt: timestamp("approved_at"),
approvedTotal: decimal("approved_total", { precision: 10, scale: 2 }),
// Slice 7 — snapshot of labor / parts financial state at approval time.
// Both nullable; WCBs approved before this slice ships read as null and
// are handled by the audit endpoint's null-tolerant parser.
approvedLaborSnapshot: text("approved_labor_snapshot"),   // NEW
approvedPartsSnapshot: text("approved_parts_snapshot"),   // NEW
```

Re-run `pnpm drizzle-kit generate:pg` (or the equivalent Drizzle
generation command used in this repo — check `package.json`) to
produce the migration SQL. The generated file should add:

```sql
ALTER TABLE wet_check_billings
  ADD COLUMN approved_labor_snapshot text,
  ADD COLUMN approved_parts_snapshot text;
```

No `NOT NULL`, no default. Existing rows stay `null`.

#### Approval path

**(P1) Find the WCB approval-stamp write in storage.ts.**

Grep for `wetCheckBillings).set({` to locate writes. The audit
identified four candidate sites (`storage.ts:3969`, `:7634`, `:7951`,
plus the `updateWetCheckBilling` general write at `:3968+`). The
relevant one is the write that sets `approvedAt` —
visit each candidate and identify the approval path. (The WC
Separate System Slice 6 design context names the helper:
`convertWetCheckToWetCheckBilling`, renamed in Slice 7 — search the
file for the rename.)

**(P2) At that approval-stamp site, extend the `set({ ... })` block.**

Old (sample shape — actual call may differ in detail):

```typescript
await tx.update(wetCheckBillings).set({
  approvedAt: now,
  approvedBy: manager.name,
  approvedByUserId: manager.id,
  approvedTotal: row.totalAmount,
  status: "approved_passed_to_billing",
  updatedAt: now,
}).where(eq(wetCheckBillings.id, row.id));
```

New (additive):

```typescript
const snapshotEnvelope = {
  laborSubtotal: parseFloat(row.laborSubtotal || "0"),
  partsSubtotal: parseFloat(row.partsSubtotal || "0"),
  totalAmount: parseFloat(row.totalAmount || "0"),
  totalHours: parseFloat(row.totalHours || "0"),
  appliedLaborRate: parseFloat(row.appliedLaborRate || row.laborRate || "0"),
};
const approvedLaborSnapshot = JSON.stringify({
  laborSubtotal: snapshotEnvelope.laborSubtotal,
  totalHours: snapshotEnvelope.totalHours,
  appliedLaborRate: snapshotEnvelope.appliedLaborRate,
});
const approvedPartsSnapshot = JSON.stringify({
  partsSubtotal: snapshotEnvelope.partsSubtotal,
  totalAmount: snapshotEnvelope.totalAmount,
});
await tx.update(wetCheckBillings).set({
  approvedAt: now,
  approvedBy: manager.name,
  approvedByUserId: manager.id,
  approvedTotal: row.totalAmount,
  approvedLaborSnapshot,   // NEW
  approvedPartsSnapshot,   // NEW
  status: "approved_passed_to_billing",
  updatedAt: now,
}).where(eq(wetCheckBillings.id, row.id));
```

**Do not write the snapshots from any other site.** The approval
path is the only site that should ever set these columns. The
auto-bill path (which transitions a WCB straight to
`approved_passed_to_billing` without manager intervention) should
also write them — locate that path (audit pointed to
`_writeRepairedInFieldBilling` at storage.ts:6964, refactored in WC
Separate System Slice 6) and apply the same block. The snapshot in
that case captures the auto-bill computed totals.

#### Audit consume

**(A1) Update the Slice 2 WCB branch** in `routes.ts:~11191+` to
remove the temporary `(wcb as any)` casts:

Old (from Slice 2):

```typescript
if ((wcb as any).approvedLaborSnapshot) {
  try {
    const snap = JSON.parse((wcb as any).approvedLaborSnapshot);
    // …
  } catch { approvedLaborSnapshot = null; }
}
```

New:

```typescript
if (wcb.approvedLaborSnapshot) {
  try {
    const snap = JSON.parse(wcb.approvedLaborSnapshot);
    const parsed = typeof snap === "number" ? snap : parseFloat(snap?.laborSubtotal ?? snap?.total ?? "");
    approvedLaborSnapshot = isNaN(parsed) ? null : parsed;
  } catch { approvedLaborSnapshot = null; }
}
if (wcb.approvedPartsSnapshot) {
  try {
    const snap = JSON.parse(wcb.approvedPartsSnapshot);
    const parsed = typeof snap === "number" ? snap : parseFloat(snap?.partsSubtotal ?? snap?.total ?? "");
    approvedPartsSnapshot = isNaN(parsed) ? null : parsed;
  } catch { approvedPartsSnapshot = null; }
}
```

The casts come off naturally because the typed `WetCheckBilling`
shape now includes these columns after the schema add. If
`pnpm typecheck` still complains, the schema regeneration step (S1)
wasn't fully run — re-run drizzle-kit.

### Tests

**Storage approval test** — new
`wcb-approval-snapshot.test.ts`:

```typescript
describe("WCB approval writes labor/parts snapshots (Slice 7)", () => {
  it("populates approvedLaborSnapshot and approvedPartsSnapshot on the manager-approval path", async () => {
    const { wc, wcb } = await seedSubmittedWcb({
      laborSubtotal: "255",
      partsSubtotal: "100",
      totalAmount: "355",
      totalHours: "3",
      appliedLaborRate: "85",
    });
    await storage.approveWetCheck(wc.id, /* manager */ { id: 7, name: "Mary" });
    const after = await storage.getWetCheckBillingById(wcb.id);
    expect(after.approvedLaborSnapshot).toBeTruthy();
    expect(after.approvedPartsSnapshot).toBeTruthy();
    const labor = JSON.parse(after.approvedLaborSnapshot!);
    const parts = JSON.parse(after.approvedPartsSnapshot!);
    expect(labor.laborSubtotal).toBe(255);
    expect(labor.totalHours).toBe(3);
    expect(labor.appliedLaborRate).toBe(85);
    expect(parts.partsSubtotal).toBe(100);
    expect(parts.totalAmount).toBe(355);
  });

  it("populates snapshots on the auto-bill path too", async () => {
    const { wc, wcb } = await seedAutoBilledWcb({
      laborSubtotal: "180",
      partsSubtotal: "60",
      totalAmount: "240",
      totalHours: "2",
      appliedLaborRate: "90",
    });
    // The auto-bill path runs synchronously during submit; no separate
    // approve call is needed — just read after submit.
    const after = await storage.getWetCheckBillingById(wcb.id);
    expect(after.approvedLaborSnapshot).toBeTruthy();
    expect(after.approvedPartsSnapshot).toBeTruthy();
  });

  it("does NOT populate snapshots for a WCB that has not been approved yet", async () => {
    const { wcb } = await seedSubmittedWcb({}); // submitted, no approval yet
    expect(wcb.approvedLaborSnapshot).toBeNull();
    expect(wcb.approvedPartsSnapshot).toBeNull();
  });
});
```

### Acceptance criteria

- `wet_check_billings` has two new nullable text columns:
  `approved_labor_snapshot` and `approved_parts_snapshot`.
- The migration SQL is committed under `lib/db/migrations/` with a
  monotonically-increasing prefix matching the existing convention.
- The WCB approval-stamp write (both manager-approval and auto-bill
  paths) populates the two snapshot columns with the JSON envelopes
  specified above.
- Slice 2's Slice 5-style `(wcb as any).…` casts are removed —
  TypeScript sees the typed columns directly.
- The financial-pulse audit endpoint returns parsed numeric values
  for `approvedLaborSnapshot` and `approvedPartsSnapshot` on WCB
  items that were approved post-deploy, and `null` for WCBs
  approved pre-deploy.
- WCBs that have not yet been approved have both columns `null`.
- `pnpm typecheck`, `pnpm test`, and `pnpm drizzle-kit migrate`
  (against a test database) pass.
- No backfill is run. Legacy approved WCBs keep their `null`
  snapshots — the audit endpoint already handles this case.

### Out of scope

- Backfilling snapshots for legacy approved WCBs. There is no source
  of truth for "what was approved" historically.
- Surfacing the snapshot deltas in a UI. The audit endpoint exposes
  them; if a UI consumer is wanted, that's its own slice.
- Equivalent snapshot fields on `wet_check_findings`. The findings
  themselves are version-controlled by the `convertedAt` /
  `wetCheckBillingId` stamping; per-finding snapshots are not in
  scope.
- Changing the BS snapshot shape or migration. BS is the model;
  WCB conforms.
- Restricting the snapshot envelope further (e.g. omitting
  `totalHours`). Carrying the extra fields in the JSON now is cheap
  and avoids a future re-write if the audit ever wants to display
  the breakdown.
