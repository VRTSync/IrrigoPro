# Slice — Estimate Command Center

**Date:** 2026-05-18
**Status:** Ready for Replit implementation
**Author:** Engineering planning
**Builds on:** Slice 7 (Pending-Approval Workflow), Slice 10a (canonical lifecycle).
**Decisions locked:**

- **Q1 = Company Admin only, for now.** The Command Center is *not* shipped to Billing Manager or Irrigation Manager in this slice. The existing `/estimates/pending-approval` page is left intact for Billing Manager (their nav entry doesn't change). Only the Company Admin's nav entry is swapped over.
- **Q2 = Replace, don't add.** The Command Center *absorbs* the current "Estimates Pending Approval" nav item for Company Admins — same group, same icon, new label, new path. The Pending-Approval data becomes one column of the Command Center's kanban + one tile in the KPI strip. The old `/estimates/pending-approval` URL stays valid but server-side it 302-redirects to `/estimates/command-center` for Company Admins so old bookmarks and email links keep working.
- **Q3 = Hybrid layout: tiles → attention → kanban → table.** No tabs. Everything is on one page and visible by default, because the brief is "very apparent / all the visibility I could want." The kanban is collapsible (default open); the table is the source of truth and is never collapsed.
- **Q4 = Reuse existing endpoints, add one summary endpoint.** The kanban + table read from `GET /api/estimates` (already paginated to 500). The tiles read from a new `GET /api/estimates/summary` that returns counts + dollar sums bucketed by `lifecycleStatus`, plus three "window" aggregates (`approvedLast30Days`, `expiringNext7Days`, `stuckInReviewOver3Days`) and a 90-day win rate. Single source of aggregation lives server-side so the tiles never disagree with the kanban totals.
- **Q5 = No drag-and-drop on the kanban (this slice).** Cards are read-only — click opens the existing `EstimateDetailModal`. Status transitions still happen through the existing route surfaces (`internal-approve`, `send-approval-email`, etc.) reachable from the modal and from per-row actions in the table. DnD is a deliberate follow-up — it implies authoring new state machines on the client and isn't needed to make the system "apparent."
- **Q6 = "Attention" rules are server-stamped, not client-recomputed.** The summary endpoint also returns a small `attention[]` array (up to 8 estimate IDs flagged with one of `stuck_in_review`, `expiring_soon`, `high_value_silent`) so the AttentionPanel doesn't have to re-derive urgency from a paginated list. The list is built from authoritative server data; the UI just renders it.

**Related (future, NOT in this slice):**
- Drag-and-drop transitions on the kanban (would also require a new audit event).
- CSV export of the filtered table.
- Saved filter views (URL-shareable querystring).
- Activity timeline column (lifecycle transitions + approvals) per estimate.
- Same page rolled out to Billing Manager and Irrigation Manager with role-aware tiles and filters.

## Executive Summary

Today, a Company Admin who wants to know "what's happening with estimates" has to juggle three places: the `/estimates/pending-approval` queue for what's awaiting internal review, the broader `/estimates` list to see what's been sent, and the `/` admin dashboard to glance at the few estimate-flavored numbers it surfaces in passing. None of these answer the three questions an owner actually asks every morning — *where is the money sitting, what's at risk of expiring, and what's stuck on my desk* — without the user doing the math themselves.

The Estimate Command Center collapses that into one Company-Admin-only page. The top strip shows six KPI tiles built off a new `/api/estimates/summary` endpoint (pipeline $, awaiting review, awaiting customer, expiring this week, approved this month, 90-day win rate). Below that, an Attention panel surfaces the 3–8 estimates the system thinks need action now, server-flagged with a reason. Below that, a six-column lifecycle kanban (`Draft / Pending review / Sent / Approved / Rejected / Expired`) using the same `LIFECYCLE_TINTS` palette already shipped in Slice 10a, so the colors match the badges everywhere else. Below that, a single filterable, sortable table of every open estimate, with per-row actions.

The nav swap is the part that makes this "feel like the Command Center" rather than "yet another page": the existing **Estimates Pending Approval** leaf in the Company Admin nav becomes **Estimate Command Center**, with the existing badge count (now meaning "awaiting review only") preserved. The old URL is a 302 to the new one.

**Slice ships when:** a Company Admin lands on `/estimates/command-center`, sees six accurate tiles backed by a single server response, sees an Attention list of the right rows in the right order, sees a six-column lifecycle kanban whose column totals match the tiles, and can filter / sort / search every estimate in one table on the same page — and the old `/estimates/pending-approval` URL still works for Company Admins (via redirect) and is unchanged for Billing Manager.

---

## Why Now

Three pressures converge on the same UI gap:

1. **The lifecycle is finally canonical.** Slice 10a (`lib/lifecycle.ts`) made `lifecycleStatus` a single server-stamped value with a fixed palette (`LIFECYCLE_TINTS`) and a fixed sort order (`LIFECYCLE_ORDER`). Until that landed, any "see all estimates by status" UI had to recompute the bucket on the client and risk drifting from the badges shown elsewhere. With 10a in place, a kanban grouped by `lifecycleStatus` will agree with every other surface on the first render.

2. **The Pending Approval page is doing two jobs.** It was Slice 7's queue for internal reviewers — a list of work the reviewer owns. Company Admins (per the original Slice 7 role table) also got that page, but they don't *primarily* use it as a queue; they use it as a shorthand for "show me what's happening." Promoting their entry point from a single-column queue to a full command center matches what they're already trying to do with it, without breaking what Billing Manager uses it for.

3. **The numbers exist but aren't summed.** `GET /api/estimates` already returns the right shape; `lifecycle.ts` already returns the right buckets; the dashboard already has `KpiTile` and the rest of the tile primitives. The page is fundamentally an assembly problem, not a new-domain problem.

Putting this off another sprint costs us "all the visibility I could want" not because the data is missing but because it's spread across three pages and aggregated by eye.

## Endpoint: `GET /api/estimates/summary`

Register **before** `GET /api/estimates/:id` for the same Express ordering reason that bit Slice 7. Place it adjacent to `/api/estimates/pending-approval` in `estimate-routes.ts` (currently line ~333) — just below it.

```ts
app.get(
  "/api/estimates/summary",
  requireAuthentication,
  requireEstimateApprovalAccess,
  async (req: any, res) => {
    try {
      const userRole = req.authenticatedUserRole;
      const userCompanyId = req.authenticatedUserCompanyId;
      let scopeCompanyId: number | null;
      if (userRole === "super_admin") {
        scopeCompanyId = null;
      } else {
        if (!userCompanyId) {
          res.status(400).json({ message: "Missing company context" });
          return;
        }
        scopeCompanyId = Number(userCompanyId);
      }
      const summary = await storage.getEstimateSummary!(scopeCompanyId);
      res.json(summary);
    } catch (error) {
      console.error("Error fetching estimate summary:", error);
      res.status(500).json({ message: "Failed to fetch estimate summary" });
    }
  },
);
```

**Response shape** (add to `lib/api-spec` and the corresponding Zod in `lib/api-zod`):

```ts
type LifecycleBucket = "draft" | "pending_review" | "sent" | "approved" | "rejected" | "expired";

type AttentionReason = "stuck_in_review" | "expiring_soon" | "high_value_silent";

interface EstimateSummary {
  byLifecycle: Record<LifecycleBucket, { count: number; totalAmount: number }>;
  windows: {
    approvedLast30Days: { count: number; totalAmount: number };
    expiringNext7Days: { count: number; totalAmount: number }; // sent and (estimateDate + 30d) ∈ [now, now+7d]
    stuckInReviewOver3Days: { count: number; totalAmount: number };
  };
  attention: Array<{
    estimateId: number;
    estimateNumber: string;
    customerName: string;
    totalAmount: number;
    lifecycle: LifecycleBucket;
    reason: AttentionReason;
    sinceDays: number; // e.g. days since createdAt for stuck_in_review, days until expiry for expiring_soon (negative if already expired)
  }>;
  winRate90d: number; // approved / (approved + rejected + expired) over the last 90 calendar days. NaN-safe: 0 if denominator is 0.
}
```

**Aggregation rules:**

- Each estimate row contributes to exactly one `byLifecycle` bucket, using the server-side `lifecycleStatus` already stamped on the row (`deriveLifecycleForWrite` in the existing code path). The aggregator never re-derives.
- `windows.expiringNext7Days` filters to `lifecycle === "sent"` and `now − estimateDate ∈ (23d, 30d]` (i.e., ≤ 7 days from the 30-day expiration boundary). Re-uses `ESTIMATE_EXPIRATION_DAYS` from `lib/lifecycle.ts` so we can't have two definitions drift.
- `windows.stuckInReviewOver3Days` filters to `lifecycle === "pending_review"` and `now − createdAt > 3 days`. The threshold lives in a single named constant `ESTIMATE_STUCK_REVIEW_DAYS = 3` in the same `lib/lifecycle.ts` so the UI's "Stuck" badge can read it without a magic number.
- `winRate90d` denominator includes `approved`, `rejected`, **and** `expired` over the last 90 calendar days (by `estimateDate`). `expired` counts as a loss. If denominator is 0, return `0` (the UI treats `0` and "no data" identically and shows "—").
- `attention[]` is bounded at 8 items and sorted by `(reason priority, sinceDays desc)` where priority is `expiring_soon` > `stuck_in_review` > `high_value_silent`. `high_value_silent` is `lifecycle === "sent" && totalAmount >= 5000 && sinceDays >= 7`. The `$5000` threshold lives in the same lifecycle.ts constants block (`ESTIMATE_HIGH_VALUE_USD = 5000`).

## Storage Layer Changes

In `lib/db/src/storage.ts` (`IStorage` + database implementation):

- **Add:** `getEstimateSummary(companyId: number | null): Promise<EstimateSummary>`.
- Implementation reads via the existing estimates query path (`includeDeleted: false`, `companyId` scoping mirroring `getEstimatesPendingApproval`), then walks the result set once to populate the buckets and windows. **No new SQL aggregations** in this slice — one pass in TS keeps the diff small and is well within budget at current row counts. If volume grows, a future slice can swap the body for an aggregate SQL query without changing the response shape.
- **Do not touch** `getEstimates`, `getEstimatesPendingApproval`, or any write path.

## Risks and Rollback

- **Summary / list disagreement.** Tiles read the summary endpoint; the kanban reads the full list. If a row's `lifecycleStatus` were computed differently in the two paths, totals would disagree visibly. Mitigation: both reads source `lifecycleStatus` from the same `deriveLifecycleForWrite` stamp the server already writes. The summary endpoint does NOT re-derive on the fly; the test suite asserts equality on a seeded set.
- **Page weight.** Six tiles + a kanban + a table on one screen is a real amount of DOM. Mitigation: tiles use the existing `KpiTile` (already optimized for the admin dashboard); the kanban caps each column at six visible cards; the table virtualizes only if it grows beyond ~200 rows (use `@tanstack/react-virtual`, which the workspace already depends on — out of scope to wire up unless we see perf regressions).
- **Nav swap surprise.** Existing Company Admin muscle memory points "Estimates Pending Approval" at a queue. New label points at a dashboard. Mitigation: the `badgeKey` is preserved so the visual cue (a numeric badge in the same nav slot) is identical, and the old URL still works via redirect.
- **Wouter `setLocation` race.** If the user starts an action on the shim before `useEffect` fires, the action lands on the page that's about to unmount. Mitigation: the shim renders `null` — there's no interactive surface to start an action from.

Rollback: revert the nav swap (single line change), revert the three route registrations, drop the summary endpoint route, leave the storage method and page files in place. Old behavior is restored on next page load.

## What This Unblocks

Once the Command Center is the Company Admin's estimate home, several follow-ups slot in cleanly:

- **Drag-and-drop kanban.** A future slice can add DnD by wiring each card's drop target to the existing transition mutations (`internal-approve`, `send-approval-email`, `reject`). No new state machine needed.
- **CSV export.** The table already filters client-side; an "Export" button at the top of the toolbar produces a CSV of the current filtered view.
- **Activity timeline.** Hang a vertical timeline component off the detail modal that reads from a future audit-log endpoint; the Command Center's Attention strip can then link directly into the timeline view.
- **Role-aware rollout.** The summary endpoint already shapes data the way Billing Manager and Irrigation Manager would consume it; a follow-up slice can register the same page in those navs with different tile sets and different default filters, without changing the server.

The Command Center is the place every future estimate-system surface plugs into.
