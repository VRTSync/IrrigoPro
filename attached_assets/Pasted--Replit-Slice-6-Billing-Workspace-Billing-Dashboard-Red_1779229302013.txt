# Replit Slice 6 — Billing Workspace (Billing Dashboard Redesign)

**Goal:** Reshape `/billing-dashboard` from a generic dashboard into a focused **Billing Workspace** — the billing manager's daily approval queue for individual billing sheets, work orders, and parts. Financial Pulse now owns the overview/dashboard role, so this page can specialize on workflow.

**Note on monthly billing:** The current monthly billing flow is intentionally left untouched in this slice. A separate audit (Slice 7) documents the existing implementation and decides whether anything needs to change. Slice 6 is strictly about daily approvals.

**Audience:** `billing_manager`, `company_admin`, `super_admin`.

**Depends on:** Slices 1–5 (especially Slice 5's data unification — this page should use the same canonical endpoints).

**Recommended rename:** `Billing Dashboard` → `Billing Workspace`. Cleaner mental model now that it's not a dashboard. New route: `/billing-workspace`. Old route should 301 to the new one. (Open to keeping the current name if the team prefers — it's a small change either way.)

---

## 1. Page shape

A slim Financial Pulse widget across the top, then three workflow zones.

### Zone 0 — Financial Pulse widget (slim)

A short `<FinancialPulseWidget variant="billing-header" />` (built in Slice 5) mounted at the very top of the page. Three tiles in a single row:

- **Billed MTD**
- **Collected MTD**
- **Outstanding A/R**

Plus a "View Financial Pulse →" link in the widget's header.

The point of this row is awareness, not workflow. It gives the billing manager a one-glance "are we keeping up?" view while they work through approvals below, without becoming a competing dashboard on the page. Visually distinct from Zone A (which is operational status) — keep it on its own row with a small bottom divider so the eye knows "this section is financial, the section below is your work queue."

### Zone A — Status strip (slim)

A single horizontal strip with 4 small indicators (not full KPI tiles — these are status, not metrics):

- **Awaiting your approval** — count of BS + WO + parts pending review, color-coded if any item is >7 days old.
- **Approved this week** — count, helps the user see their own throughput.
- **Drafts from techs in last 24h** — count, gives heads-up on incoming volume.
- **QuickBooks sync** — green/yellow/red status badge. If red, click → existing QB reconnect flow.

### Zone B — Approval Queue (primary work area)

A single unified list combining billing sheets, work orders, and pending part approvals. This is the page's main canvas.

**Columns:**
- Type icon (BS / WO / Part) + a small tag pill
- Customer name
- Technician
- Description (truncated, e.g., the first line of the AI-generated description if present)
- Submitted (relative time: "2h ago", "Yesterday", "5 days ago")
- Dollar value (parts + labor subtotal)
- Status pill (`Pending Review`, `Needs Photos`, `Flagged`, etc.)
- Quick actions (3-dot menu)

**Filters along the top:**
- Type: All / Billing Sheets / Work Orders / Parts
- Customer (searchable dropdown)
- Technician (searchable dropdown)
- Age: All / Today / This week / >7 days old
- Status

**Sort:** Default is oldest first (drives FIFO behavior). Toggle for newest first, highest dollar value first, customer A→Z.

**Saved views:** Lightweight — three preset buttons above the filter row: `All Pending`, `Just Parts`, `>$1,000`. v1 ships these three hard-coded; user-defined saved views is a future enhancement.

**Pagination:** 50 rows per page. Sticky header. Keyboard-friendly (see Zone D below).

### Zone C — Detail Panel (right-side drawer or modal)

When a row is clicked, the full record opens in a right-side drawer that overlays ~40% of the screen. Two reasons for a drawer over a modal:
1. The user can still see the queue context behind it.
2. After approving, the drawer closes and the next row auto-selects (smooth pipeline).

**Drawer contents (same shape for BS, WO, and parts — just hides irrelevant sections):**
- Header: customer, tech, date, dollar value, status
- AI-generated description (collapsible, with "Regenerate" if the user has access)
- Parts table (name, qty, unit price, line total) with inline edit
- Labor section (hours, rate, total) with inline edit
- Photos (grid, click to lightbox; flag if any item is missing required photos)
- Notes (tech notes + space for billing manager notes)
- Activity log (who approved/edited/flagged, when)

**Actions footer (sticky):**
- **Approve** (primary button) — keyboard `A`
- **Kick back** (secondary, opens a small text input for reason) — keyboard `K`
- **Flag** (for review by manager) — keyboard `F`
- **Save edits without approving** — keyboard `Ctrl+S`
- **Close** — keyboard `Esc`

After approval, the drawer transitions: row drops out of queue, next row in the current sort opens. If queue is empty, drawer closes and a small "All caught up" celebration message appears in the queue area.

### Zone D — Keyboard navigation

This is what makes the page feel fast. Implement:
- `J` / `K` — next / previous row in queue (open the drawer for that row)
- `Enter` — open currently focused row
- `A` — approve current item
- `K` — kick back (only when drawer is open; conflicts with previous nav binding so use `K` for kickback only when drawer is open and item is focused)
- `F` — flag
- `/` — focus the search/filter input
- `?` — show keyboard shortcuts cheat sheet

Document the shortcuts in a small `?` button in the corner of the page.

## 2. Collections — recommendation

You mentioned collections is tracked in QuickBooks, not in Irrigo. **Recommendation: don't duplicate QuickBooks here.** Rebuilding collections workflows inside Irrigo would create a competing source of truth and double the workload to keep it accurate.

What to do instead:

- **Status indicator** in Zone A (and optionally on Financial Pulse) showing "X invoices overdue in QuickBooks" — sourced from a small new endpoint `GET /api/quickbooks/overdue-summary` that calls QB and caches for ~15 minutes.
- **Deep link** from that indicator into the QuickBooks A/R aging report (QB has a stable URL pattern for this).
- **Optional later:** a `lastReminderSentAt` field on invoices so Irrigo can show "reminder sent 4 days ago via QB" without owning the workflow.

That gives billing managers awareness without forcing them to maintain two systems. If collections later becomes a real product surface (e.g., you want to send reminders from inside Irrigo), spin it out as its own slice — but I'd avoid that bet until you know QB's tools aren't enough.

## 3. Data sources

Reuse what exists. Where current Billing Dashboard logic exists:

- Pending parts approval endpoint → folded into the unified approval queue.
- Any client-side aggregation of totals → replaced with calls to `/api/financial-pulse/customer/:id/summary` (the endpoint from Slice 5).
- Existing monthly billing flow → left in place, untouched. Slice 7 will document and decide its fate separately.

**New endpoints to add:**

- `GET /api/billing-workspace/queue?type=&customer=&tech=&age=&status=&sort=&page=` — returns the unified approval queue.
- `GET /api/quickbooks/overdue-summary` — cached overdue count + dollar total.

All under the same `billing_manager | company_admin | super_admin` guard.

## 4. Acceptance criteria

1. `/billing-workspace` (or kept-name) loads in under 1.5s on a fixture with 200 pending items.
2. The Financial Pulse widget (Zone 0) renders three tiles and a working "View Financial Pulse →" link. Field-tech tokens see no widget at all (it returns null per Slice 5 behavior).
3. A billing manager can navigate the queue with `J/K`, approve with `A`, and kick back with `K` (while drawer is open). After approval, the next item auto-opens.
4. The approval queue shows BS, WO, and parts in one list; type-filtering correctly isolates each.
5. The QuickBooks overdue indicator in Zone A reflects the QB sandbox's overdue count and links to the correct QB URL.
6. Old `/billing-dashboard` route redirects to `/billing-workspace` (or kept-name).
7. Tests: keyboard shortcuts, queue filter combinations, approval and kickback flows, drawer auto-advance behavior.
8. The existing monthly billing flow continues to work exactly as it does today — no regressions, no UI changes to it as part of this slice.

---

## 5. Explicitly out of scope

- **Any changes to the existing monthly billing flow** — that's Slice 7's territory after its audit.
- User-defined saved queue views (only 3 hard-coded presets in v1).
- Customer-facing invoice preview before sending (assumed to be QB's job).
- Reminder/dunning workflows inside Irrigo — defer to QB.
- A separate per-tech approvals queue. (The Technician filter handles this for now.)
- Mobile-optimized layout (desktop-first, mobile is best-effort responsive but not designed-for).
- AI-suggested approvals ("this looks normal, auto-approve?") — future slice.

## 6. Implementation order

1. New endpoints (`/queue`, `/quickbooks/overdue-summary`).
2. Page shell + Zone 0 (`billing-header` widget mount) + Zone A status strip + route + redirect from old path.
3. Approval queue (Zone B) with filters and sorting.
4. Detail drawer (Zone C) with all sections and the keyboard shortcuts (Zone D).
5. QuickBooks overdue indicator + deep link.
6. Tests across all the above.
7. Delete legacy Billing Dashboard sections that are no longer relevant — **except** anything tied to the existing monthly billing flow, which stays. Clean up dead imports.

Each step can land independently if any one hits a snag.
