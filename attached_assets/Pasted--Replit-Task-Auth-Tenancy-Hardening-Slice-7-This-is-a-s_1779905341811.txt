**Concrete changes.**

### 1. Extend the pino-http serializer (app.ts)

Today the request serializer (app.ts:23–34) emits `id`, `method`, and
`url`. Add `tenantCompanyId` and `actorRole`. The serializer runs
*before* `requireAuthentication` populates the values, so it reads from
the request properties — they may be undefined for unauthenticated
endpoints, which is fine.

```ts
serializers: {
  req(req) {
    return {
      id: req.id,
      method: req.method,
      url: req.url?.split("?")[0],
      tenantCompanyId: (req as any).authenticatedUserCompanyId ?? null,
      actorRole: (req as any).authenticatedUserRole ?? null,
    };
  },
  res(res) {
    return { statusCode: res.statusCode };
  },
},
```

Pino logs every line through this serializer at request-end; child
logs created inside the request inherit the request id. For more
detail-rich logging inside a handler, future work can add
`req.log.info({ tenantCompanyId, ... }, "message")` — but that's not
required by this slice.

### 2. Structured tenant-boundary attempt emitter

Schema, in `lib/db/src/schema/schema.ts`, adjacent to the other audit
tables:

```ts
export const tenantBoundaryAttempts = pgTable("tenant_boundary_attempts", {
  id: serial("id").primaryKey(),
  occurredAt: timestamp("occurred_at").defaultNow().notNull(),
  // Resource the attempt targeted.
  resourceType: text("resource_type").notNull(), // 'work_order' | 'billing_sheet' | future
  resourceId: integer("resource_id").notNull(),
  // Actor — never null for an authenticated attempt (Slices 1, 2
  // require requireAuthentication before the guard).
  actorUserId: integer("actor_user_id").references(() => users.id).notNull(),
  actorCompanyId: integer("actor_company_id").references(() => companies.id),
  actorRole: text("actor_role").notNull(),
  // The tenant the resource actually belongs to. Nullable for the
  // rare case of an orphan row (no resolvable companyId).
  targetCompanyId: integer("target_company_id").references(() => companies.id),
  // Route info for forensics.
  method: text("method").notNull(),
  path: text("path").notNull(),
}, (table) => ({
  occurredAtIdx: index("tenant_boundary_attempts_occurred_at_idx").on(table.occurredAt),
  actorIdx: index("tenant_boundary_attempts_actor_idx").on(table.actorUserId),
}));
```

Migration `0006_tenant_boundary_attempts.sql`:


Emitter, `lib/tenant-boundary-attempts.ts`:

```ts
import { db } from "../db";
import { tenantBoundaryAttempts } from "@workspace/db";

// Process-local counter. The /api/admin/tenancy/attempts endpoint
// reads the table for forensics; the counter is for live dashboards
// (the same shape as other in-process counters in this codebase —
// no Prometheus client added in this slice).
let attemptCounter = 0;
export function getTenantBoundaryAttemptCount(): number {
  return attemptCounter;
}

export async function recordTenantBoundaryAttempt(args: {
  resourceType: "work_order" | "billing_sheet";
  resourceId: number;
  actorUserId: number;
  actorCompanyId: number | null;
  actorRole: string;
  targetCompanyId: number | null;
  method: string;
  path: string;
}): Promise<void> {
  attemptCounter += 1;
  try {
    await db.insert(tenantBoundaryAttempts).values({
      resourceType: args.resourceType,
      resourceId: args.resourceId,
      actorUserId: args.actorUserId,
      actorCompanyId: args.actorCompanyId,
      actorRole: args.actorRole,
      targetCompanyId: args.targetCompanyId,
      method: args.method,
      path: args.path,
    });
  } catch (err) {
    // Insertion failures must not block the rejection — log and
    // continue so the original 404 still reaches the client.
    console.error("Failed to record tenant boundary attempt:", err);
  }
}
```

Replace the `console.warn` in `requireSameCompanyAsWorkOrder` (Slice 1)
and `requireSameCompanyAsBillingSheet` (Slice 2) with `await
recordTenantBoundaryAttempt({ ... })`. Both middlewares already have
all the values needed.

Super-admin endpoint:

```ts
app.get(
  "/api/admin/tenancy/attempts",
  requireAuthentication,
  async (req, res) => {
    if (req.authenticatedUserRole !== "super_admin") {
      res.status(403).json({ message: "Super admin only" });
      return;
    }
    const limit = Math.min(parseInt(String(req.query.limit ?? "100")) || 100, 1000);
    const rows = await db
      .select()
      .from(tenantBoundaryAttempts)
      .orderBy(desc(tenantBoundaryAttempts.occurredAt))
      .limit(limit);
    res.json({ count: getTenantBoundaryAttemptCount(), rows });
  },
);
```

### 3. Retire dead role strings

Two known sites: routes.ts:9322 and routes.ts:9398. Both read:

```ts
const managerUsers = managers.filter(u => u.role === "irrigation_manager" || u.role === "admin");
```

Replace with the canonical role set:

```ts
const managerUsers = managers.filter(
  (u) => u.role === "irrigation_manager" || u.role === "company_admin",
);
```

Also scope the `getUsers()` call to the work order's companyId so the
notification fan-out stops crossing tenants:

```ts
// before
const managers = await storage.getUsers();

// after
const managers = await storage.getUsers(workOrder.companyId);
```

(Slice 4 added `workOrder.companyId`; this slice consumes it.)

Search-and-destroy pass for any other `=== "admin"` or `=== "manager"`
on a role string. Grep:

```
git grep -nE '\.role\s*===\s*"(admin|manager)"'
git grep -nE 'userRole\s*===\s*"(admin|manager)"'
```

Each hit is either dead code (replace with the canonical value or
remove the branch) or a real bug (the branch was supposed to fire and
hasn't been for years). Triage each on its merits; the most common
case is dead.

### 4. Consolidate `/complete` endpoints

Today:
- `POST /api/work-orders/complete` (body carries `workOrderId`) — used by web.
- `POST /api/work-orders/:id/complete` (path id) — used by mobile.

Canonical going forward: `POST /api/work-orders/:id/complete`.

Step 1: bring the path-id endpoint up to parity with the body-form's
behavior. Compare the two handlers (routes.ts:9146 vs routes.ts:9351)
and merge any feature the path-form lacks (the path-form does not
currently accept `usedParts`, `aiInputs`, `aiShortDescription`,
`aiDetailedDescription`, `branchName`, `laborMode` — most of these
are field-tech-relevant). Lift the body-form's handler body verbatim
into the path-form, parameterized on `req.params.id`. Delete any
duplicated logic.

Step 2: the body-form route stays mounted for one release cycle as
a 308 redirect to the canonical form:

```ts
app.post("/api/work-orders/complete", requireAuthentication, (req, res) => {
  const workOrderId = req.body?.workOrderId;
  if (!workOrderId || !Number.isFinite(parseInt(String(workOrderId)))) {
    res.status(400).json({ message: "workOrderId required" });
    return;
  }
  // 308 preserves the request method on redirect; clients that
  // upgrade to the canonical form transparently. Web client is
  // updated to the new URL in this same slice.
  res.redirect(308, `/api/work-orders/${parseInt(String(workOrderId))}/complete`);
});
```

Step 3: update `artifacts/irrigopro/src/components/work-orders/work-order-completion.tsx:312`:

```ts
// before
return apiRequest("/api/work-orders/complete", "POST", data);

// after
return apiRequest(`/api/work-orders/${workOrder.id}/complete`, "POST", data);
```

The mobile client is already on the canonical form; no change there.

The 308 redirect goes away in a follow-up cleanup commit once telemetry
confirms zero traffic on the old path for a full release cycle.
