# BUILD — Comprehensive Ticket-Total Drift Repair (one Super-Admin run) + Heal-at-Invoice Prevention

**What & Why**
Billing sheets #272/#273 blocked invoice 82027's PDF with the old parts-dropped drift (stored total = labor only). The **write bug is fixed** (generic PATCH / labor-hours / rate-mode all route through the #1669 stored-fallback helper — no new drift). But these are **legacy stragglers**: the existing `reconcile-billing-sheet-invoice-totals` migration only repairs **invoiced** billing sheets (`invoiceId IS NOT NULL`, `reconcile-…-db.ts:88`), so records that drifted while un-invoiced were skipped every run and only surface when finally billed. This ticket (1) replaces the narrow migration with **one comprehensive drift-repair migration** that fixes every drifted record — billing sheets, work orders, and WCBs, invoiced or not — in a single run, and (2) adds a **heal-at-invoice-generation** guard so latent drift can never reach the PDF validator again.

## Slice 1 — Comprehensive drift-repair migration (Super Admin)
**File:** new `artifacts/api-server/src/lib/migrations/repair-ticket-total-drift.ts`, id `repair-ticket-total-drift-v1`, title "Repair Ticket Total Drift (Billing Sheets, Work Orders, WCBs)". Register it in `registry.ts`. **Retire** the narrow `reconcile-billing-sheet-invoice-totals` from the registry (fully subsumed by this — one drift migration on the page, not two). Follow the `MigrationDefinition` contract (`check`/`preview`/`run`, per-record audit log, per-step try/catch surfacing the real error, acknowledgement gate since it changes money, idempotent done-marker).

**Scope — every record whose stored total drifts from parts + labor, regardless of invoice status:**
- Scan `billing_sheets`, `work_orders`, `wet_check_billings` for `ABS( (COALESCE(parts,0) + COALESCE(labor,0)) − COALESCE(total,0) ) > 0.01`. Do **not** gate on `invoiceId` — this is the fix for the un-invoiced-straggler gap.
- **Repair each record:** set `totalAmount = money(partsSubtotal) + money(laborSubtotal)` (canonical, `money()`), leaving the (correct) stored subtotals intact — add-parts semantics, never lower a real total to match a bad one.
- **For invoiced records:** propagate the delta to the parent invoice — recompute the invoice's `partsSubtotal` / `laborSubtotal` / `totalAmount` from its (now-repaired) member tickets so the invoice and its PDF validator agree. (This is what the old reconcile did for billing sheets; extend to WO/WCB.)
- **Activity stream (Rule 6):** write an `audit_log` entry on each repaired ticket via `recordAuditEvent` — `action: "total_drift_repair"`, before→after totals, `migration` id. So the repair is visible on the ticket's Activity tab.

**Preview must show:** per-record `current total → repaired total (= parts + labor)`, the delta, invoice status/id, and — grouped — per-invoice total change. Counts: `driftedBillingSheets`, `driftedWorkOrders`, `driftedWcbs`, `invoicesAffected`, total `+$` to customers. Verify **#273 → $388.68**, **#272 → $463.68**, invoice 82027 up by $327.36.

**Safety:** company-scoped; idempotent (clean re-run finds 0); add-parts only (bills the correct higher amount, never lowers); acknowledgement gate lists the money change before Run.

## Slice 2 — Heal drift at invoice generation (stop recurrence)
**File:** `artifacts/api-server/src/routes/routes.ts` (monthly invoice generation / the path that attaches tickets to an invoice, ~6340–6540).
Today invoice generation uses the ticket's **stored** `totalAmount` as authoritative (`bs.totalAmount` at ~6356), so it inherits any latent drift and the PDF validator blocks it later. Instead, **when a ticket is attached to an invoice, recompute its total canonically first** (`total = money(parts) + money(labor)`), and if it drifted, persist the corrected total on the ticket (with a `total_drift_repair` activity entry) before including it. Net: an invoice can never inherit a drifted total; the validator stops being a tripwire that surfaces months later — drift is healed at the last responsible moment (per Rule 6, on the source ticket).
