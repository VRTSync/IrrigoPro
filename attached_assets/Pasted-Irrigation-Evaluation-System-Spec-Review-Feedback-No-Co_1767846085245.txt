Irrigation Evaluation System – Spec Review & Feedback (No Code)

This is a review of the current flows for:
	•	Estimate Form
	•	Work Order Form
	•	Billing Sheet Form

Plus key requirements around billing, roles, and visibility of pricing.

⸻

1. Overall Workflow – Confirmed

The high-level design is solid and makes sense operationally. There are three valid paths into billable work:
	1.	Estimate → Work Order → Invoice
	2.	Direct Work Order → Invoice
	3.	Billing Sheet (ad-hoc work) → Invoice

Conceptually, all three flows work and fit what we need in the field.

⸻

2. Main Gaps and Required Additions

To make this production-ready and avoid billing/clarity issues, a few structural pieces need to be added or clarified.

A. Work Order Pricing Snapshot

Work orders should contain enough information to stand alone as a billable record.

Please add pricing fields to the Work Order header, similar to the Estimate and Billing Sheet:
	•	Labor rate used for that job
	•	Labor subtotal
	•	Parts subtotal
	•	Total amount (overall total for that work order)

Right now, work orders only reference total hours and total parts cost. We need a clear, consistent total on every work order for billing and history.

⸻

B. Link to Invoices (to Prevent Double Billing)

Both Work Orders and Billing Sheets will be rolled into monthly invoices. To prevent double billing and to allow clean reporting, each should:
	•	Be able to reference the invoice they were included on
	•	Optionally store when they were billed

In practice, that means each work order and billing sheet needs:
	•	A reference to the invoice it was included on
	•	An optional “billed at” timestamp

This way, the system can easily filter for “completed but not yet billed” items when generating invoices.

⸻

C. Status Lifecycles – Make Them Explicit

The flows mention statuses, but the life cycle for each object should be clearly defined and consistently used.

Estimates
Suggested statuses:
	•	Pending – created and awaiting customer action
	•	Approved – customer approved
	•	Rejected – customer rejected
	•	Expired – optional, if approval links time out

It would also help to track when an estimate is approved or rejected (approval and rejection timestamps) and whether it has already been converted into a work order.

Work Orders
Suggested statuses:
	•	Pending – created but not yet assigned
	•	Assigned – assigned to a technician
	•	In progress – technician has started
	•	Completed – work finished and ready for billing
	•	Cancelled – job cancelled

Billing state can be handled by the invoice linkage, but if needed, “billed” can be used as well. The key is to clearly separate “finished in the field” from “already invoiced.”

Billing Sheets
Suggested statuses:
	•	Draft – technician is still editing
	•	Submitted – technician has submitted for office review
	•	Completed – reviewed/approved and ready for billing
	•	Billed – included on an invoice

Currently, the text describes “submit → completed,” which skips the “submitted” state. It will be clearer to use each state consistently.

⸻

D. Estimate → Work Order Conversion Details

The flow says an approved estimate automatically creates a work order. For clarity and to avoid confusion later, we should be explicit about what gets carried over.

When an estimate is approved and converted:
	•	The work order should inherit the estimate’s header fields (customer, project name, address, notes, labor rate if applicable).
	•	Zones associated with the estimate should be copied into the work order (if zones are being shown to techs).
	•	Estimate items should be copied into work order items as the starting point. The technician can later adjust quantities and add/remove parts as needed.

It would also be useful to keep:
	•	A reference from the estimate to the work order that was created
	•	A stored value for the original estimated total on the work order header so we can compare estimated vs actual later

⸻

E. Approval Link / Token Behavior

The Estimate flow mentions: “Token generated, email sent to customer, customer approves/rejects via unique link.”

For this to be robust, it should include:
	•	A unique approval token per estimate or per approval request
	•	An expiration for that link
	•	A way to record when the link was used and whether it was approval or rejection

This is mostly for security, auditability, and preventing repeated approvals or “stale” clicks.

⸻

F. Audit and Ownership Fields

For all major entities (Estimates, Work Orders, Billing Sheets, Invoices), it would be helpful to track:
	•	When they were created
	•	When they were last updated
	•	Who created them

Optionally, a soft-delete timestamp can be used if you want to hide records without permanently removing them.

⸻

3. Form-Specific Notes

Estimates

Structure is strong and covers what we need.

A couple of clarifications:
	•	Please clearly define how totals are calculated (parts subtotal, labor subtotal, total).
	•	It should be clear whether labor hours are “per line item” or per quantity. To avoid confusion, it’s usually best to treat labor hours as per line item (that is, total hours for that line), not automatically multiplied by the quantity.

It would also be helpful if the estimate stored a reference to the work order created from it, once it is converted.

⸻

Work Orders

The Work Order structure is generally good.

Key additions needed:
	•	Pricing snapshot: labor rate, labor subtotal, parts subtotal, and total amount
	•	Linkage to invoice (which invoice, if any, this work order has been billed on)

If multi-day / multi-technician jobs are expected frequently, there may eventually need to be a more detailed time entry structure. For now, a single start time, end time, and total hours can work for a first version.

⸻

Billing Sheets

Billing Sheets are designed well for ad-hoc field work.

Improvements:
	•	Use the full status lifecycle: draft → submitted → completed → billed
	•	Add invoice linkage so it’s clear if a billing sheet has already been invoiced
	•	Make the technician ID required (the name can be stored as a snapshot, but the system should always know which user performed the work)

Location fields are fine as long as it’s clear which address is treated as the main one for display and search.

⸻

4. Critical Requirement: Pricing Visibility by Role

This is very important operationally:

Field technicians must not see any pricing or money values anywhere in the app.

That includes:
	•	Labor rates
	•	Part prices
	•	Unit prices
	•	Line totals
	•	Subtotals (labor or parts)
	•	Markup or tax amounts
	•	Overall totals or invoice totals

Technicians should be able to see and work with:
	•	Which parts to use (names and quantities)
	•	Work descriptions and zones
	•	Project name and address
	•	Access instructions and notes
	•	Their own time fields (hours worked)
	•	Photos and attachments relevant to the job

Managers, office staff, and irrigation managers should see all pricing and totals.

This needs to be enforced not just in the visual layout, but in how data is returned:
	•	When a technician is using the system, any responses or screens they see should simply not include any pricing fields at all.
	•	Managers and office roles should see the full financial data.

The existing data fields for labor rates and prices are correct and can stay exactly as designed; the difference is purely in which roles are allowed to see them.

⸻

5. Permissions Summary

For clarity:
	•	Admin / Office / Irrigation Manager
	•	Can see and edit full financial information
	•	Can create and manage estimates, work orders, and billing sheets
	•	Can generate invoices and run billing
	•	Field Technician
	•	Can view assigned work orders (no prices shown)
	•	Can fill in hours, quantities, notes, and photos
	•	Can create and submit billing sheets (without seeing prices)
	•	Cannot see any dollar amounts or rate fields

⸻

6. Overall Conclusion

The current Replit breakdown is very close to what is needed. The core flows are correct and aligned with real-world use.

The main changes requested are:
	•	Add explicit pricing snapshot fields on work orders
	•	Add invoice linkage on work orders and billing sheets
	•	Clarify and consistently use status lifecycles
	•	Define estimate → work order conversion details more explicitly
	•	Tighten the approval token behavior
	•	Add basic audit metadata
	•	Enforce role-based visibility so technicians never see pricing

With these adjustments, the system should be robust, clear, and ready for everyday use in the field and in the office.
