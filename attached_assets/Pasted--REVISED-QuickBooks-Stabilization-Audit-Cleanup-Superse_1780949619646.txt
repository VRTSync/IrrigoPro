## Task 1 — QB Production Scoping + Cross-Tenant Lockdown (P0)

### What & Why

Every legacy QuickBooks route resolves the caller's company from the dev-only `x-user-company-id` header (`headerUserCompanyId`, routes.ts:85), which returns `undefined` in production because `isHeaderAuthAllowed()` is hard-off in prod (routes.ts:72). So in production these routes run with `companyId = null`, causing three P0 symptoms at once:

1. **Re-authorize always 400s** — OAuth state is stored with `companyId: null`; the callback guard (routes.ts:8688) aborts when it reads it back.
2. **Status falsely "green"** — `getQuickBooksCustomerStatus(null)` falls through to an unscoped `SELECT … LIMIT 1` returning an arbitrary tenant's row.
3. **Cross-tenant leak** — Company B sees Company A's connection via `/connection`, `/status`, **and `/health`**.

The correct identity is already on `req.authenticatedUserCompanyId` (set by the global auth middleware, routes.ts:857; used 93+ times elsewhere and by the Slice 1116 stale endpoint).

### Done looks like

- A `company_admin` in production: Re-authorize → consent → back on Settings with a fresh `realm_id`, no 400; callback log shows a real numeric company id, not null.
- `GET /api/quickbooks/connection` for a company with no integration returns `isConnected: false`.
- **Cross-tenant isolation across all reads:** Company A connected, Company B not → each of `connection`, `health`, `connection/stale` returns only the caller's own data; B never sees A.
- `sync-customers` and a single estimate sync succeed end-to-end post-reconnect; both 400 cleanly when the company has no connection.

### Out of scope

- `USE_DB_OAUTH_STATE` / the durable Postgres OAuth state store — leave as-is.
- `isHeaderAuthAllowed()` — production-off is correct.
- Non-QuickBooks `headerUserCompanyId` sites (routes.ts ~4713, 7584, 11069, 11508, 12446, 12745, 12961, 13192, 13705) — several already correct; do not touch.

### Steps

1. **Add `resolveCompanyId` helper** near `headerUserCompanyId` (routes.ts ~85):
   ```ts
   function resolveCompanyId(req: Request): string | null {
     const authed = (req as any).authenticatedUserCompanyId;
     if (authed != null) return String(authed); // text column → string
     const hdr = headerUserCompanyId(req);       // undefined in prod by design
     return hdr ? String(hdr) : null;
   }
   ```
2. **Swap company resolution** — replace `(headerUserCompanyId(req) as string) || null` with `resolveCompanyId(req)` at the routes that survive cleanup: `/api/quickbooks/auth` (~8589), `/api/quickbooks/disconnect` (~8829), `/api/quickbooks/connection` (~8951), `/api/quickbooks/sync-customers` (~9061), `/api/quickbooks/sync-estimate/:id` (~9323). *(Skip `/status`, `/customers`, `/sync-parts` — Task 2 deletes them; don't waste effort converting them.)*
3. **Remove the unscoped fallback** in `getQuickBooksCustomerStatus` (storage.ts:3156): the `else` (no companyId) branch must return `integration = []`, never a bare `SELECT … LIMIT 1`.
4. **Propagate company id through `syncQuickBooksCustomers`** (storage.ts:3196) — it currently calls `getQuickBooksCustomerStatus()` with no argument. Give it a `companyId` parameter and pass `resolveCompanyId(req)` from `/api/quickbooks/sync-customers`; 400 if null.
5. **[CORRECTED — was missing] Scope `/api/quickbooks/health` (routes.ts:9022).** It currently returns `getQuickBooksAllIntegrations()` — **every company's** integration to any authenticated user. Replace with a single company-scoped lookup keyed on `resolveCompanyId(req)`, returning only the caller's connection health (or `{ connection: null }`). The background token-refresh job may keep `getQuickBooksAllIntegrations()` internally — that's server-side iteration, not an API surface.
