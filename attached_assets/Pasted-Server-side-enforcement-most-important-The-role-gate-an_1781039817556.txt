Server-side enforcement (most important). The role gate and billed/invoiced lock are described as UI behavior (pencil hidden, canEdit). Confirm the PATCH endpoint independently authorizes billing_manager | company_admin | super_admin and rejects edits to billed/invoiced records. UI gating alone is bypassable, and a record can flip to billed while a modal is open — re-check the lock at save time, server-side.
Textarea + Enter conflict. "Enter/Escape" applied globally will break multi-line fields — Enter should insert a newline in the textarea variant, not save. Use Cmd/Ctrl+Enter to save there. Affects description, notes, special instructions, locationNotes, inspectionNotes.
Date timezone. toDateInput / scheduledDate / workDate are classic off-by-one territory (UTC stored vs local <input type=date>). Verify a date saved and refetched shows the same day across timezones.
fieldOverrides staleness. Optimistic override state needs a clear reset path — on successful refetch and on record change. Otherwise an override can mask the real server value, or persist onto a different record. Confirm what happens when the server returns a value different from the override.
Save-during-switch race. If a field is mid-save (spinner showing) and the user opens another field, does auto-cancel discard the in-flight PATCH or let it land? Define which wins.
Save failure path. Spinner covers the in-flight state, but there's no mention of error handling — does the field revert, stay open, toast? Worth specifying.
Field-name duality. scheduledDate/workDate, description/workDescription suggest one logical field with two model names across surfaces. Make sure each EditableField patches the correct backend field — easy to cross-wire.
Validation. No client validation mentioned for number/date — e.g., negative or absurd totalHours. At minimum bound it and rely on server validation.
Minor: the pre-existing TS errors left untouched reduce the type-safety signal for this change — fine to defer, but they can hide a new error in the same files.
