# Admin Migration â€” Run the Billing-Sheet / Invoice Reconcile from Super Admin

**What & Why**
The billing-sheet â†” invoice total reconciliation (Task #1422) exists only as a CLI script, so a super admin can't run it from the app. When an invoiced billing sheet's stored total disagrees with `parts + labor` (e.g. BS #201: off by $3.50), the **Invoice Detail Report PDF refuses to generate** until the record is repaired. The startup auto-repair only fixes *uninvoiced* sheets, so invoiced ones need this manual run.

This wires the existing reconcile logic into the app's **admin migrations framework** so the super admin can **preview** (see exactly which invoiced sheets drift, the per-sheet delta, and which invoices change) and then **run** it from `/admin/migrations` â€” with the framework's acknowledgement gate as the review/consent step.

**Reuse, don't rebuild.** The repair math + orchestration already live in `reconcile-billing-sheet-invoice-totals-core.ts` (built CLI-agnostic for exactly this). The migration framework (`lib/migrations/`), the runner UI (`MigrationRunner`), and the endpoints (`GET /api/admin/migrations`, `/:id/preview`, `POST /:id/run`, `/:id/status`) all already exist and are super-admin guarded.

Single slice. No schema change. No new endpoint or UI (the framework renders registered migrations automatically).

---

## Important â€” this changes money
The reconcile uses **add-parts semantics**: when a sheet's stored total is too low, it recomputes the sheet and **folds the missing-parts delta into the parent invoice's total** â€” i.e. the customer is billed more. The preview must make this unmistakable, and the run must be gated behind the framework's acknowledgement.

## Done looks like
- A new migration appears on `/admin/migrations` (super-admin only), e.g. **"Reconcile billing-sheet / invoice totals."**
- **Preview** (read-only, no mutation) returns a `MigrationPreview` that lists every invoiced billing sheet whose `parts + labor â‰  stored total`: sheet id/number, parts+labor, stored total, **delta**, the parent invoice id/number, and the invoice's **new total** after repair. Includes a clear summary line ("N sheets Â· K invoices Â· total +$X to customers").
- The preview populates the framework's **acknowledgement gate** (`orphanRows` / `hasOrphans` mechanism the page already uses to require `acknowledged` before Run) â€” repurposed here as "these invoices will change total; acknowledge to proceed." Run stays disabled until acknowledged.
- **Run** executes the reconcile core (apply mode), emitting per-sheet progress via the `ProgressEmitter`, and returns `MigrationStepResult[]`. Idempotent + resumable (the core already persists `reconcileBillingSheetInvoiceTotals.done` / `.failed` in `app_settings`); a clean re-run reports 0 repaired.
- After running, the affected Invoice Detail Report PDFs generate (validation passes).

## Steps
1. **Add a `MigrationDefinition`** at `artifacts/api-server/src/lib/migrations/reconcile-billing-sheet-invoice-totals.ts`:
   - `id: "reconcile-billing-sheet-invoice-totals-v1"`, `title`, `description` (state the add-parts / bills-more behavior).
   - `preview()`: call the core in **read-only** mode to compute the drift set; map it into `MigrationPreview` (summary + the per-sheet/per-invoice detail; set the orphan/ack count to the number of invoices that will change so the UI forces acknowledgement). No writes.
   - `run(emit)`: call the core's `runReconciliation` (apply), translating its per-sheet results into `emit(...)` progress events and the returned `MigrationStepResult[]`. Reuse the core's `app_settings` done/failed keys for resumability.
2. **Register it** in `lib/migrations/registry.ts` so `listMigrations()` includes it (alongside `companyIdColumnsMigration`).
3. Confirm the existing runner UI renders it with preview â†’ acknowledge â†’ run, progress, and last-run state â€” **no UI or endpoint changes needed**. If the core's preview/apply split needs a small read-only entry point, add it to `-core.ts` (keep the CLI working).
4. Leave the CLI script in place (still valid for ops); both call the same core.
