Replit audit prompt

We have a critical data integrity issue in Irrigo Pro.

Problem

When a billing sheet or work order is created and submitted, the original submission is losing the parts list for that ticket somewhere in the process. This is unacceptable for an irrigation billing system because the parts list is a core billing record.

Goal

Perform a full audit of the billing sheet and work order creation/submission pipelines and identify exactly where the parts list is being dropped, overwritten, excluded, or not loaded into the final record.

What I need you to audit

Trace the full lifecycle of the parts list through these stages:

Original ticket / work order creation
Where the parts list is first entered
How it is stored in state on the frontend
How it is sent in the API payload
How it is validated before save
Database persistence
Confirm the parts list is actually being written to the correct table/fields
Confirm the schema supports multiple parts line items correctly
Confirm no save/update process is nulling or replacing the parts list
Confirm whether parts are stored inline as JSON or in a related child table, and whether that relationship is being preserved
Submission / status transition flow
Audit what happens when a draft becomes submitted
Check whether a transform/mapping function is rebuilding the object and accidentally omitting parts
Check whether only certain fields are copied forward into the “final” billing sheet or work order
Check for any update statements that overwrite the existing parts list with empty arrays, null, or defaults
Billing sheet generation
Trace how billing sheets are built from the original ticket/work order
Confirm the generator is reading the parts data source correctly
Confirm the final billing record stores and renders the parts list rather than depending on transient state only
Check whether parts are displayed from the wrong object, wrong relation, or wrong version of the record
Work order generation
Same audit as billing sheets
Confirm the final generated work order includes all original parts line items
Confirm no serialization/deserialization issue is stripping nested part data
Frontend rendering
Confirm whether the parts list is actually missing from the saved final record or just not rendering in the UI
Audit component props, selectors, loaders, and API responses for mismatched naming or missing relations
Confirm whether the parts list exists in network responses but is being dropped in the display layer
Specific things to look for

I want you to aggressively look for these common failure points:

Field name mismatches between frontend, API, and database
Draft object vs submitted object using different shapes
Child table relations not being eagerly loaded
API DTO / serializer omitting nested parts
Update handlers replacing parts arrays instead of merging
Transaction steps that save parent record but not child part rows
Submission functions that create a new final record without copying parts
UI forms holding parts in local state but not including them in submit payload
Backend validation stripping unknown nested fields
PDF/export/final-sheet builder using a stale or incomplete source record
Race conditions where parts save after submission and miss the final snapshot
Any “map/pick/clone” logic that excludes partsList, materials, or lineItems fields
Deliverables I want back
A written audit of the full data flow for parts
The exact root cause of where the parts are being lost
A recommended fix
Any related vulnerabilities in the workflow that could cause future billing data loss
A hardening plan so this never happens again
Required hardening after fix

Once root cause is found and fixed, implement these protections:

Add logging for parts payload at create, update, submit, and final generation stages
Add backend validation that prevents submission if parts were entered but would be lost
Add automated tests covering:
create ticket with parts
save draft with parts
submit with parts
generate billing sheet with parts
generate work order with parts
edit submitted record without dropping existing parts
Add a regression test specifically for “parts entered on original ticket must appear on final billing sheet and final work order”
Ensure all final documents read from persisted data, not temporary frontend state
If parts are relational, ensure transactions save parent + child rows atomically
Success criteria

Done means:

A ticket created with multiple parts retains those parts through every workflow stage
Submitted billing sheets show the exact original parts list
Submitted work orders show the exact original parts list
No transition, update, or regeneration step can silently remove parts
Tests prove this workflow is stable

Please start by mapping the current data flow end-to-end before changing code. I want root cause identified first, then fix, then hardening.
