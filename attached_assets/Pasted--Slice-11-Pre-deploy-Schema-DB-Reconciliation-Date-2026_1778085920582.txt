## Executive Summary

Replit's migration prompt for the next deploy includes two destructive operations:

1. `DROP TABLE billing_number_counters CASCADE` — drops a live, in-use table that backs billing-sheet number generation. The table is created at runtime via raw SQL `CREATE TABLE IF NOT EXISTS` in `server/storage.ts:2610` and is not declared in `shared/schema.ts`, so Drizzle thinks it's an orphan and proposes dropping it. The "1 item" being deleted is the current counter value for the active billing-number prefix.
2. `DROP TABLE estimate_zones CASCADE` plus `DROP COLUMN zone_id` on `work_order_items` (18 items) and `estimate_items` — leftover state from Slice 1's flat-line-items strip-down. The 18 work-order-item zone associations are historical and no current code reads them.

Slice 11 fixes #1 by adding `billing_number_counters` to `shared/schema.ts` so Drizzle tracks it instead of dropping it. The raw-SQL bootstrap in `storage.ts` keeps working unchanged — this slice does not refactor runtime code, only declares the table to Drizzle.

Slice 11 accepts #2 as intentional and adds a CSV backup of the 18 affected rows under `migrations/backups/` before the drop runs, so the historical zone data is recoverable if anyone asks later.

After Slice 11 lands and the migration is regenerated, the deploy is safe.

**Slice 11 ships when:** `shared/schema.ts` declares `billingNumberCounters` matching the existing live table shape; a Drizzle migration regeneration produces a migration that **does not** include `DROP TABLE billing_number_counters`; the 18 zone-association rows are exported to `migrations/backups/work_order_items_zone_ids_pre_drop.csv` and committed; the `missing_photos_notifications_last_sms_sid_idx` drop is verified as intentional; and `npm run check` plus `npm test` both pass.

---

## Why Now

The migration Replit prompted with **will run on the next deploy** of any branch that touches the schema. That includes Slice 10a. So Slice 11 has to ship first, or 10a's migration will silently destroy the billing-counter state.

This is also the second time Drizzle has been confused about `billing_number_counters` (the prior prompt asked whether to rename it to `photo_late_additions`). The pattern will keep recurring on every future migration until the table is declared in the schema. Fixing it once is cheap; ignoring it forces re-litigating "do we drop the billing counter?" on every deploy.

## What Changes

### `shared/schema.ts` gains one new pgTable declaration:

```ts
// Counter table for sequential billing-sheet numbers, keyed by prefix.
// Created at runtime via raw SQL in server/storage.ts:2610 (CREATE TABLE IF NOT EXISTS).
// Declared here so Drizzle's migration generator tracks it instead of proposing a drop.
export const billingNumberCounters = pgTable("billing_number_counters", {
  prefix: text("prefix").primaryKey(),
  lastSeq: integer("last_seq").notNull().default(0),
});

export type BillingNumberCounter = typeof billingNumberCounters.$inferSelect;
```

Place near the other top-level tables (alphabetical or grouped with billing-related tables — `billing_sheets` is already in the file; put it adjacent).

### `migrations/backups/work_order_items_zone_ids_pre_drop.csv` (new file)

A one-shot dump of the 18 affected rows, run from a Postgres console before approving the migration:

```sql
COPY (
  SELECT id, work_order_id, zone_id
  FROM work_order_items
  WHERE zone_id IS NOT NULL
  ORDER BY work_order_id, id
) TO STDOUT WITH CSV HEADER;
```

Save the output as `migrations/backups/work_order_items_zone_ids_pre_drop.csv` and commit it. The file is under ~1KB so committing is fine.

If `estimate_items.zone_id` also has rows (the migration warning didn't quote a count, so possibly zero), do the same dump for it: `migrations/backups/estimate_items_zone_ids_pre_drop.csv`. If empty, skip.

### `migrations/` regenerated migration

After the schema change lands, run Drizzle's migration generator (`drizzle-kit generate` or whatever the project's wrapper is). The new migration file should:

- **Not** include `DROP TABLE billing_number_counters`.
- **Not** include any new operations on `billing_number_counters` at all (since the schema now matches the live DB).
- Still include the rest of the operations from the original prompt: create `photo_late_additions`, drop `estimate_zones`, drop the two `zone_id` columns, the FK constraint renames, the index drop, the `work_location_*` / `controller_letter` / `zone_number` column additions.
- Still include the `wet_check_finding_single_target` constraint replacing `one_routing_target`.

Diff the new migration against the original prompt to confirm only the billing-counter lines disappeared. If anything else changed, investigate before approving.

## Risks and Rollback

- **Schema declaration doesn't perfectly match the live table shape.** If `prefix` is actually `VARCHAR(N)` not `TEXT`, or if `last_seq` has a different default, Drizzle will see a diff and propose ALTERs. Mitigation: introspect the live table first (`\d billing_number_counters` in psql) and match the declaration exactly. The raw SQL in `storage.ts:2610` says `prefix TEXT PRIMARY KEY` and `last_seq INTEGER NOT NULL DEFAULT 0`, so use those.
- **The regenerated migration includes additional unexpected changes.** If the schema file has drifted from the live DB in other ways we haven't noticed, the generator may surface new ALTERs. Mitigation: diff carefully. If anything new appears, treat it as a separate scope question rather than rubber-stamping the regen.
- **The 18 zone associations turn out to be load-bearing somewhere.** Grep was clean, but a customer-facing PDF or report could be reading the column and silently producing different output once it's gone. Mitigation: the CSV backup means we can reconstruct. Stronger mitigation: before approving, search the codebase one more time for `zone_id` references in `work_order_items` context (`grep -rn "work_order_items.*zone_id\|zone_id.*work_order"`).
- **`missing_photos_notifications_last_sms_sid_idx` drop is unintentional.** If the schema declares the index and Drizzle is still proposing to drop it, something else is off. Mitigation: the verify step in this slice catches it before deploy.
- **`estimate_items.zone_id` has more rows than expected.** The migration warning only quoted 18 rows on `work_order_items`. If `estimate_items` has many rows, dump those too before approving.

**Rollback:** if the deploy goes wrong, the CSV backup plus a one-off `ALTER TABLE work_order_items ADD COLUMN zone_id INTEGER` and a re-population script can restore the zone associations. The `billing_number_counters` table is recoverable via the runtime self-heal path even without intervention. The `photo_late_additions` table is empty at deploy time; dropping it on rollback is harmless.
