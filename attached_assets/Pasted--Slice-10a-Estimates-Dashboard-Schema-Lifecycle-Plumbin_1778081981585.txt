## Executive Summary

Slice 10 is a small redesign of `/estimates` that adds a kanban-style **Estimates Dashboard** with five status columns â€” Drafts, Pending review, Sent, Approved, Rejected â€” plus a callout strip for expired estimates that are eligible to resend. Slice 10a is the foundation: schema-side normalization plus a single computed `lifecycleStatus` field returned by the API so the new frontend has one cleanly-typed string to switch on instead of having to reconstruct status from `status` and `internalStatus` together.

This slice does **not** ship any UI. It ships:

1. A real `'draft'` value in the `internalStatus` validator/zod schema (the schema comment in `shared/schema.ts:344-345` already reserves it; Slice 10a promotes it).
2. A pure function `computeLifecycleStatus(estimate)` in `shared/lifecycle.ts` that maps `(status, internalStatus, estimateDate)` to one of `draft | pending_review | sent | approved | rejected | expired`.
3. The same field returned on every estimate-shaped response from the server (single estimate, list, customer-facing approval lookup).
4. A new endpoint `POST /api/estimates/:id/transition` that handles three explicit transitions: `submit_for_review` (draft â†’ pending_review), `send_to_customer` (pending_review â†’ sent, triggers existing email), and `resend` (expired â†’ sent, resets `estimateDate`, triggers existing email). Invalid transitions return 400 with a human-readable error.
5. Tests covering the lifecycle computation and the transition endpoint's authorization + transition-validity rules.

**Slice 10a ships when:** every estimate shape returned by the API includes a `lifecycleStatus` string that matches the computed mapping; the existing wizard, customer-facing approval flow, and Slice 7 pending-approval queue all continue to work unchanged; the new transition endpoint accepts the three valid actions and rejects everything else; and the lifecycle computation has unit tests covering all six output values plus the boundary cases (29-day-old vs. 31-day-old sent estimates, etc.).

## Why Now

Slice 10b (Board view) and 10c (List view) both need to render five mutually-exclusive lifecycle buckets. Doing the bucketing on the frontend means duplicating the rule across two components; doing it inline means querying both `status` and `internalStatus` and timestamps on every render and is brittle when a future state gets added. A single computed field on the backend is the right shape.

A second motivation: drafts. Today every estimate created via the wizard immediately enters `pending_approval`, which means Slice 10b would render a Drafts column that's always empty. Promoting `'draft'` to a real value lets Slice 10c add a "Save as draft" affordance to the wizard without any further server work.

A third motivation: expiration is currently invisible. The schema declares `'expired'` as a possible `status` value (`shared/schema.ts:342`) but nothing ever writes it. The customer-facing approval flow uses a token-based `tokenExpiresAt` for security, but there's no business-side expiration of "this estimate has been sitting unanswered for too long." Slice 10a derives that cleanly without storing it.

## What Lifecycle Becomes

After Slice 10a, every estimate has exactly one `lifecycleStatus` value at any moment in time:

| `lifecycleStatus` | Predicate | Who can move it forward |
|---|---|---|
| `draft` | `internalStatus = 'draft'` | Irrigation manager â†’ submits for review |
| `pending_review` | `internalStatus = 'pending_approval'` | Billing manager / company admin â†’ approves and sends |
| `sent` | `internalStatus = 'sent_to_customer'` AND `status = 'pending'` AND `estimateDate >= now() - 30d` | Customer (via approval link) |
| `approved` | `status = 'approved'` | (terminal until conversion to work order) |
| `rejected` | `status = 'rejected'` | (terminal) |
| `expired` | `internalStatus = 'sent_to_customer'` AND `status = 'pending'` AND `estimateDate < now() - 30d` | Irrigation manager â†’ resends |

The mapping is deterministic and total â€” every (status, internalStatus, estimateDate) tuple maps to exactly one bucket. The function falls through to `pending_review` for any unrecognized internal state to fail closed (a partially-migrated estimate shouldn't disappear from the board).

   Behavior:
   - `submit_for_review`: requires current `internalStatus = 'draft'`, sets it to `'pending_approval'`, sets `updatedAt = now()`. Returns the updated estimate (including new `lifecycleStatus`). 400 if current status isn't `draft`.

   - All three transitions are gated by the existing role check pattern (whatever `requireAuth`/`requireRole` middleware is already in use). `submit_for_review` allowed for irrigation_manager (and admin). `send_to_customer` allowed for billing_manager and company admin only. `resend` allowed for irrigation_manager and above.

## API Specification

### Read-side: every estimate response includes `lifecycleStatus`

```ts
// Response shape, additive on every endpoint that returns an estimate
{
  id: 1042,
  estimateNumber: "EST-1042",
  status: "pending",
  internalStatus: "sent_to_customer",
  estimateDate: "2026-04-01T...",
  // ... all existing fields ...
  lifecycleStatus: "expired",  // NEW â€” computed server-side
}
```

### Write-side: new transition endpoint

```
POST /api/estimates/:id/transition
Authorization: existing session/cookie auth
Content-Type: application/json

Body:
{
  "action": "submit_for_review" | "send_to_customer" | "resend"
}

200 Response: the updated estimate (with new lifecycleStatus)
400 Response: { error: "Invalid transition: cannot submit_for_review when lifecycleStatus is 'sent'" }
401: not authenticated
403: authenticated but role doesn't permit this action
404: estimate not found or not visible to this user
```

### What the existing endpoints do not change

- `POST /api/estimates` continues to default `internal_status` to `'pending_approval'` for backwards compatibility. A caller can pass `"internalStatus": "draft"` to opt into starting as a draft. Slice 10c's wizard will use this.
- `PUT /api/estimates/:id` continues to update fields; lifecycle moves still go through the new transition endpoint.
- `POST /api/estimates/:id/approve` (the existing internal-approval endpoint from Slice 7) â€” keep it. It currently does the `pending_approval â†’ approved_internal` move and probably also kicks off the send. **Decision:** if the current approve endpoint already includes the send, keep it as-is; Slice 10a's `send_to_customer` transition is just a more explicit alias for the same operation. If they diverge, that's fine â€” both paths exist. Document the choice in the PR.
- The customer-facing approval-link endpoints (which set `status = 'approved'` or `'rejected'` based on the customer's click) are unchanged. Their effect on `lifecycleStatus` is automatic via the computed field.

## Data & Schema

**No DB migration.** Both columns involved (`status`, `internal_status`) are already `text` with defaults, so adding `'draft'` as a valid `internal_status` value requires only a validator/comment update.

**Existing rows** are unaffected. An estimate currently sitting in `internal_status = 'pending_approval'` keeps its row exactly as it is and starts appearing in the "Pending review" column as soon as Slice 10b ships. An estimate in `'sent_to_customer'` for >30 days will start showing as `'expired'` in `lifecycleStatus` immediately on deploy of 10a â€” but since no UI consumes the field yet, this is invisible until 10b/10c.

**Backfill:** none required. The default for new estimates remains `pending_approval` until Slice 10c flips the wizard's submit to start as `'draft'`.

## Risks and Rollback

- **Computed-field drift:** if a future caller mutates `status` or `internalStatus` without going through the transition endpoint and forgets that the lifecycle is derived from both, the board could show a stale-looking value. Mitigation: the `computeLifecycleStatus` function is the single source of truth, called every time the estimate is read out. As long as no caller caches the field across writes (and the API layer doesn't), this is self-correcting.
- **Boundary inconsistency on expiration:** "30 days" is computed against `estimateDate` not `approvalSentAt` â€” if those ever diverge significantly (they currently don't because the wizard sets `estimateDate = now()` on creation and the send happens shortly after), expiration could be off. Mitigation: the lifecycle function takes whichever timestamp the schema considers authoritative for the customer-facing clock. We use `estimateDate` because it's `notNull` and present from creation. If a future slice needs more precise per-event timing, switch to `approvalSentAt` then. Not now.
- **Role-check regression on the new transition endpoint:** if the existing role middleware is loosely typed and we copy-paste it wrong, the transition endpoint could allow a field tech to send an estimate to a customer (bypassing the billing-manager review). Mitigation: the integration test for `send_to_customer` runs with each of the four user roles and asserts only billing_manager + admin succeed.
- **Comment-block fix is a no-op for runtime but may reorder lines that other tools care about:** drizzle-kit may regenerate snapshots based on file content. Mitigation: run `drizzle-kit check` after the comment fix and commit any snapshot churn.

**Rollback:** `git revert` the slice commit. Because no DB migration ran and the only behavior change is an additive response field plus a new endpoint, reverting is a single-step operation. Existing clients ignore the new `lifecycleStatus` field; no client breaks.

## Migration / Rollout

Single deploy. No flag, no phased rollout. The slice is purely additive on read and adds one new endpoint on write. The wizard, customer approval flow, and pending-approval queue all continue to work because:

- The wizard still sends `internalStatus: 'pending_approval'` (default) and gets back an estimate with `lifecycleStatus: 'pending_review'` â€” but it doesn't read that field yet, so nothing changes.
- The pending-approval queue (Slice 7) reads `internal_status = 'pending_approval'` directly; the new computed field is ignored.
- The customer approval flow reads/writes `status` directly; the new computed field is ignored.

Slice 10b is what actually consumes `lifecycleStatus`. 10a just makes it available.
