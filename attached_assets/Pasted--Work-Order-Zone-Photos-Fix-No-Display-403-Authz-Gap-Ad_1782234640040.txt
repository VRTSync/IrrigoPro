# Work Order Zone Photos ‚Äî Fix "No Display" (403 Authz Gap) + "Add Add photo" Label

**What & Why**
On an inspection-origin work order, completed-work photos upload successfully (the "(1)" count appears) but render as a **broken image** ‚Äî they never display. Root cause is a **server authorization gap**, not the upload.

**Root cause:** Per-zone photos are stored in the structured **`work_order_zone_photos`** table (added with #1460; the zone checklist reads/writes `/api/work-orders/:id/zone-photos`). But the photo-serving authorization gate **`assertCanViewPhoto`** (`artifacts/api-server/src/routes/routes.ts:13926`) only authorizes photos found in:
1. `wet_check_photos` (joined via `wet_checks`),
2. the flat `work_orders.photos` text[] array,
3. `billing_sheets.photos` text[],
4. `estimates.photos` text[].

It **never checks `work_order_zone_photos`.** So when `<PhotoImage>` requests the signed URL / bytes for a zone photo, `assertCanViewPhoto` finds no owning record and returns **403 Forbidden** ‚Üí the image fails ‚Üí broken-image icon. Wet-check photos display because they live in a table the gate *does* check.

This affects **all** zone photos (camera *and* camera-roll) ‚Äî it's independent of the recent camera-roll change, which is working correctly.

Plus a cosmetic bug in the same view: the button reads **"Add Add photo."** `FileUpload` renders its button as `Add ${label}` (`file-upload.tsx:652`), and the zone checklist passes `label="Add photo"`, producing "Add Add photo."

Two small fixes. No schema change.

---

## Fix 1 ‚Äî Authorize `work_order_zone_photos` (the real bug)
### File
- `artifacts/api-server/src/routes/routes.ts` ‚Äî `assertCanViewPhoto` (~13926‚Äì13982), used by both `GET /api/photos/{*photoId}/signed-url` and `GET /api/photos/{*photoId}`.

### Change
Add a check for `work_order_zone_photos`, scoped to the caller's company via `work_orders ‚Üí customers`, mirroring the existing checks (and using the same `candidates` array that already normalizes leading-slash / variant-suffix / double-`photos/` forms):

```ts
// 5) work_order_zone_photos rows owned by this company
//    (structured per-zone store ‚Äî #1460).
const wozRows = await db
  .select({ id: workOrderZonePhotos.id })
  .from(workOrderZonePhotos)
  .innerJoin(workOrders, eq(workOrders.id, workOrderZonePhotos.workOrderId))
  .innerJoin(customers, eq(customers.id, workOrders.customerId))
  .where(and(
    eq(customers.companyId, user.companyId),
    sql`${workOrderZonePhotos.url} = ANY(${sql.param(candidates)}::text[])`,
  ))
  .limit(1);
if (wozRows.length > 0) return true;
```

Place it alongside the other company-scoped checks (before the final `403`). Import `workOrderZonePhotos` from the schema if not already imported in this file. Super-admin cross-tenant access already short-circuits above ‚Äî unchanged.
