# Wet Check Billing Rebuild — Implementation Plan

**Purpose:** Replace the current flat parts-list billing for BS-WC sheets with a zone-grouped layout in both the in-app modal and the customer PDF. This document is the source of truth — every slice below is intended to be handed to Replit as a self-contained task with no room for reinterpretation.

**Read order:**
1. Section 1: Design Context (all locked decisions, one source of truth)
2. Section 2: Open Decision Before Slicing (read and confirm before handing Slice 4 to Replit)
3. Section 3: Slices 1–8 (handed to Replit one at a time, in order)

---

## 1. Design Context (locked decisions)

These were agreed in conversation and are referenced by every slice. Replit should not deviate from any of them.

**Scope.** Only billing sheets whose `billingNumber` starts with `BS-WC-` are affected. Work-order and other billings keep the existing flat-parts layout.

**Source of truth for the new layout.** `wetCheckFindings` joined to `wetCheckZoneRecords`. Each finding already has `billingSheetId`, `zoneRecordId`, `issueType`, `partId`, `partName`, `partPrice`, `quantity`, `laborHours`. No data migration is needed — historical BS-WC sheets render correctly from existing data.

**Layout.** Zone-grouped. Each zone displays as: zone header (`<controllerLetter> <zoneNumber>` + issue display label), part lines, per-zone repair labor row showing `hours × rate = total`, per-zone subtotal. A single wet-check-level "Inspection Labor" line appears once per sheet.

**Repair labor entry.** Per-zone, stepper UI, 0.25-hour increments, 0.25-hour minimum. Free-form decimal inputs are not permitted in the new flow.

**Labor rate.** Always the per-customer global rate (`customers.laborRate`), snapshotted onto the billing sheet at creation (`billing_sheets.appliedLaborRate`). Never reprice based on later customer-rate changes. Existing snapshot logic is preserved.

**Issue type display.** Always use `issueTypeConfigs.displayLabel` for the customer's company. Never display the raw `issueType` field. If no config row exists for a `(companyId, issueType)` pair, fall back to title-cased `issueType.replace(/_/g, " ")`.

**$0.00 line item suppression on PDF.** Suppress when `lineTotal === 0 AND laborHours === 0 AND NOT isLaborOnly`. Labor-only findings (qty=0, partPrice=0, noPartNeeded=true) are kept because they represent real work performed.

**PDF treatment.** Repairs Summary table at the top (zone label → subtotal). Each zone wrapped in a div with `page-break-inside: avoid`. Labor math shown on PDF (`hours × rate = total`). Labor subtotal split into "Inspection Labor" + "Repair Labor". Issue type display labels everywhere.

**Issue type config normalization.** `issueTypeConfigs.defaultLaborHours` must be a multiple of 0.25 with a 0.25 minimum. The seed and admin validation must enforce this.

**Safety patch.** `deleteWetCheck` must refuse when any of the wet check's findings has `billingSheetId IS NOT NULL`, even if the BS is not yet on an invoice. Today's blocker only checks invoice linkage, which leaves a back door that the new renderer would silently break against.

**No PDF backward compatibility.** All wet check billings live in the current billing cycle (the wet check feature didn't exist before this cycle). There is no "old vs new" PDF reconciliation to handle. Regenerating an existing BS-WC PDF in the new format is the correct behavior.

## 2. Open Decision Before Slicing — Per-Zone Repair Labor Storage

The design language is "per-zone repair labor." The codebase today stores labor per-finding on `wetCheckFindings.laborHours`. Two viable implementations exist; pick one before handing Slice 4 to Replit, because the data model and the tech UI diverge between them.

**Option A — Sum-of-findings (no schema change).** Per-zone repair labor is computed as `SUM(wetCheckFindings.laborHours)` for findings in that zone. The tech enters labor per finding using the stepper (FindingSheet today). Zero migration, fits the existing data model perfectly. Downside: there is no zone-level value to display in the tech UI as a single source of truth — the zone's labor is always derived.

**Option B — New column on zone records.** Add `wetCheckZoneRecords.repairLaborHours decimal(5,2) default "0.00"`. The tech enters one labor value per zone via a stepper in ZoneScreen. Per-finding `laborHours` becomes an estimate field used only by the issue-type defaults and other (non-BS-WC) flows. Requires a column, a backfill, and a per-zone PATCH endpoint. Cleanest semantic model for the new UI.

**Recommendation: Option B** — it matches "per-zone repair labor" literally, makes the tech UI unambiguous (one stepper per zone, not one per finding), and the migration is mechanical (sum existing per-finding hours into the new column).

**This plan is written assuming Option B.** If you pick Option A, fold Slice 4 into Slice 6, skip the migration, and replace the PATCH endpoint in Slice 4 with continued use of the existing per-finding PATCH.

**Confirm Option A or B before Slice 4 is handed off.**

- POST/PATCH `/api/admin/issue-types` rejects `0.10`, `0.33`, `0`, returning HTTP 400 with the refine message.
- After running the backfill on a fixture with mixed inputs, zero rows have `default_labor_hours % 0.25 != 0`.

### Slice 3 — Backend: zone-grouped billing view assembler + endpoint

**Scope.** Build the server-side data shape that the new in-app modal (Slice 5) and the new PDF (Slice 7) both consume. One pure assembler function, one storage method, one new endpoint.

**Files modified.**

**Concrete changes.**

Define these TypeScript interfaces in `wet-check-billing-view.ts`:

```ts
export interface WcvLineItem {
  findingId: number;
  partName: string;           // f.partName || cfg.displayLabel fallback
  quantity: number;
  unitPrice: number;
  lineTotal: number;
  isLaborOnly: boolean;       // qty===0 && partPrice===0 && noPartNeeded===true
  laborHours: number;         // per-finding labor (preserved for display + reconciliation)
  notes: string | null;
}

export interface WcvZone {
  zoneRecordId: number;
  controllerLetter: string;
  zoneNumber: number;
  issueTypeKey: string;
  issueDisplayLabel: string;       // from issueTypeConfigs; never the raw key
  items: WcvLineItem[];
  repairLaborHours: number;        // see Slice 4: from zone column (Option B) or Σ findings (Option A)
  repairLaborRate: number;         // = sheet.appliedLaborRate
  repairLaborTotal: number;        // hours × rate
  partsSubtotal: number;
  zoneSubtotal: number;            // parts + labor for this zone
}

export interface WcvInspection {
  inspectionLaborHours: number;    // = wetChecks.totalLaborHours
  laborRate: number;
  inspectionLaborTotal: number;
}

export interface WetCheckBillingView {
  billingSheetId: number;
  billingNumber: string;
  isWetCheck: true;                // discriminator for the modal
  wetCheckId: number;
  customerId: number;
  customerName: string;
  laborRate: number;               // sheet snapshot (appliedLaborRate ?? laborRate ?? customer.laborRate)
  inspection: WcvInspection;
  zones: WcvZone[];                // ordered by controllerLetter, then zoneNumber ascending
  repairsSummary: Array<{ zoneRecordId: number; label: string; subtotal: number }>;
  partsSubtotal: number;
  laborSubtotal: number;           // inspection + Σ zone labor
  grandTotal: number;
}
```

Implement `buildWetCheckBillingView(input): WetCheckBillingView` as a **pure function** (no I/O). Heavily unit-tested.

Implement `storage.getBillingSheetWetCheckView(billingSheetId, companyId): Promise<WetCheckBillingView | null>`:
- Loads the BS.
- Resolves `wetCheckId` by reading any finding with `billingSheetId = X`.
- Loads the wet check + all findings with `billingSheetId = X` joined to their zone records.
- Loads `issueTypeConfigs` for the customer's company.
- Calls `buildWetCheckBillingView`.
- Returns `null` if the BS has no findings linked (i.e., it's a non-wet-check BS).

Add route `GET /api/billing-sheets/:id/wet-check-view` alongside the existing `/items` route (~routes.ts:12021). Same auth pattern as `/api/billing-sheets/:id`. Returns:
- HTTP 200 with the `WetCheckBillingView` payload, or
- HTTP 404 if the BS doesn't exist, or
- HTTP 422 with `{ message: "Not a wet-check billing sheet" }` if the storage method returns `null`.

**Label fallback rule.** `issueDisplayLabel` resolution: prefer `issueTypeConfigs.displayLabel` for the BS's company; if no row exists, fall back to `issueType.replace(/_/g, " ")` with each word title-cased. Never copy the raw `issueType` verbatim into `issueDisplayLabel`.

**Labor rate precedence.** `laborRate` in the response: `bs.appliedLaborRate ?? bs.laborRate ?? customer.laborRate`, parsed to number.

**Acceptance criteria.**
- `buildWetCheckBillingView` test covers: single zone single finding, single zone multiple findings, multi-zone (verifies ordering by controllerLetter then zoneNumber), labor-only finding, `$0.00` line item with `isLaborOnly: false` (suppression flag is on the consumer side; the view still returns the item).
- Endpoint returns the contract above for a real BS-WC sheet seeded in test setup.
- Endpoint returns HTTP 422 with the exact message for a non-wet-check BS.
- `displayLabel` is read from `issueTypeConfigs` for the BS's company.
- Labor rate in the response uses the precedence above.
- Zones are ordered deterministically: `controllerLetter` ascending, `zoneNumber` ascending.
- `partsSubtotal + laborSubtotal === grandTotal` in every test fixture.

**Out of scope.**
- Modifying the existing flat `/items` endpoint (keep for backwards compat).
- Modifying the PDF (Slice 7).
- Per-zone repair labor writes (Slice 4).
- Frontend changes.

**Dependencies.** Slice 1 merged.
