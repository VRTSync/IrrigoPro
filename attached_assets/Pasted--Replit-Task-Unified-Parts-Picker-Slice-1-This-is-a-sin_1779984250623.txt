6. `artifacts/irrigopro/src/components/manager/finding-card.tsx` — inline

   replace with PartPicker (dialog, single, `allowClear`, `categoryFilter`).

**No backwards-compat shim.** The old export name `PartsSearchModal` is
removed from the codebase. There are exactly five import sites; update all of
them in this slice. Do not leave a re-export.

**File location.** New home: `artifacts/irrigopro/src/components/parts/`,
alongside the existing `part-form-dialog.tsx` and `bulk-import.tsx`. The old
file `artifacts/irrigopro/src/components/estimates/parts-search-modal.tsx` is
deleted in this slice.

**Search.** Token-based ranking over a normalised projection of the part —
not Levenshtein, not a fuzzy library. Every whitespace-delimited token in
the query must match somewhere on the part for it to qualify (AND semantics,
same as today). The fields searched and their weights are locked in the
"Search algorithm" subsection below.

**No new dependencies.** Don't add Fuse.js, FlexSearch, MiniSearch, or any
other library. The ranking is ~50 lines of pure functions.

**Mobile-first sizing.** Tap targets ≥44px (the iOS minimum); row min-height
56px; sheet variant is full-screen below `sm` and right-side-panel at `sm+`.

**Recents.** The `/api/parts/popular?limit=` endpoint already exists and is
company-scoped. Surface up to 6 recents on an empty query, in BOTH the sheet
and dialog presentations (today only the dialog showed them).

**Category filter.** Two flavors: (a) the existing chip row (`showCategoryChips`),
user-driven; (b) a new hard filter prop `categoryFilter?: string | null`,
caller-driven. When a hard filter is set, chips only offer categories that
match the hard filter, and the user cannot select "All".

**Clear option.** New prop `allowClear?: boolean` + `onClear?: () => void`.
When true, the result list shows a "— No part —" row at the top that calls
`onClear` and closes the picker. This is the wet-check FindingCard's
existing affordance, preserved in the unified API.

**No "create part from search query" affordance.** Out of scope for this
slice. Adding a missing part still goes through the catalog management UI.

**Intentional UX deltas vs today.** These are deliberate, not regressions —
do not "fix" them back to the old behavior:

1. **Dialog default title.** The old `PartsSearchModal` defaulted to
   `"Add Parts to Billing Sheet"` for the dialog presentation. That string
   was misleading at the `edit-parts-modal` (which can wrap any parts list,
   not just billing) and `field-tech` callsites. The new dialog default is
   `"Add Parts"`. Callsites that need something specific pass `title`.

2. **Work-order completion swaps inline browse for a picker.** Today
   `work-order-completion.tsx` renders an always-visible "Quick Add Parts"
   list inside the form; techs can scroll the catalog without opening a
   modal. After this slice, that list is gone — replaced by an "Add Part"
   button that opens the unified sheet picker. This is the same pattern
   billing-sheet-wizard already uses. The used-parts list (Estimate vs
   Field-Added subsections, quantity steppers) below the button is
   **preserved unchanged**.

3. **Recents appear on the sheet presentation too.** Previously recents were
   dialog-only. After this slice they appear in both presentations on an
   empty query, capped at 6.

**Popular endpoint shape constraint.** The `/api/parts/popular` route
returns a narrower projection than `Part`:
`{ id, companyId, name, description, sku, category, price, usageCount }`.
It does **not** return `brand`, `material`, `size`, `fittingType`, `detail`,
`cost`, `isActive`, or `approvalStatus`. The recents tiles in the picker
must render only fields the endpoint actually returns (the spec below
limits them to `name`, `usageCount`, and `sku`). Do not extend the recents
tile to show e.g. `brand` without first extending the endpoint.

### 1a. Search algorithm — `part-search.ts`

Pure functions, no React. The component imports `normalisePart`, `searchParts`,
and the `NormalisedPart` type from this file.

**Field weights** (highest first; locked):

| Field         | Max weight |
| ------------- | ---------- |
| name          | 100        |
| sku           | 80         |
| brand         | 50         |
| category      | 40         |
| description   | 25         |
| material      | 20         |
| size          | 20         |
| fittingType   | 20         |
| detail        | 15         |

**Normalisation.** For both query and each searched field on the part:
lowercase → NFKD → strip combining marks → replace any run of
non-`[a-z0-9\s]` with a space → collapse repeated whitespace → trim. This
collapses `"Rain-Bird"`, `"rainbird"`, and `"RAIN BIRD"` into a consistent
token stream.

**Tokenisation.** The normalised query is split on whitespace. Empty queries
yield `[]`, which the caller treats as "match everything" with equal score
(stable secondary sort by name).

**Token-vs-field score** (returns a number in `[0, 1]`, multiplied by the
field's weight):

- field equals token exactly → `1.0`
- any word in the field equals the token → `0.95`
- any word in the field starts with the token → `0.8`
- the field as a whole contains the token (handles SKU fragments like
  `1804` inside `BCV-1804-2`) → `0.5`
- otherwise → `0`

**Part score across all tokens.** AND semantics: if any token scores `0`
across every field, the part is dropped entirely. Otherwise the part's
total score is the sum of `bestPerToken × fieldWeight` across tokens. Add
a `+1` tie-break bonus when the best field for a token is `name`.

**Sort.** Descending by score, with a stable secondary alphabetical sort by
normalised name (so an empty query produces a calm, predictable list).

**Public API.**

```typescript
export function normaliseForSearch(raw: string | null | undefined): string;
export function tokeniseQuery(query: string): string[];
export interface NormalisedPart { part: Part; fields: Record<SearchableField, string>; }
export function normalisePart(part: Part): NormalisedPart;
export interface SearchPartsOptions {
  categoryFilter?: string | null;   // case-insensitive exact match
  maxResults?: number;              // hard cap; sort first, then slice
}
export function searchParts(
  normalisedParts: NormalisedPart[],
  query: string,
  options?: SearchPartsOptions,
): Part[];
```

The component must memoise `normalisedParts` against the `/api/parts` query
result so the ~5ms-per-1k-parts normalisation pass only runs when the catalog
changes, not every keystroke.

### 1b. Component — `part-picker.tsx`

**Public API (locked).**

```typescript
export type PartPickerPresentation = "dialog" | "sheet";
export type PartPickerSelectMode = "single" | "multi";

export interface PartPickerProps {
  open: boolean;
  onOpenChange: (open: boolean) => void;
  onSelectPart: (part: Part, quantity?: number) => void;

  presentation?: PartPickerPresentation;        // default "dialog"
  selectMode?: PartPickerSelectMode;            // default "single"
  showCategoryChips?: boolean;                  // default false
  keyboardNav?: boolean;                        // default false
  showRecents?: boolean;                        // default true
  title?: string;
  categoryFilter?: string | null;               // hard filter, default null
  allowClear?: boolean;                         // default false
  onClear?: () => void;
}

export function PartPicker(props: PartPickerProps): JSX.Element;
```

**Behaviour rules.**

- On `open` going `false → true`: reset `searchQuery`, debounced query,
  category chip selection, active row index, "just added" confirmation. Focus
  the search input after a ~60ms timeout (lets the Sheet/Dialog mount).
- Debounce keystrokes by **150ms** before re-ranking the catalog. (Old
  modal used 200ms; tightening because the new ranker is faster.)
- `RENDER_CAP = 200`. Pass `maxResults: RENDER_CAP + 1` to `searchParts` so
  the component can detect overflow and surface "Showing first 200 of N
  matches — refine your search". Sort first, then cap (this is already how
  `searchParts` is specified above).
- `RECENTS_LIMIT = 6`. Recents are hidden whenever `debouncedSearch` is
  non-empty, and filtered to obey `categoryFilter` when set.
- Multi mode: on each select, fire `onSelectPart(part, 1)`, set
  `justAdded = part.name` for a 1400ms transient confirmation, keep the
  picker open. Done button at the bottom dismisses.
- Single mode: on select, fire `onSelectPart(part, 1)`, close the picker,
  clear the search query.
- `allowClear`: prepends a "— No part —" row at the top of the result list
  (not affected by search or category). Clicking calls `onClear?.()` then
  closes the picker.
- Keyboard nav (only when `keyboardNav` prop is true):
  - ArrowDown / ArrowUp navigate `activeIdx` (clamped to `[0, visible.length-1]`)
  - Enter selects the active row
  - Escape: `e.stopPropagation()` so an outer wizard never sees it and
    opens its discard dialog. The Sheet/Dialog's own Escape handler still
    fires.
  - Each navigated row is scrolled into view via
    `rowRefs.current[activeIdx]?.scrollIntoView({ block: "nearest" })`.
- Row content: name (1 line, truncate), then a wrapping metadata row with
  category Badge, brand, SKU (mono), and size — separated by gaps, all
  text-xs. Price right-aligned, semibold.

**Sizing and chrome.**

- Sheet presentation: `side="right"`, `className="p-0 w-screen sm:max-w-md md:max-w-lg sm:w-full flex flex-col h-full"`.
  Sticky header (Package icon + title + close X), sticky search row
  (search + chips + count line), scrollable body. Multi-mode footer
  (justAdded inline live region + Done button).
- Dialog presentation:
  `className="max-w-2xl max-h-[85vh] overflow-hidden p-0 flex flex-col"`,
  same sticky header (DialogTitle), search row, scrollable body, optional
  multi-mode footer.
- Search Input: `h-11` (44px), placeholder `"Name, SKU, brand, size…"`,
  pl-10 with a left Search icon.
- Result rows: `min-h-[56px]`, `px-4 py-3`, full-width left-aligned button.
- Recents tiles: `min-h-[56px]`, 1-col grid on `<sm`, 2-col grid on `sm+`.

**data-testid contract.** The old modal exposed
`part-picker-title`, `part-picker-search`, `part-picker-row-<id>`,
`part-picker-cat-all`, `part-picker-cat-<category>`, `part-picker-count`,
`part-picker-added`, `part-picker-done`, `part-picker-close`. These names
are kept identically. Add new ones for the new affordances:
`part-picker-recents`, `part-picker-recent-<id>`, `part-picker-results`,
`part-picker-empty`, `part-picker-clear`.

**Data fetching.** Use `useArrayQuery<Part>({ queryKey: ["/api/parts"], enabled: open })`
and `useArrayQuery<Part & { usageCount: number }>({ queryKey: ["/api/parts/popular"], enabled: open && recentsEnabled })`
— same imports the old modal uses (`@/lib/queryClient`).

**Loading state.** Show 5 skeleton rows in the result body while
`isLoading` is true. The search input remains usable; ranking just operates
on an empty list until the catalog arrives.

**Empty state.** Package icon + "No parts match `\"<query>\"`." when there's
a search query; "No parts in catalog." otherwise.

### Out of scope (do not include in this slice)

- Server-side search via `/api/parts/search?q=` — the catalog comfortably
  fits in memory client-side. Revisit if catalog size exceeds ~3000 parts.
- "Create a new part from the current query" affordance in empty state.
- Levenshtein / typo correction (e.g. `"rainbrd" → "rainbird"`). The
  normalisation already handles punctuation/case; true edit-distance is a
  follow-up if the tokeniser proves insufficient in the field.
- Reorganising the parts schema or the `/api/parts/popular` endpoint.
- The `companyId`-scoped popular endpoint already returns per-company data;
  no auth changes needed.
