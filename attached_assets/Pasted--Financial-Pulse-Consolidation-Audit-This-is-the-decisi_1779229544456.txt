This is the decision document. It captures what's currently displayed across the app, where Financial Pulse now overlaps, and the verdict for each.

**Status: verdicts approved.** Slice 5 implements them, with two amendments:

- Billing Dashboard changes deferred to Slice 6 (full redesign rather than a stopgap widget).
- Technician detail variant dropped — no tech profile page exists in the codebase to host it. Park until one is built.

## Data-source disagreement risks (the bugs hiding in plain sight)

Before consolidating UI, fix the data layer. Three places currently compute the same metric multiple different ways. Until these are unified, any "Financial Pulse widget" that pulls from one source while a legacy display pulls from another will visibly disagree — and the legacy ones are what users have trusted longest, so the widget will look wrong even when it's right.

| Metric | Today's situation | Fix |
|---|---|---|
| **"This Month Billed"** | Admin dashboard sums `/api/customers/billing-preview` client-side **and** calls `/api/dashboard/this-month-billed` separately. Both render on the same page; nothing guarantees they agree. (Task #662 already moved one path to the dedicated endpoint, but the client-side sum was left in.) | Delete the client-side sum. Route everything to `/api/financial-pulse/kpis` (the new canonical source) and deprecate `/api/dashboard/this-month-billed`. |
| **"Total Unbilled" / "Unbilled Exposure"** | Three code paths: admin dashboard derives from `/api/customers/billing-preview`; Financial Pulse computes its own via invoice/billing-sheet queries; Customer Billing page sums work-orders + billing-sheets client-side. | Define "unbilled" once in the Financial Pulse service. Replace the other two with calls to `/api/financial-pulse/kpis` (admin) and a new `/api/financial-pulse/customer/:id/summary` (per-customer). |
| **Per-customer unbilled & A/R** | Customer Billing page rebuilds totals client-side from work-order + billing-sheet + invoice lists. | New per-customer summary endpoint that returns the same numbers FP shows in its By Customer row, scoped to one customer. |

These three fixes are the real win. The widget rollout below is mostly visual cleanup riding on top.

## The widget

A single shared component used everywhere:

```tsx
<FinancialPulseWidget
  variant="admin-dashboard"
    | "customer-detail"
    | "ar-aging"
    | "top-customers-compact"
    | "billing-header"          // added in Slice 5.1
  customerId?: string           // required for customer-detail
  limit?: number                // for top-customers-compact
/>
```

Variants are thin presentational wrappers over a small set of tile/list configurations. All variants:

- Pull from `/api/financial-pulse/*` only (no new business logic).
- Include a "View on Financial Pulse →" link in the header, pre-filtered to the relevant entity or section.
- Show a compact skeleton during loading; fail soft (render a small "—" tile with retry) if the endpoint errors.
- Respect role guards: render nothing for users who can't access Financial Pulse endpoints.

This means five variants total (four in Slice 5, fifth in Slice 5.1) but exactly one component, one data path, one set of caching rules.

3. **A/R aging buckets on Invoices page are clickable filters.** Each bucket navigates to the invoices list filtered by aging.
4. **Technician detail variant dropped.** No tech profile page exists. Park until one is built.
