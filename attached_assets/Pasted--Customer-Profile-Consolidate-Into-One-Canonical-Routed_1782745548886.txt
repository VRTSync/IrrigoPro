# Customer Profile ‚Äî Consolidate Into One Canonical Routed Page

**What & Why**
There are **two** customer-profile components that have **diverged**, each carrying features the other lacks. Which one a user sees depends on role and navigation path ‚Äî so features land on one and are invisible from the other (that's why the Irrigation System Profile is unreachable for you). This is a maintenance/visibility vulnerability: building on a page you're not actually viewing. **Collapse to one canonical profile and delete the other.** No unreachable pages left in the system.

**The two today:**
- **A ‚Äî `pages/customer-profile.tsx`** (routed `/customers/:id/profile`, 601 lines). Has: Contact details, Labor Rate + Discount, **Irrigation System card + "Controllers & Zones" button**, **Financial Pulse**, **Budget + alerts**, **Invoice list**, Property Notes, View Site Map. Already reached by: field-tech customer click (`customers.tsx:244`), Financial Pulse widget (`financial-pulse-widget.tsx:709`), the Irrigation Profile back link (`IrrigationProfile.tsx:1880`), and the table "profile" button (`customers.tsx:358`).
- **B ‚Äî `components/customers/customer-profile.tsx`** (rendered **inline** when a non-field-tech clicks a customer, 678 lines). Has: KPI count tiles, **Estimates / Work Orders / Billing Sheets tabs** (full lists + `EstimateDetailModal` / `CompletedWorkDetailModal`), **Property Boundary section**, **Emergency Labor Rate**, View Site Maps (modal). **No** irrigation/financial/budget/invoices.

**Decision: keep A (the routed page) as canonical**, merge B's unique sections into it, repoint every entry point to the route, and **delete B** plus the inline-render path. End state: one deep-linkable customer profile, reached the same way for every role, with all features in one place.

Multi-tenancy unaffected (reads already company-scoped). This is a behavior-preserving refactor ‚Äî the risk is **feature loss**, so parity is the core acceptance criterion.
