## Design Context (locked decisions)

These were agreed in conversation and are referenced by every slice in this
rebuild. Do not deviate from any of them.

**Scope.** Only billing sheets whose `billingNumber` starts with `BS-WC-` are
affected by the broader rebuild today. After Phase 2 completes, those rows
will live in `wet_check_billings` with `WC-YYYY-NNNN` numbering.

**Wet check billing status enum.** Wet check billings auto-submit on creation
(no `'draft'` state). Possible statuses: `'submitted'`,
`'pending_manager_review'`, `'approved_passed_to_billing'`, `'billed'`. This
is narrower than the billing_sheets enum which also has `'draft'` and
`'completed'`.

**Layout for detail view.** Zone-grouped (reuse the existing
`WetCheckBillingView` component). Each zone displays as: zone header
(`<controllerLetter> <zoneNumber>` + issue display label), part lines,
per-zone repair labor row showing `hours × rate = total`, per-zone subtotal.
A single wet-check-level "Inspection Labor" line appears once per sheet.

**Edit flow is deferred.** Wet check billings are NOT editable through this
page or modal in this slice. Phase 2 Slice 8 will design the edit experience
separately. For now, the detail modal is read-only — no Edit button.

**Wet checks feel like their own thing.** Wet check billings are presented
with wet-check-native columns (issues count, zones count) and link back to
their originating wet check, not as a special case of billing sheets.

**(1) New `WetCheckBillingListItem` type** in

```typescript
export type WetCheckBillingListItem = WetCheckBilling & {
  /** Count of findings linked to this wet check billing. */

  zonesCount: number;
};
```

**(2) New storage method `getAllWetCheckBillingsWithCounts`** in storage.ts.
Place adjacent to `getAllWetCheckBillings` (added in Slice 1):

```typescript
async getAllWetCheckBillingsWithCounts(): Promise<WetCheckBillingListItem[]>
```

Implementation: a single SQL query that LEFT JOINs `wet_check_findings` to
`wet_check_billings` on `wcf.wetCheckBillingId = wcb.id`, GROUPs by `wcb.id`,
and aggregates `COUNT(wcf.id) AS issues_count` plus
`COUNT(DISTINCT wcf.zone_record_id) AS zones_count`. Return rows sorted by
`wcb.workDate DESC, wcb.id DESC`. Use the Drizzle query builder, not raw SQL,
where possible.

**(3) New `GET /api/wet-check-billings` endpoint** in routes.ts. Place
adjacent to `GET /api/billing-sheets` (~line 9649):

```typescript
app.get("/api/wet-check-billings", requireAuthentication, async (req, res) => {
  try {
    const items = await storage.getAllWetCheckBillingsWithCounts();
    res.json(applyPricingVisibility(req, items));
  } catch (error) {
    console.error(error);
    res.status(500).json({ message: "Failed to fetch wet check billings" });
  }
});
```

Notes:
- Uses `requireAuthentication` (same middleware as other list endpoints).
- Applies `applyPricingVisibility` so field techs get pricing fields stripped
  (mirrors the existing `/api/billing-sheets` behavior).
- No query parameters in this slice. Filtering is client-side. Server returns
  the full list; the page filters into sections.

**(4) New `GET /api/wet-check-billings/:id` endpoint** in routes.ts. Returns
the wet check billing PLUS its `WetCheckBillingView` (the zone-grouped
detail). The detail modal needs both shapes in one request:

```typescript
app.get("/api/wet-check-billings/:id", requireAuthentication, async (req, res) => {
  try {
    const id = parseInt(req.params.id, 10);
    if (!Number.isFinite(id) || id <= 0) {
      res.status(400).json({ message: "Invalid id" });
      return;
    }
    const wcb = await storage.getWetCheckBillingById(id);
    if (!wcb) { res.status(404).json({ message: "Not found" }); return; }
    const view = await storage.getWetCheckBillingViewById(id, null);
    if (!view) { res.status(404).json({ message: "View not buildable" }); return; }
    res.json(applyPricingVisibility(req, { wetCheckBilling: wcb, view }));
  } catch (error) {
    console.error(error);
    res.status(500).json({ message: "Failed to fetch wet check billing" });
  }
});
```

Response shape (exact — do not add fields):

```typescript
{
  wetCheckBilling: WetCheckBilling;
  view: WetCheckBillingView;
}
```

`getWetCheckBillingViewById` is added in Slice 2. If Slice 2 hasn't been
merged yet when you're working on this slice, **surface a question and stop**
rather than reimplementing the method here.
