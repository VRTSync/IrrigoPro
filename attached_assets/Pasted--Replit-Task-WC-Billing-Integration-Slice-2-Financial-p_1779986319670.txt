## Design Context (locked decisions)

These were agreed in the audit
(`WC Billing Integration - Slice 0 End-of-Month Audit.md`) and apply to
every slice in this family.

**Polymorphic invoice items.** `invoice_items.sourceType` is one of
`'work_order' | 'billing_sheet' | 'wet_check_billing'`. WCB invoice
items have `sourceType === 'wet_check_billing'`, `sourceId === wcb.id`,
and `wetCheckBillingId === wcb.id` (the dedicated FK column added when
the WCB sourceType was introduced).

**Source-of-truth precedence.** When the source row carries an
authoritative total (`wcb.totalAmount > 0`), use it. The
`invoice_item.totalPrice` is computed at invoice-creation time and is
the fallback for invoices created before snapshotting was reliable.
The BS branch demonstrates the pattern at `routes.ts:11181–11189`.

**Approval snapshots.** Wet check billings do NOT currently carry
`approvedLaborSnapshot` / `approvedPartsSnapshot` columns. That is a
separate slice (Slice 7). For now, return `null` for both. When Slice 7
ships, the same code path will populate them automatically because the
read uses the columns directly.

**No status remap.** The BS branch returns `bs.status` verbatim. The
WCB branch returns `wcb.status` verbatim. Downstream UIs handle the
status enum.

## Slice 2 — Add wet_check_billing branch to the invoice audit enrichment

### Scope

The invoice audit endpoint (the one consumed by the financial-pulse
drilldown) enriches invoice items at `routes.ts:11103+`. The current
implementation has a `work_order` branch (`:11124+`) and a
`billing_sheet` branch (`:11157+`), then falls through to the
invoice-item-level data for any other `sourceType`. WCB items
therefore appear with raw fields only — no status from the source row,
no authoritative totals, no `billedAt`. This slice adds the missing
`wet_check_billing` branch.

### Concrete changes

**(1) Add `wetCheckBillingId` to the returned shape.**

In `routes.ts:11193+`, the current `return { ... }` block exposes
`id`, `sourceType`, `sourceId`, `workOrderId`, `billingSheetId`,
`description`, `status`, `laborTotal`, `partsTotal`, `ticketTotal`,
`workDate`, `createdAt`, `approvedAt`, `billedAt`,
`approvedLaborSnapshot`, `approvedPartsSnapshot`. Add one more key —
`wetCheckBillingId` — adjacent to the existing `billingSheetId` field:

```typescript
return {
  id: item.id,
  sourceType: item.sourceType,
  sourceId: item.sourceId,
  workOrderId: item.workOrderId,
  billingSheetId: item.billingSheetId,
  wetCheckBillingId: item.wetCheckBillingId,   // NEW
  description,
  status,
  // … unchanged
};
```

`item.wetCheckBillingId` is already populated on
`invoice_items` rows whose `sourceType === 'wet_check_billing'` (see
the writer at `routes.ts:6929`).

**(2) Add the `wet_check_billing` enrichment branch.**

After the `billing_sheet` branch (at `routes.ts:~11190`, immediately
before the closing `}` of the `else if (item.sourceType ===
"billing_sheet" …)` block), add:

```typescript
} else if (item.sourceType === "wet_check_billing" && item.wetCheckBillingId) {
  const wcb = await storage.getWetCheckBillingById(item.wetCheckBillingId, userCompanyId ?? null);
  if (wcb) {
    status = wcb.status || "billed";
    description = `WC Billing ${wcb.billingNumber}`;
    workDate = wcb.workDate || item.workDate;
    createdAt = wcb.createdAt ? wcb.createdAt.toISOString() : null;
    // WCBs do not yet carry approval timestamps as dedicated columns
    // (Slice 7 will add them); approvedAt is left null and billedAt
    // comes from the WCB row's billedAt column.
    approvedAt = wcb.approvedAt ? wcb.approvedAt.toISOString() : null;
    billedAt = wcb.billedAt ? wcb.billedAt.toISOString() : null;
    // Approval snapshots — Slice 7 will populate these columns. For
    // now they read as undefined and the parse fallback returns null.
    if ((wcb as any).approvedLaborSnapshot) {
      try {
        const snap = JSON.parse((wcb as any).approvedLaborSnapshot);
        const parsed = typeof snap === "number" ? snap : parseFloat(snap?.laborSubtotal ?? snap?.total ?? "");
        approvedLaborSnapshot = isNaN(parsed) ? null : parsed;
      } catch { approvedLaborSnapshot = null; }
    }
    if ((wcb as any).approvedPartsSnapshot) {
      try {
        const snap = JSON.parse((wcb as any).approvedPartsSnapshot);
        const parsed = typeof snap === "number" ? snap : parseFloat(snap?.partsSubtotal ?? snap?.total ?? "");
        approvedPartsSnapshot = isNaN(parsed) ? null : parsed;
      } catch { approvedPartsSnapshot = null; }
    }
    // Use authoritative source totals when available
    const wcbLabor = parseFloat(wcb.laborSubtotal || "0");
    const wcbParts = parseFloat(wcb.partsSubtotal || "0");
    const wcbTotal = parseFloat(wcb.totalAmount || "0");
    if (wcbTotal > 0) {
      ticketTotal = wcbTotal;
      laborTotal = wcbLabor;
      partsTotal = wcbParts;
    }
  }
}
```

The `(wcb as any).approvedLaborSnapshot` casts are TEMPORARY — they
allow this slice to ship before Slice 7 adds the columns. When Slice 7
lands, the cast comes off naturally.

### Acceptance criteria

- The invoice audit endpoint returns `wetCheckBillingId` on every
  enriched item (null for non-WCB items, populated for WCB items).
- WCB items have their `status`, `description`, `workDate`,
  `createdAt`, `billedAt`, `ticketTotal`, `laborTotal`, `partsTotal`
  sourced from the underlying `wet_check_billings` row when the row
  exists.
- WCB items return `approvedLaborSnapshot: null` and
  `approvedPartsSnapshot: null` until Slice 7 ships. (After Slice 7
  ships, this same code path will populate them — no further change
  needed here.)
- The three new tests pass against the real database.
- Existing WO and BS enrichment behavior is unchanged. No fields
  removed, no fields renamed.
- `pnpm typecheck` and `pnpm test` pass.

- Frontend changes to surface `wetCheckBillingId` on the financial-pulse
