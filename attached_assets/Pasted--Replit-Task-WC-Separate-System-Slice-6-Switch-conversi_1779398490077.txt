# Replit Task — WC Separate System, Slice 6 (Switch conversion path)

This is a single self-contained task. The **Design Context** below is locked
decisions from the broader rebuild — it is reference material so you understand
the intent. The actual work for this task is in the **Slice 6** section at the
bottom. Do not deviate from any file paths, function names, schema field names,
or payload shapes specified. If you believe a deviation is necessary, surface it
and stop rather than proceeding.

> **This slice flips user-visible behavior.** After this deploys, every new
> wet check submission that auto-bills will create a `wet_check_billings` row
> (not a `billing_sheets` row). The new `/wet-check-billings` page (Slice 4)
> starts populating with real data. Old BS-WC rows in `billing_sheets`
> remain unchanged — Slice 5's migration script moves them.

> **Deploy ordering.** Deploy this BEFORE running Slice 5's migration script.
> Otherwise wet check submissions during migration create new BS-WC rows in
> `billing_sheets` that the migration's snapshot misses, leaving orphans.

> **Rule of thumb for this slice:** the refactor touches a critical
> transactional helper (`_writeRepairedInFieldBilling`) that's called from
> two places. The behavior change is meaningful — be conservative. Do not
> rename public methods, do not modify non-wet-check code paths, do not
> remove the legacy `billingSheetId` field on findings (it stays for legacy
> un-migrated rows until Slice 7 cleanup).

## Design Context (locked decisions)

These were agreed in conversation and are referenced by every slice in this
rebuild. Do not deviate from any of them.

**Scope.** Only the wet check conversion path is affected. Work-order
conversion to `billing_sheets` is untouched.

**Numbering.** New wet check billings use `WC-YYYY-NNNN` format, with the
sequence starting at 1000 per Slice 1. The conversion path consumes
`getNextWetCheckBillingNumber()` for each new row.

**Wet check billing status enum.** `'submitted'`, `'pending_manager_review'`,
`'approved_passed_to_billing'`, `'billed'`. Wet check billings auto-submit on
creation (no `'draft'` state); the conversion path always creates with
`status: "submitted"`.

**Layout for detail view.** Zone-grouped. Each finding is a line item;
per-zone repair labor comes from `wet_check_zone_records.repairLaborHours`
(authoritative); per-finding labor hours are vestigial for estimation
purposes.

**Source of truth for line items.** Wet check billings derive line items
from `wetCheckFindings` directly — there is NO `wet_check_billing_items`
table parallel to `billing_sheet_items`. The conversion path therefore does
NOT insert any items table rows; it only stamps `wetCheckBillingId` on the
findings.

**Findings dual-FK.** `wet_check_findings` has both `billingSheetId`
(legacy) and `wetCheckBillingId` (new, added in Slice 1). New conversions
set the new column and clear the legacy column. Legacy un-migrated rows
keep their `billingSheetId` set until Slice 5 migration moves them.

**Safety patch on `deleteWetCheck`.** The patch must block deletion when
any finding references EITHER a billing sheet OR a wet check billing,
regardless of invoice status. Three priority tiers: invoiced records >
uninvoiced BS > uninvoiced WCB.

**(3) Add new typed error class** in storage.ts. Place adjacent to
`WetCheckHasBillingSheetError` (~line 206–219):

```typescript
// Thrown by deleteWetCheck when one or more of the wet check's findings has
// wetCheckBillingId IS NOT NULL but the wet check billing is not yet on an
// invoice. The invoiced-records error takes priority; this fires only when
// there are no invoiced blockers AND no legacy BS-WC blockers. Surface
// mapped to HTTP 409 by the route layer.
export class WetCheckHasWetCheckBillingError extends Error {
  readonly code = "WET_CHECK_HAS_WET_CHECK_BILLING";
  billingNumbers: (string | null)[];
  constructor(wetCheckId: number, blockers: Array<{ id: number; billingNumber: string | null }>) {
    const parts = blockers.map(b => `wet check billing ${b.billingNumber ?? `#${b.id}`}`);
    super(`Cannot delete wet check #${wetCheckId}: linked to ${parts.join("; ")}. Remove it from the wet check billing first.`);
    this.name = "WetCheckHasWetCheckBillingError";
    this.billingNumbers = blockers.map(b => b.billingNumber);
  }
}
```

**(4) Extend `deleteWetCheck` safety patch** (storage.ts:6423–6446).

After the existing two-pass collection logic (invoiced BS + uninvoiced

```typescript
// Third pass: any wet check billing linked by a finding but NOT yet on an
// invoice is also a blocker. After Phase 2 migration, all wet check
// findings reference wet_check_billings via wetCheckBillingId — this pass
// catches the new world.
const findingWcbIds = await db
  .select({ wetCheckBillingId: wetCheckFindings.wetCheckBillingId })
  .from(wetCheckFindings)
  .where(and(
    eq(wetCheckFindings.wetCheckId, id),
    isNotNull(wetCheckFindings.wetCheckBillingId),
  ));

  findingWcbIds.map(r => r.wetCheckBillingId).filter((x): x is number => x != null)
));
const wcbRows = wcbIds.length > 0
  ? await db.select({
      id: wetCheckBillings.id,
      billingNumber: wetCheckBillings.billingNumber,
      invoiceId: wetCheckBillings.invoiceId,
    }).from(wetCheckBillings).where(inArray(wetCheckBillings.id, wcbIds))
  : [];
const invoicedWcbBlockers = wcbRows.filter(w => w.invoiceId != null);
const uninvoicedWcbBlockers = wcbRows.filter(w => w.invoiceId == null);

// Invoiced WCBs roll into the existing WetCheckHasInvoicedRecordsError —
// add them to the `blockers` array before the existing throw with
// kind="wet_check_billing".
for (const w of invoicedWcbBlockers) {
  let invoiceNumber: string | null = null;
  if (w.invoiceId != null) {
    const [inv] = await db.select({ invoiceNumber: invoices.invoiceNumber })
      .from(invoices).where(eq(invoices.id, w.invoiceId));
    invoiceNumber = inv?.invoiceNumber ?? null;
  }
  blockers.push({
    kind: "wet_check_billing",
    id: w.id,
    displayNumber: w.billingNumber ?? null,
    invoiceId: w.invoiceId,
    invoiceNumber,
  });
}
```
