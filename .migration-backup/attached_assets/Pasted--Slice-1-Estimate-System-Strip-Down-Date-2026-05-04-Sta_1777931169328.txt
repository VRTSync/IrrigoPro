# Slice 1 — Estimate System Strip-Down

**Date:** 2026-05-04
**Status:** Approved for implementation
**Author:** Engineering planning
**Decisions locked:** Q1 = single description field; Q2 = production estimates table is empty; Q3 = canonical modal absorbs edit-mode, enhanced variant deleted; Q4 = use existing `server/logger.ts`.
**Related:** Pre-work for Slice 2 (Wet Check System)

---

## Executive Summary

The estimate system has been doing two jobs: generating real customer-facing scoped quotes for one-off non-contracted work, and acting as an ad-hoc structured audit log of controller/zone-level findings during wet checks. The data model shows the strain — `estimate_zones.controller_id` is a free-text column that ignores the canonical `controllers` and `irrigation_zones` tables already in the schema. The result is a brittle hybrid that's hard to extend, hard to bill correctly from, and hard to reason about.

This slice peels the wet-check role out of estimates so the wet check feature can be built cleanly in Slice 2. After Slice 1, an **estimate** means *one document, one customer, one scoped quote, line items only — no zone breakdown.*

Because production has zero rows in `estimates` and `estimate_zones`, this slice is a **single forward-only migration with no legacy compatibility paths**. The deprecation tail that an empty-data refactor would normally require disappears entirely.

**Slice 1 ships when:** new estimates are created in flat (zone-less) mode, the `estimate_zones` table and `work_order_items.zone_id` column are removed, the canonical modal handles both create and edit, and the customer approval email renders the new flat shape cleanly.

---

## Why Now

Three pressures converged.

First, the business needs a real wet check tool that enforces full-property coverage every visit. The current pattern of "create an estimate with one zone per area we looked at" produces estimates that aren't really quotes — they're inspection reports masquerading as quotes. Customers get confused, billing gets confused, and the data is too unstructured to drive reporting like "which zones fail most often."

Second, the recompute incidents documented in `docs/billing-audit-report.md` and the WO-1774885124045-754 forensic note traced their root cause to downstream layers re-deriving totals because the upstream snapshot was inconsistent. The estimate→work order→billing chain works only when each layer trusts the previous layer's snapshot. Adding the wet check role on top of that chain — and pretending it's an estimate — increases the surface area where snapshots can diverge. Slimming estimates back to one job reduces that surface area.

Third, the front-end already shows the seam. There are two parallel modal components — `estimate-modal.tsx` and `enhanced-estimate-modal.tsx` — both with locally-defined `EstimateZone` interfaces, indicating an in-flight refactor that never landed. Slice 1 finishes that refactor in the right direction.

## What an Estimate Becomes

After Slice 1, an estimate is:

- A scoped quote for non-contracted, non-routine work that requires explicit customer approval before execution.
- One customer, one project (e.g., "Install pressure vacuum breaker," "Replace mainline at side of building," "Add new drip zone for new flowerbed").
- A flat list of line items: parts (with quantity and unit price) and labor hours.
- Carries customer info, project name, project address, location notes, access instructions, photos, attachments, and totals — all of which already exist on the `estimates` table.
- Goes through the existing approval flow: pending → approved → converted to work order. That flow stays untouched.

After Slice 1, an estimate **is not**:

- A property-wide audit. (That's a wet check.)
- A multi-zone breakdown of findings. (That's a wet check report.)
- A monthly maintenance billing ticket. (That's a billing sheet, generated by routine work or by a wet check conversion.)

### In Scope

1. Drop the `estimate_zones` table and its types (`EstimateZone`, `InsertEstimateZone`, `EstimateWithZones`).

9. Update `server/email-service.ts` to render flat line items only — no zone fallback.
10. Update `server/seed.ts` to seed flat-item example estimates.
11. Make `client/src/components/estimates/estimate-modal.tsx` the canonical modal — fold in edit-mode capability (accept `estimateId?: number | null`, fetch existing estimate via `useQuery`, populate the form).
12. Delete `client/src/components/estimates/enhanced-estimate-modal.tsx` and update `client/src/components/manager/estimates-manager.tsx` to import the canonical modal.
13. Replace the multi-zone editor inside the canonical modal with a flat line-items editor: one row per part with `partId`, `quantity`, `description`, `laborHours`. Single description field per row (Q1: Option A).
14. Update `client/src/components/work-orders/work-order-completion.tsx` and `work-order-details.tsx` to render flat line items — remove all zone-grouping rendering paths.
15. Update `client/src/components/work-orders/edit-work-order-modal.tsx` to drop the `zoneId` field.
16. Add the first tests in `tests/`: at minimum, snapshot tests for the new email render and an integration test for `createEstimateFromFlatItems`.

### Out of Scope (Slice 1)

- Building the wet check system. That's Slice 2.
- Refactoring `propertyZones` / `zones` tables (the field-tech-clock-number tables — separate problem, separate slice).
- Touching `controllers` or `irrigation_zones`. Wet check work in Slice 2 will harden those.
- Any changes to billing sheets, invoices, or QuickBooks integration.

## Schema Changes

A single Drizzle migration. Forward-only, no down-migration needed (production has no estimate data to preserve).

```ts
// shared/schema.ts changes

// estimate_items: add two columns
export const estimateItems = pgTable("estimate_items", {
  id: serial("id").primaryKey(),
  estimateId: integer("estimate_id").references(() => estimates.id).notNull(),
  // zoneId removed (was: references estimateZones.id)
  partId: integer("part_id").references(() => parts.id).notNull(),
  partName: text("part_name").notNull(),
  partPrice: decimal("part_price", { precision: 10, scale: 2 }).notNull(),
  quantity: integer("quantity").notNull(),
  laborHours: decimal("labor_hours", { precision: 5, scale: 2 }).notNull(),
  totalPrice: decimal("total_price", { precision: 10, scale: 2 }).notNull(),
  description: text("description"),                      // NEW — optional per-line description
  sortOrder: integer("sort_order").default(0).notNull(), // NEW — preserves display order
});

// estimate_zones: TABLE DROPPED
// EstimateZone, InsertEstimateZone, EstimateWithZones, insertEstimateZoneSchema: ALL REMOVED

// work_order_items: zoneId column DROPPED
export const workOrderItems = pgTable("work_order_items", {
  id: serial("id").primaryKey(),
  workOrderId: integer("work_order_id").references(() => workOrders.id).notNull(),
  // zoneId removed (was: references estimateZones.id)
  partId: integer("part_id").references(() => parts.id),
  partName: text("part_name").notNull(),
  partPrice: decimal("part_price", { precision: 10, scale: 2 }).notNull(),
  quantity: integer("quantity").notNull(),
  laborHours: decimal("labor_hours", { precision: 5, scale: 2 }).notNull(),
  totalPrice: decimal("total_price", { precision: 10, scale: 2 }).notNull(),
  actualQuantityUsed: integer("actual_quantity_used"),
  actualLaborHours: decimal("actual_labor_hours", { precision: 5, scale: 2 }),
  notes: text("notes"),
});

// New compound type replacing EstimateWithZones:
export type EstimateWithItems = Estimate & {
  items: EstimateItem[];
};
```

**Sanity check before running the migration:** the implementation runs the four-row count query first (estimates, estimate_zones, estimate_items, work_order_items with non-null zone_id). If any return non-zero, halt and reassess. Belt and suspenders even though Q2 is confirmed.

## API Changes

### `POST /api/estimates`

New schema in `server/routes.ts:254`:

```ts
const createEstimateSchema = z.object({
  estimate: insertEstimateSchema.extend({
    estimateDate: z.union([z.date(), z.string()]).optional(),
    partsSubtotal: z.union([z.string(), z.number()]).optional(),
    laborSubtotal: z.union([z.string(), z.number()]).optional(),
    totalAmount: z.union([z.string(), z.number()]).optional(),
    laborRate: z.union([z.string(), z.number()]),
  }),
  items: z.array(z.object({
    partId: z.number(),
    partName: z.string(),
    partPrice: z.union([z.string(), z.number()]),
    quantity: z.number().int().positive(),
    laborHours: z.union([z.string(), z.number()]),
    description: z.string().optional(),
    sortOrder: z.number().int().optional(),
  })).min(1, "Estimate must have at least one line item"),
});
```

Handler at `routes.ts:4624` simplifies: parse, compute `partsSubtotal` and `laborSubtotal` from items, insert estimate, insert items. No nested zone iteration.

## Migration

One Drizzle migration. Forward-only.

```sql
-- 1. Add new columns to estimate_items
ALTER TABLE estimate_items
  ADD COLUMN description TEXT,
  ADD COLUMN sort_order INTEGER NOT NULL DEFAULT 0;

-- 2. Drop the FK from estimate_items.zone_id (if it exists), then drop the column
ALTER TABLE estimate_items DROP CONSTRAINT IF EXISTS estimate_items_zone_id_fkey;
ALTER TABLE estimate_items DROP COLUMN IF EXISTS zone_id;

-- 3. Drop the FK from work_order_items.zone_id, then drop the column
ALTER TABLE work_order_items DROP CONSTRAINT IF EXISTS work_order_items_zone_id_fkey;
ALTER TABLE work_order_items DROP COLUMN IF EXISTS zone_id;

-- 4. Drop the estimate_zones table
DROP TABLE IF EXISTS estimate_zones;
```

The implementation runs the count query before `db push` to confirm zero rows exist in the affected tables. If any count is non-zero, the migration aborts and we revisit.

## Risks and Rollback

The risk surface is small because there's no production data to lose. The remaining risks:

- **A non-production environment with seed data may reference the dropped table.** Mitigation: `seed.ts` is updated as part of the implementation PR. Confirmed not just by code review but by re-seeding a fresh database and exercising the create/edit/approve/convert flow end-to-end.
- **The canonical modal regressing on edit-mode behavior.** Today edit-mode lives in the deleted `enhanced-estimate-modal.tsx`. Folding it into the canonical modal needs careful prop wiring. Mitigation: the implementation PR includes a manual test checklist (create estimate, edit it, save, re-open, verify all fields populated).
- **Tests don't exist.** The empty `tests/` directory has been a known risk for the project. Slice 1 adds the first ones — narrowly scoped to the email render and the create-flat-estimate API integration. We're not retrofitting full coverage; we're seeding the testing pattern.

Rollback is `git revert` plus a Drizzle migration that recreates `estimate_zones` (empty) and adds back the dropped columns. Because production has no estimate data, the rollback loses nothing.

## What This Unblocks

After Slice 1 ships, the wet check feature (Slice 2) gets a clean integration point: it owns the controller/zone hierarchy outright. When a wet check produces a finding that requires customer approval ("this property needs a one-off backflow replacement"), it produces a flat estimate via the simplified API. When a wet check produces a finding that's covered under contract, it produces a billing sheet directly without touching estimates at all.

The estimate system gets out of the wet check's way, and the wet check system never has to bend its data model around the estimate's old assumptions.
