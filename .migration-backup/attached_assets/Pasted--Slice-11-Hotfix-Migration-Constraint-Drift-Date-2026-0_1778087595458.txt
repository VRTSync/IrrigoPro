# Slice 11 Hotfix — Migration Constraint Drift

**Date:** 2026-05-06
**Status:** Active hotfix during Slice 11 deploy
**Type:** Operational / one-shot Replit agent prompt

The deploy migration failed on this statement:

```
ALTER TABLE "work_order_items" DROP CONSTRAINT "work_order_items_zone_id_estimate_zones_id_fk";
constraint "work_order_items_zone_id_estimate_zones_id_fk" of relation "work_order_items" does not exist
```

Drizzle's snapshot expects the constraint under its new naming convention (`{table}_{col}_{ref_table}_{ref_col}_fk`), but the live DB has it under PostgreSQL's default name (almost certainly `work_order_items_zone_id_fkey`). The same drift likely exists for `estimate_items.zone_id` on the next migration line. This hotfix diagnoses, repairs, and re-runs the deploy.

## Why this happens

Drizzle's migration generator works off two snapshots: the schema file and Drizzle's internal `migrations/meta/_journal.json` history. When a constraint was originally created via raw SQL or via an older Drizzle version that used PostgreSQL's default naming (`{table}_{col}_fkey`), the live DB ends up with one name while Drizzle's snapshot assumes the new convention (`{table}_{col}_{ref_table}_{ref_col}_fk`). The `DROP CONSTRAINT` then references a name that doesn't exist.

Two long-term mitigations, both out of scope for this hotfix:

1. **Audit and rename all `_fkey`-named FK constraints to the Drizzle naming convention.** A one-time DB cleanup that removes this class of drift permanently. Worth a follow-up slice.
2. **Tell Drizzle to introspect the live DB** before generating a migration, so the snapshot reflects reality. `drizzle-kit introspect` can produce a fresh schema file from the live DB, which can then be diffed against the desired schema. More work, more reliable.

For now, the `IF EXISTS` defensive pattern (Path B) plus selective renames (Path A) gets the deploy unstuck.
