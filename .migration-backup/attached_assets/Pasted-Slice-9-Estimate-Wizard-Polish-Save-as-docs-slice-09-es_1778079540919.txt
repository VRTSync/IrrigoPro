Slice 9 — Estimate Wizard Polish & UX Tightening
Date: 2026-05-06
Status: Ready for Replit implementation
Author: Engineering review
Type: Polish / bug fix (no schema, API, or storage changes)
Estimated effort: 4–6 hours
Related: Builds on Slice 8 (Estimate Creation Wizard Redesign). Touches the same five files (estimate-wizard.tsx + the four under wizard/). Server-side processEstimatePayload and the storage layer are not modified. The fixes here resolve real UX failures from Slice 8 and tighten the visual language now that the map pin / controller-zone additions have stretched the customer step.

Executive Summary
The estimate wizard ships, but field testing and review have surfaced a cluster of small but compounding issues: pressing Enter inside a quantity field bounces the manager to the Review step; project addresses are locked read-only on every edit; native drag-and-drop thrashes the line-items list as the cursor crosses row midpoints; the customer step has grown to 485 lines with five sub-sections crammed into one card; the part picker keyboard navigation doesn't scroll the focused row into view; mobile tooltips never appear because there's no hover; and a manager-approved estimate silently reverts to pending_approval on every edit because the wizard never passes internalStatus through.
Slice 9 ships when: every issue in §"Issues Found" is closed, the customer step reads as a clean two-step layout (Customer → Project) instead of one big card with hidden sub-sections, the line items step's drag-reorder is calm, the part picker keyboard flow scrolls the active row into view, and the wizard's edit mode preserves project address editability, customer email/phone overrides, and internalStatus correctly.

Issues Found
Real bugs (high priority)

Enter on a text/number input bounces to the next step. estimate-wizard.tsx handleKeyDown excludes TEXTAREA, BUTTON, and A, but not INPUT. Editing a quantity, labor-hours, description, or any text field and pressing Enter advances the wizard. (Lines 356–375.)
Edit-mode project address is locked read-only. estimate-wizard.tsx line 215 hardcodes useDifferentAddress: false in the hydration block. Combined with addressReadOnly = !!value.customer && !value.useDifferentAddress in the customer step, the address can't be edited until the user clicks "Use a different address" — even though the existing estimate's address may already differ from the customer's record.
Native drag-and-drop reorders on onDragOver, not on drop. estimate-wizard-line-items-step.tsx lines 144–154. The cursor crossing back and forth between two adjacent rows thrashes the list during the drag. Reorder should fire on onDrop (or guarded by midpoint check on onDragOver).
Edits silently flip estimates back to pending_approval. The wizard never passes internalStatus in the payload, so server-side processEstimatePayload defaults it to "pending_approval". Editing a manager-approved estimate sends it back through the queue. Confirm with whoever owns the approval workflow whether this is intentional; if not, pass existing.internalStatus through on edit.
Controllers query has a misleading "empty" state. estimate-wizard-customer-step.tsx line 415: when controllers.length === 0 the Select is disabled with placeholder "No controllers on file". During the initial fetch this string flashes even when the customer DOES have controllers — should distinguish loading vs truly-empty using useQuery's isLoading.
Tooltip on disabled Continue button is invisible on touch. estimate-wizard-line-items-step.tsx lines 514–531. Mobile users see a greyed-out "Continue to Review" with no explanation. Replace with inline helper text (matching the Step 1 pattern that already does this for "Project Name *").

Layout / hierarchy issues (high priority)

The customer step is one card with five sub-sections. Project name → address toggle → LocationFields (address, location notes, access instructions) → border-t map picker → border-t controller/zone. On mobile that's a long uninterrupted scroll with weak section boundaries (just hairlines). Should split into discrete cards: Customer (existing), Project Details (name + address + notes + access), Work Location (map + pin), Controller & Zone (selectors). One concern per card. The same icon-in-rounded-tile header pattern already used.
Customer email/phone editable inputs read as customer-record edits. Without an explanatory caption, a manager will assume they're updating the customer's contact info. Add a helper line under the email/phone pair: "Used for this estimate only — won't update the customer record."
Pinned-location card uses green (success state) but it's an informational confirmation, not a success. Same green is already used for running total and submit CTA — over-greens the wizard. Switch the pinned-location confirmation to blue (info) with the same border-l-4 border-l-blue-500 pattern used elsewhere for highlighted-info surfaces.
Mobile cards in line items use a 2×2 grid for Qty / Unit / Labor / Line Total. Cramped on 360px screens. Stack fully vertically on <400px or use a 1-column layout with right-aligned values for unit price + line total. The spec called for "stacked cards"; current is gridded.

Polish (medium priority)

Part picker doesn't scroll the active row into view on ArrowUp/ArrowDown. With 200 visible rows, the keyboard cursor disappears below the fold within a few presses. Track an array of refs (or compute via getElementById from the row's data-testid) and call scrollIntoView({ block: "nearest", behavior: "auto" }) on activeIdx change.
Part picker shows no result count. A small caption under the search field ("Showing 47 of 234 parts" / "Showing 200 of 1,247 — refine your search to see more") helps users understand whether their search is too broad or too narrow.
Number inputs show native browser spinners alongside the explicit +/– buttons. Redundant, looks busy on desktop Chrome. Hide native spinners with [appearance:textfield] [&::-webkit-inner-spin-button]:appearance-none [&::-webkit-outer-spin-button]:appearance-none (Tailwind v3+ syntax) on the Qty / Labor Hrs inputs.
Mobile sticky-footer button verbs are inconsistent ("Continue" → "Review" → "Save"/"Submit") while the desktop footer uses ("Continue" → "Continue to Review" → "Submit for Approval"). Align mobile to the desktop verbs (truncated): "Continue" → "Continue" → "Submit" is fine, but use the same verb across both layouts so the user learns one mental model.
The map button label "Hide Map" / "Select Location on Map" is verbose and asymmetric. Use "Open map" / "Close map" or pair with an icon: <MapPin /> Pin location / <X /> Close.
Variable names in callers still say showEstimateModal / setShowEstimateModal / estimateModalOpen. Affected: pages/estimates.tsx, pages/dashboard.tsx, pages/manager-dashboard.tsx, pages/operations.tsx, components/manager/estimates-manager.tsx, components/customers/customer-profile.tsx. Functionally identical but a renaming pass to showEstimateWizard improves readability.
Edit-mode wizard header doesn't indicate loading. While existingLoading && !hydratedRef.current, the header still reads "Step 2 of 3 · Line Items" with the wizard title Edit Estimate #N. Add a subtle "Loading…" inline next to the title, or replace the step caption with "Loading estimate…" until hydration completes.
Discard-confirm dialog wording is generic. "You have unsaved changes. Closing the wizard will discard them." Tighten to action-oriented copy: "Discard this estimate? Your line items, customer details, and uploads won't be saved." Same dialog used in both new and edit; copy can branch on isEdit.

Optional / nice-to-haves

No step-transition animation. A 150ms slide-fade between steps would add polish (CSS animate-in slide-in-from-right-4 fade-in on the active step container, keyed on step). Cheap.
No localStorage draft persistence. Refreshing the page mid-edit drops everything. Persist customerStep + items + photos.urls + attachments.urls to localStorage keyed by estimateId ?? "new", restore on open. Out of scope unless requested.
Reverse-geocode the dropped pin. When a user clicks a spot on the map and no address comes back, the confirmation shows raw lat/lng. Reverse-geocoding (via the existing geocoding utility) would let the confirmation read as a real address. Out of scope.

Issue 3 — Drag-reorder thrashing. Track the dragged row in dragRowId (already done). Replace the onDragOver reorder logic with a midpoint check, OR move the splice into a new onDrop handler. Recommended:
tsconst onDrop = (rowId: string) => (e: React.DragEvent) => {
  e.preventDefault();
  if (!dragRowId || dragRowId === rowId) return;
  const fromIdx = items.findIndex((i) => i.rowId === dragRowId);
  const toIdx = items.findIndex((i) => i.rowId === rowId);
  if (fromIdx < 0 || toIdx < 0) return;
  const next = [...items];
  const [moved] = next.splice(fromIdx, 1);
  next.splice(toIdx, 0, moved);
  onItemsChange(next);
};
const onDragOver = () => (e: React.DragEvent) => { e.preventDefault(); };
Wire onDrop on each <tr> and keep onDragEnd resetting dragRowId.

Acceptance Criteria

Pressing Enter inside any text or number input on any step does NOT advance the wizard.
Opening an existing estimate to edit shows the project address as fully editable, with the toggle reading "Use customer address" by default. Toggling to customer-address mode locks it to the customer's stored address.
Editing a manager-approved estimate does not silently flip its internalStatus back to pending_approval (or, if intentional, surfaces a clear banner on Step 3 explaining that submitting will re-route through approval).
Drag-reordering line items on desktop only commits on drop; hovering between rows does not thrash the list.
The customer step renders as four discrete cards: Customer, Project Details, Work Location, Controller & Zone — each with its own icon header.
The customer-email and customer-phone inputs are accompanied by helper copy clarifying they're estimate-scoped overrides.
The pinned-location confirmation card uses the wizard's blue accent, not green.
Mobile cards in line items stack their fields cleanly at narrow viewports (≤400px).
The part picker scrolls the active keyboard row into view on Up/Down and shows a result count (or "No matches") under the chips.
The "Continue to Review" / "Review" CTA on Step 2 has inline helper text when disabled — no tooltip required.
The discard-changes dialog reads correctly in both new and edit modes.
Native browser number-spinners are not visible on Qty / Labor Hrs inputs in any browser.
Caller files use showEstimateWizard / setShowEstimateWizard consistently; no EstimateModal-named state remains.
Online and offline behavior of the existing flows are unchanged. processEstimatePayload is not modified. Server endpoints unchanged.
